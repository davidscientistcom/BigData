Anatomía Nuclear del Contenedor

Desmitificación del proceso aislado. Profundidad sobre primitivas del Kernel de Linux (Namespaces y CGroups) y el mito de la máquina virtual compartida.

1. El falso mito de la Máquina Virtual

Históricamente, el aislamiento de procesos se conseguía virtualizando hardware completo mediante un Hypervisor. El coste asociado a arrancar un sistema operativo invitado gigante solo para ejecutar un servidor web de 20MB suponía un desperdicio del 90% de los recursos.

Docker no virtualiza hardware. Docker es un demonio (proceso root silencioso) que habla directamente con la API del Kernel de Linux para crear procesos aislados lógicamente entre sí. Lo hace exprimiendo dos características base del sistema operativo:

Contenedores contra Máquinas Virtuales
Namespaces (El Muro de Cristal)

Hacen creer al proceso que es el único dueño y señor de la máquina. Proporcionan vistas de sistema únicas incluyendo: MNT (sistema de archivos), PID (árbol de procesos empieza en 1), NET (interfaces de red dedicadas) y UTS (hostname del servidor).

CGroups (La Cuota Dictatorial)

Las Control Groups regulan agresivamente los recursos físicos permitidos e imponen un techo de cuota para el proceso aislado. Ejemplo: Limitamos el CGroup del Namespace a usar como máximo el 15% de RAM o 2 hilos nativos del procesador.

2. Inyeción Subyacente e Instanciación

Una Imagen no corre. Es un artefacto inmutable guardado en disco. Consta de capas geológicas comprimidas (UnionFS) de binarios Unix.

El Contenedor nace en el instante en el que Docker invoca el Kernel con una petición clone() y asigna los Namespaces pertinentes. Una fina capa R/W efímera es montada en la cima del sistema de carpetas y el ciclo arranca al invocar su binario maestro.

bash — Inicialización de Nginx Atrapado 
# Instanciación en Foreground.
# Nginx no entrega el control de la terminal (STDOUT), si haces Control-C lo matas de muerte inmediata (SIGKILL).
docker run nginx:alpine

# Instanciación en Detached (Demonizado).
# Ejecuta en el fondo y Docker devuelve un Hash UUID hexadecimal de rastreo.
docker run -d --name engranaje-core nginx:alpine

3. Violar el Aislamiento (Modo Interactivo TTY)

Al ser simplemente procesos Unix confabulados por el daemon, un administrador maestro (Root del servidor Host) puede saltarse el plano exterior e inyectar un nuevo subproceso directamente en los dominios del Namespace aislado simulando un acceso SSH, sin el peso inherente del propio protocolo SSH.

Análisis Crítico: exec no es equivalente a hacer login completo a un servidor. No corren daemons init como systemd ni se ejecutan scripts globales como un bash-profile completo. Solo estás empujando el binario local /bin/bash dentro de la jaula virtual del proceso principal.
bash — Saltándose el Namespace 
# La i minúscula indica Interactive, la t indica seudo-TTY. 
# Si tu imagen no tiene Bash precompilado (como Distroless o Alpine básico), 
# usarás `/bin/sh` u otro binario menor equivalente.
docker exec -it engranaje-core /bin/bash

# Comprobación de que estamos dentro de una simulación completa (el árbol cuelga del PID 1 de Docker):
root@8b5ef2c2c011:/# ps -ef

4. La Naturaleza de la Limpieza Profunda

Mantener servidores encendidos "por si acaso" es un antipatrón en la era Serverless y Cloud. Los UUID inactivos de contenedores engordan la base de datos de Docker provocando desgaste progresivo. La administración correcta de un ecosistema es destruir el material inactivo radicalmente.

bash — Poda Inmutable 
# Detener grácilmente un servicio activo (envía señal limpia SIGTERM y espera 10s al FS)
docker stop engranaje-core

# Poda universal en el servidor de Integración Continua (Limpia GB en segundos).
# Esto destruye todos los contenedores cerrados y purga las Imágenes que no dependan actualmente de un contenedor existente.
docker system prune -f
