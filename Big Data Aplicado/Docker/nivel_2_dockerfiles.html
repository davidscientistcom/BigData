Construcción y UnionFS

Una Imagen de Docker no es un monolito rígido. Es una estructura apilada de capas de sistema de archivos inmutables unidas por el Kernel de Linux.

1. El Sistema de Ficheros Unificado (UnionFS)

El poder de Docker reside en cómo almacena las imágenes en disco local. En vez de guardar un disco virtual inmenso de 1GB, el demonio subdivide el almacenamiento usando una técnica denominada Union File System. Cada comando instruido (como un RUN o un COPY) construye y memoriza un estrato diferencial e inmutable.

El beneficio arquitectónico es doble: reducción drástica de tiempos entre redespuntes de subidas de registro a Azure CR o Docker Hub, y un uso óptimo del caché durante el ciclo en C.I.

Capas Inmutables en Docker UnionFS

2. Patrones de Diseño: Invalidación de Caché

Al procesar un Dockerfile top-down, si Docker detecta que el archivo local ha cambiado frente al registrado, descarta al instante la capa de caché en proceso y todas las capas subyacentes siguientes. Por tanto, copiar el código fuente agresivamente al principio de nuestro documento forzará al servidor de Build a descargar los 800MB de frameworks una vez por edición de backend.

Estrategia Suspendida y Lenta
  • COPY . /app
  • RUN pip install -r reqs.txt
  • Modificar el readme rompe el pip.
Estrategia Eficiente y Optimizada
  • COPY requirements.txt /app/
  • RUN pip install -r reqs.txt
  • COPY src/ /app/src/

3. El Atacante Root (Riesgos Estructurales)

Si nuestro contenedor corre internamente como root y sufre una brecha de RCE por una vulnerabilidad de Node.js, y nuestro clúster Kubernetes no restringe la directiva privileged, el atacante controlará el FS inyectando volúmenes laterales perimetrales hacia el host real de AWS/GCP.

dockerfile — Desacoplamiento de Privilegios 
FROM node:18-alpine

WORKDIR /usr/app-segura

# 1. Compilación con privilegios administrativos del daemon durante el CI/CD:
COPY package.json package-lock.json ./
RUN npm ci --only=production

COPY src ./src

# 2. Hardening: Configuración de subgrupo de sistema local carente de shells.
# Otorgamos permisos de acceso únicamente al binario o carpeta.
RUN addgroup -S securitygroup && adduser -S serviceuser -G securitygroup
RUN chown -R serviceuser:securitygroup /usr/app-segura

# 3. Transición obligatoria sin retorno: Declinamos nuestro rango Root en tiempo de ejecución.
USER serviceuser

EXPOSE 3000

# 4. Proceso en Namespace blindado:
CMD ["npm", "start"]

4. Multi-Stage Builds (Microscópicas e Inexpugnables)

Distribuir compiladores al cliente final y en producción implica mantener miles de librerías base expuestas (como gcc o pyinstaller). Herramientas nativas compiladoras, tales como Golang o Rust, habilitan una técnica destructiva: empaquetar un contenedor pesado como Builder, exportar su ejecutable CGO hacia un contenedor ínfimo, y lanzar las dependencias a la basura antes de enviar la huella final.

dockerfile — Compilación Destructiva Bifásica (Go) 
# --- START: PRIMERA FASE: EL MAQUINISTA (1.2GB)
FROM golang:1.20-alpine AS worker_builder

WORKDIR /sys_compiler
COPY . .

# Inhabilitamos llamadas dinámicas embebidas hacia librerías que solo existirían en Alpine.
# El binario dependerá única y exclusivamente del kernel directo.
RUN CGO_ENABLED=0 GOOS=linux go build -o /worker_kernel_daemon main.go


# --- END: SEGUNDA FASE: MICRO-RUNTIME ISOTEMPORAL (3MB)
# Partimos de una imagen purga y vacía ("scratch" o "distroless"). Ni siquiera posee un comando de bash disponible.
FROM gcr.io/distroless/static-debian11

# Puente de red trans-contenedor: importamos nuestro binario desde el ente extinto worker_builder
COPY --from=worker_builder /worker_kernel_daemon /bin/nuclear_daemon

USER nonroot:nonroot
CMD ["/bin/nuclear_daemon"]
