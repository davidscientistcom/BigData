Arquitectura de Flink

Cómo está construido un clúster Flink: JobManager, TaskManagers, paralelismo, checkpoints y tolerancia a fallos.

1. Visión General del Clúster

Un clúster de Flink tiene tres componentes principales que trabajan juntos para procesar datos de forma distribuida:

Arquitectura de un clúster Apache Flink

2. JobManager: el Cerebro

El JobManager es el nodo coordinador del clúster. Solo hay uno activo a la vez (en modo HA puede haber réplicas en standby). Sus responsabilidades son:

📋

Job Scheduling

Recibe el grafo de ejecución del job y lo distribuye entre los TaskManagers disponibles, asignando tareas a slots.

💾

Checkpoint Coordinator

Coordina la creación periódica de checkpoints para garantizar la tolerancia a fallos y la recuperación.

🌐

REST API & Web UI

Ofrece una interfaz web y API REST para monitorizar jobs, ver métricas y gestionar el clúster.

3. TaskManagers: los Trabajadores

Los TaskManagers son los nodos trabajadores. Pueden ser decenas o cientos en producción. Cada uno ejecuta las tareas asignadas por el JobManager.

Cada TaskManager tiene un número fijo de slots: unidades de trabajo paralelo. Un slot puede ejecutar una tarea del job. El número de slots determina el paralelismo máximo del clúster.

💡
Analogía: El JobManager es el director de una orquesta. Los TaskManagers son los músicos. Los slots son los instrumentos. El JobManager decide quién toca qué parte, y los músicos ejecutan la partitura en paralelo.

4. Paralelismo

El paralelismo define cuántas instancias paralelas de un operador se ejecutan simultáneamente. Más paralelismo = más velocidad de procesamiento (hasta el límite del hardware).

java — Configurar paralelismo en Flink 
StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();

// Paralelismo global para todo el job
env.setParallelism(4);

DataStream<String> stream = env
    .addSource(source)           // paralelismo = 4
    .map(x -> x.toUpperCase())  // paralelismo = 4
    .filter(x -> x.length() > 3) // paralelismo = 4
    .setParallelism(2)           // ← solo este operador con 2 instancias
    .print();

env.execute("Mi Job Flink");

5. Checkpoints y Tolerancia a Fallos

Los checkpoints son la clave de la tolerancia a fallos en Flink. Periódicamente, Flink guarda el estado completo del job en almacenamiento duradero (HDFS, S3). Si algo falla, Flink puede retomar desde el último checkpoint.

1
Flink inserta "barreras de checkpoint" en el stream

Son marcas especiales que fluyen a través del grafo de operadores indicando dónde empieza un checkpoint.

2
Cada operador guarda su estado al recibir la barrera

ValueState, ListState, MapState... todo se serializa y guarda en el backend de estado configurado.

3
El JobManager confirma el checkpoint completo

Cuando todos los operadores han guardado su estado, el checkpoint se considera exitoso.

4
En caso de fallo, se restaura desde el último checkpoint

Flink reinicia los TaskManagers y restaura el estado desde el checkpoint, retomando el procesamiento sin perder datos.

java — Configurar checkpoints 
// Activar checkpoints cada 30 segundos
env.enableCheckpointing(30_000);

// Configuración avanzada
env.getCheckpointConfig()
    .setCheckpointingMode(CheckpointingMode.EXACTLY_ONCE);

env.getCheckpointConfig()
    .setMinPauseBetweenCheckpoints(5_000); // mín 5s entre checkpoints

env.getCheckpointConfig()
    .setCheckpointTimeout(60_000); // timeout de 60s

// Usar RocksDB como backend de estado (para grandes estados)
env.setStateBackend(new RocksDBStateBackend("hdfs://namenode:8020/flink/checkpoints"));

6. Garantías de Procesamiento

Garantía Descripción Cuándo usarla
At-most-once Cada evento se procesa como máximo una vez. Puede haber pérdidas. Logs de baja criticidad
At-least-once Cada evento se procesa al menos una vez. Puede haber duplicados. Cuando los duplicados son aceptables
Exactly-once Cada evento se procesa exactamente una vez. Sin pérdidas ni duplicados. Fraude, pagos, inventarios
⚡
Exactly-once en Flink: Es la garantía más potente. Flink la consigue combinando checkpoints con transacciones en los sinks (ej: Kafka transaccional). El dato se procesa una vez y se escribe una vez, incluso si el sistema falla y se reinicia.
