SQL Relacional en Tiempo Real

Entendiendo Flink SQL

El DataStream API es potente, pero a veces escribir código en Java o Python es excesivo. Flink ofrece la Table API y Flink SQL, permitiéndote escribir consultas SQL estándar que se ejecutan continuamente sobre flujos infinitos de datos.

¿Por qué SQL para Streaming?

SQL es el lenguaje universal para manipular datos. Originalmente fue diseñado para bases de datos estáticas (datos en reposo), pero Flink ha conseguido adaptarlo para procesar datos en movimiento (streams).

🎯
La magia de Flink SQL: Escribes la misma consulta SQL que harías en PostgreSQL o MySQL, pero Flink se encarga de que devuelva resultados que se van actualizando en tiempo real a medida que entran nuevos eventos.

Las ventajas principales

  • Declarativo: Dices qué quieres (ej: "suma de ventas por país"), no cómo calcularlo (usando bucles, estados, triggers...).
  • Optimizador integrado: Flink tiene un motor llamado Catalyst (heredado de Calcite) que optimiza las consultas SQL automáticamente antes de ejecutarlas, a menudo logrando mejor rendimiento que el código manual.
  • Accesibilidad: Accesible para Analistas de Datos y Data Engineers sin experiencia en programación funcional recursiva de bajo nivel.
  • Unificación Batch/Stream: La misma consulta sirve para procesar meses históricos (batch) y para conectarse al topic de Kafka en directo (stream) sin tocar una coma.

El Concepto Clave: Tablas Dinámicas

En una base de datos normal, haces una consulta sobre los datos que existen en ese exacto momento y obtienes un resultado estático. ¿Pero cómo aplicamos SQL a una cola infinita de mensajes (como Apache Kafka)?

Flink resuelve esto introduciendo las Tablas Dinámicas (Dynamic Tables) y las Consultas Continuas (Continuous Queries).

¿Qué es el Changelog Stream (Retracciones)?

Imagina que escribes esta consulta SQL en Flink:

SELECT pais, COUNT(*) AS visitas FROM Clicks GROUP BY pais;

A medida que entran clicks en tiempo real, el conteo por país cambia constantemente. Para emitir este resultado a un sistema externo (como otro topic de Kafka o una base de datos PostgreSQL), Flink genera un Stream de Cambios que no solo "inserta" eventos, sino que tiene mensajes especiales de Retracción (Delete/Update Antes) y mensajes de Inserción (Update Después).

  • Append-only stream: Si haces un simple SELECT * WHERE pais='ES', Flink simplemente empuja cada evento válido hacia adelante. Nunca cambia eventos anteriores.
  • Retract stream: Si usas GROUP BY, COUNT o SUM, cuando llega un nuevo evento Flink "deshace" el resultado anterior enviando un mensaje -U (Update Before) y envía el nuevo con +U (Update After).

El Tiempo en SQL (Time Attributes)

Cuando trabajamos en SQL con datos estáticos, agrupar por mes o año es sencillo porque las fechas ya están fijas en una columna varchar o timestamp. En Streaming, el Tiempo dicta cuándo comienza y cierra una "ventana temporal" sobre el flujo infinito.

Para hacer agrupaciones de tiempo (como "cuenta de clicks de los últimos 5 minutos"), Flink SQL requiere que declares una columna especial en la creación de tu tabla (DDL) que actúe como Time Attribute.

Hay dos formas de definir el tiempo en Flink SQL:

1. Event Time (Tiempo del Evento)

El tiempo real en que ocurrió la acción en el dispositivo o aplicación de origen, extraído del propio objeto JSON o mensaje. Requiere el uso de Watermarks.

CREATE TABLE logs (
  usuario_id INT,
  evento STRING,
  tiempo_accion TIMESTAMP(3),
  WATERMARK FOR tiempo_accion AS tiempo_accion - INTERVAL '5' SECOND
) ...

Aquí le decimos a Flink SQL que use 'tiempo_accion' como Event Time y que tolere retrasos (desorden de red) de hasta 5 segundos.

2. Processing Time (Tiempo de Procesamiento)

El momento exacto en el servidor en el que el registro es ingerido por Flink. Se usa cuando el tiempo del dispositivo no importa, se genera artificialmente añadiendo una pseudo-columna.

CREATE TABLE logs (
  usuario_id INT,
  evento STRING,
  tiempo_proc AS PROCTIME() -- pseudo columna mágica
) ...

¿Cómo usar Flink SQL en la práctica?

A diferencia de escribir código Java/Python, Flink ofrece clientes interactivos para ejecutar SQL directamente sin compilar archivos JAR.

  • 1. Flink SQL Client (CLI): Viene integrado con Flink. Abres la terminal, escribes ./bin/sql-client.sh y se abre una consola similar a la de MySQL donde escribes CREATE TABLE y SELECT, viendo el resultado en tiempo real en consola.
  • 2. Embebido en Java/Python: Puedes instanciar un entorno de tablas y pasar strings SQL por código. tableEnv.executeSql("SELECT * FROM ..."). Muy usado para pipelines en producción.
  • 3. Flink SQL Gateway / Plataformas (Confluent, Ververica): Ofrecen interfaces visuales web o endpoints REST para enviar consultas SQL a procesar.
