Operadores y Transformaciones

Todos los operadores de Flink con ejemplos reales: map, flatMap, filter, keyBy, reduce, aggregate, join y connect.

1. Operadores de Transformación Básicos

Estos operadores trabajan elemento a elemento: reciben un evento y devuelven cero, uno o varios eventos como resultado.

🔄

map

Transforma cada elemento en exactamente un elemento. 1 entrada → 1 salida. El operador más básico.

🔀

flatMap

Transforma cada elemento en cero o más elementos. 1 entrada → N salidas. Útil para expandir datos.

🔍

filter

Mantiene solo los elementos que cumplen una condición. Los que no la cumplen se descartan.

🔑

keyBy

Particiona el stream por clave. Todos los eventos con la misma clave van al mismo operador downstream.

📉

reduce

Combina el elemento actual con el estado acumulado. Ej: suma running, mínimo o máximo.

📊

aggregate

Agrega elementos en una ventana temporal. Más flexible que reduce: separación entre acumulador y resultado.

java — map, flatMap y filter en acción 
DataStream<String> rawStream = env.fromElements(
    "Hola Mundo", "Apache Flink", "Big Data", "ok"
);

// map: convertir a mayúsculas (1 → 1)
DataStream<String> upper = rawStream.map(String::toUpperCase);
// ["HOLA MUNDO", "APACHE FLINK", "BIG DATA", "OK"]

// flatMap: dividir en palabras (1 → N)
DataStream<String> words = rawStream.flatMap(
    (line, out) -> Arrays.stream(line.split(" ")).forEach(out::collect)
);
// ["Hola", "Mundo", "Apache", "Flink", "Big", "Data", "ok"]

// filter: solo palabras largas (filtro)
DataStream<String> longWords = words.filter(w -> w.length() > 3);
// ["Hola", "Mundo", "Apache", "Flink", "Data"]
python — map, flatMap y filter en PyFlink 
# map: convertir a mayúsculas
upper = raw_stream.map(lambda x: x.upper(), output_type=Types.STRING())

# flatMap: dividir en palabras
def split_words(line):
    for word in line.split(" "):
        yield word

words = raw_stream.flat_map(split_words, output_type=Types.STRING())

# filter: solo palabras largas
long_words = words.filter(lambda w: len(w) > 3)

2. keyBy: el Operador Más Importante

El operador keyBy es crítico: agrupa el stream por una clave para que puedes aplicar operaciones stateful (con estado) por clave independientemente.

java — keyBy para contar ventas por categoría de producto 
DataStream<Sale> sales = ...;  // stream de ventas

// Agrupar por categoría → estado independiente por cada categoría
sales
    .keyBy(Sale::getCategory)   // "electronics", "books", "food"...
    .window(TumblingEventTimeWindows.of(Time.hours(1)))
    .aggregate(new SalesAggregator())
    .print();

// Resultado cada hora, por categoría:
// {"category": "electronics", "total": 15420.50, "count": 234}
// {"category": "books", "total": 3210.00, "count": 89}
python — keyBy en PyFlink 
# Agrupar por categoría (accediendo a índice de tupla o diccionario)
sales \
    .key_by(lambda sale: sale['category']) \
    .window(TumblingEventTimeWindows.of(Time.hours(1))) \
    .reduce(suma_ventas_function) \
    .print()
⚠️
Importante: Después de keyBy todos los eventos con la misma clave van al mismo slot del TaskManager. Esto garantiza que el estado de esa clave es consistente. Si la distribución de claves es muy desigual (hot key problem), puede saturar un slot. En ese caso, hay que repensar la clave de particionado.

3. reduce y aggregate: acumulaciones

java — reduce vs aggregate 
// REDUCE: acumula combinando el elemento anterior con el actual
// Suma running de ventas por usuario
sales
    .keyBy(Sale::getUserId)
    .reduce((sale1, sale2) -> {
        return new Sale(sale1.getUserId(), sale1.getAmount() + sale2.getAmount());
    });

// AGGREGATE: más flexible, separa acumulador y resultado final
public class AvgAggregate implements AggregateFunction<Sale, Tuple2<Double, Long>, Double> {

    public Tuple2<Double, Long> createAccumulator() {
        return Tuple2.of(0.0, 0L);  // (suma, contador)
    }

    public Tuple2<Double, Long> add(Sale sale, Tuple2<Double, Long> acc) {
        return Tuple2.of(acc.f0 + sale.getAmount(), acc.f1 + 1);
    }

    public Double getResult(Tuple2<Double, Long> acc) {
        return acc.f0 / acc.f1;  // media = suma / contador
    }

    public Tuple2<Double, Long> merge(Tuple2<Double, Long> a, Tuple2<Double, Long> b) {
        return Tuple2.of(a.f0 + b.f0, a.f1 + b.f1);
    }
}
python — reduce vs aggregate en PyFlink 
# REDUCE en PyFlink
sales \
    .key_by(lambda x: x['user_id']) \
    .reduce(lambda a, b: {
        'user_id': a['user_id'],
        'amount': a['amount'] + b['amount']
    })

# AGGREGATE personalizado en Python
class AvgAggregateFunction(AggregateFunction):
    def create_accumulator(self):
        return (0.0, 0)  # suma, contador

    def add(self, sale, acc):
        return (acc[0] + sale['amount'], acc[1] + 1)

    def get_result(self, acc):
        return acc[0] / acc[1] if acc[1] != 0 else 0

    def merge(self, a, b):
        return (a[0] + b[0], a[1] + b[1])

4. Joins: combinar dos streams

Flink permite hacer join entre dos streams dentro de una ventana temporal. Muy útil para enriquecer eventos con información de otro stream.

java — Join: enriquecer transacciones con datos de usuario 
DataStream<Transaction> transactions = ...;
DataStream<UserProfile>  profiles     = ...;

// Join: unir transacciones con perfiles en ventana de 5 minutos
transactions
    .join(profiles)
    .where(Transaction::getUserId)     // clave del stream izquierdo
    .equalTo(UserProfile::getUserId)  // clave del stream derecho
    .window(TumblingEventTimeWindows.of(Time.minutes(5)))
    .apply((tx, profile) -> new EnrichedTransaction(
        tx.getId(),
        tx.getAmount(),
        profile.getRiskScore(),
        profile.getCountry()
    ))
    .print();
python — Join en PyFlink 
def enrich_transaction(tx, profile):
    return {
        'id': tx['id'],
        'amount': tx['amount'],
        'risk': profile['risk_score'],
        'country': profile['country']
    }

transactions \
    .join(profiles) \
    .where(lambda tx: tx['user_id']) \
    .equal_to(lambda profile: profile['user_id']) \
    .window(TumblingEventTimeWindows.of(Time.minutes(5))) \
    .apply(enrich_transaction) \
    .print()

5. connect y CoProcessFunction

A diferencia del join, connect une dos streams de tipos diferentes sin ventana. Muy útil para implementar lógica de control dinámica: cargar reglas de fraude desde un stream y aplicarlas al stream de transacciones.

java — connect: aplicar reglas dinámicas a transacciones 
DataStream<Transaction>  txStream    = ...;
DataStream<FraudRule>    rulesStream = ...;  // reglas que cambian en tiempo real

txStream
    .connect(rulesStream)
    .keyBy(Transaction::getUserId, FraudRule::getUserId)
    .process(new CoProcessFunction<>() {
        private ListState<FraudRule> rules;

        // Cuando llega una nueva regla, la guardamos en estado
        public void processElement2(FraudRule rule, Context ctx, ...) {
            rules.add(rule);
        }

        // Cuando llega una transacción, aplicamos las reglas actuales
        public void processElement1(Transaction tx, Context ctx, ...) {
            for (FraudRule rule : rules.get()) {
                if (rule.matches(tx)) {
                    out.collect(new Alert(tx, rule));
                }
            }
        }
    });
python — connect y CoProcessFunction en PyFlink 
class RuleMatcher(CoProcessFunction):
    def __init__(self):
        self.rules_state = None

    def open(self, runtime_context):
        self.rules_state = runtime_context.get_list_state(
            ListStateDescriptor("rules", Types.PICKLED_BYTE_ARRAY())
        )

    def process_element1(self, tx, ctx, out):
        # Procesar transacción con las reglas almacenadas
        for rule in self.rules_state.get():
            if evaluate_rule(rule, tx):
                out.collect({"tx": tx, "rule": rule, "alert": True})

    def process_element2(self, rule, ctx, out):
        # Guardar la nueva regla
        self.rules_state.add(rule)

tx_stream \
    .connect(rules_stream) \
    .key_by(lambda tx: tx['user_id'], lambda rule: rule['target_user']) \
    .process(RuleMatcher())

6. Caso Práctico Completo: Detección de Fraude en Tiempo Real

Vamos a construir un caso realista en PyFlink donde analizamos un flujo de transacciones bancarias. Si un usuario realiza una transacción pequeña de prueba (< 1€) seguida de una transacción grande (> 10,000€) en menos de 1 minuto, generaremos una alerta de fraude. Usaremos el poderoso KeyedProcessFunction para manejar el estado y los temporizadores (timers).

python — PyFlink: Detección de Fraude con ProcessFunction 
from pyflink.datastream import StreamExecutionEnvironment
from pyflink.datastream.functions import KeyedProcessFunction
from pyflink.datastream.state import ValueStateDescriptor
from pyflink.common.typeinfo import Types

class FraudDetector(KeyedProcessFunction):
    def open(self, runtime_context):
        # Estado 1: marca si hemos visto una transacción muy pequeña
        self.flag_state = runtime_context.get_state(
            ValueStateDescriptor("small_tx_flag", Types.BOOLEAN())
        )
        # Estado 2: guarda el temporizador para limpiarlo si es necesario
        self.timer_state = runtime_context.get_state(
            ValueStateDescriptor("timer", Types.LONG())
        )

    def process_element(self, transaction, ctx, out):
        last_flag = self.flag_state.value()

        # Si el flag está activo (hubo tx pequeña) y esta tx es grande (>10k)...
        if last_flag and transaction['amount'] > 10000:
            # Emitir la alerta al stream de salida
            alert = f"🚨 FRAUDE DETECTADO - Usuario: {ctx.get_current_key()}"
            out.collect(alert)
            self.clean_up(ctx)  # Resetear estado

        # Si la transacción es pequeña, activamos el flag y un contador
        if transaction['amount'] < 1.00:
            self.flag_state.update(True)
            
            # Crear temporizador a 1 minuto vista desde ahora
            timer = ctx.timer_service().current_processing_time() + (60 * 1000)
            ctx.timer_service().register_processing_time_timer(timer)
            self.timer_state.update(timer)

    def on_timer(self, timestamp, ctx, out):
        # Si se ejecuta este método, significa que pasó 1 minuto y no hubo fraude
        self.clean_up(ctx)

    def clean_up(self, ctx):
        self.flag_state.clear()
        timer = self.timer_state.value()
        if timer is not None:
            ctx.timer_service().delete_processing_time_timer(timer)
        self.timer_state.clear()

# Flujo principal del Job
env = StreamExecutionEnvironment.get_execution_environment()
# Supongamos 'transactions' viene de Kafka ya parseado...
transactions = ... 

alerts = transactions \
    .key_by(lambda tx: tx['user_id']) \
    .process(FraudDetector(), output_type=Types.STRING())

alerts.print()
env.execute("Mejor Detector de Fraude Flink")

7. Tabla resumen de Operadores

Operador Entradas → Salidas Uso típico
map() 1 → 1 Transformar formato, extraer campo, enriquecer
flatMap() 1 → N Dividir en palabras, expandir arrays, normalizar
filter() 1 → 0 o 1 Filtrar por condición, descartar eventos no relevantes
keyBy() shuffle Particionar por clave para stateful operations
reduce() acumulación Suma, min, max running por clave
aggregate() ventana → 1 Estadísticas complejas (media, percentil) por ventana
join() 2 streams Unir streams con clave común en ventana temporal
connect() 2 streams distintos Lógica dinámica con reglas desde otro stream
union() N streams iguales Fusionar múltiples fuentes del mismo tipo
process() bajo nivel Acceso a timers, side outputs, lógica compleja
