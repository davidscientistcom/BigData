Problemas Reales

Problemas concretos del mundo real modelados como grafos. Cada uno con el planteamiento, el modelo en Neo4j y las preguntas que puedes responder.

🎯
Cómo usar esta sección: Para cada problema, primero intenta imaginar cómo lo modelarías como grafo (¿qué son los nodos? ¿qué son las aristas?) antes de ver la solución. Es el razonamiento más importante en proyectos de grafos reales.
🗺️

Problema 1 — Red Ferroviaria de España

Planificación de rutas óptimas entre ciudades con trenes de alta velocidad

Grafo ponderado Dirigido Dijkstra MST (Kruskal/Prim)

Situación: Renfe tiene estaciones en Madrid, Barcelona, Valencia, Sevilla, Bilbao y Zaragoza. Cada trayecto directo tiene un tiempo en minutos. No todas las ciudades tienen trayecto directo entre sí.

Red simplificada de AVE — tiempos en minutos
🚄 Caso real: Renfe y Google Maps usan exactamente este modelo. Las estaciones son nodos, las rutas directas son aristas ponderadas con tiempo/precio, y se usa Dijkstra para encontrar el viaje óptimo.

Preguntas:

  • ¿Cuál es la ruta más rápida de Madrid a Lleida?
  • Si quisiéramos conectar todas las ciudades con rutas AVE nuevas minimizando el total de km de vía a construir, ¿qué algoritmo usaríamos y por qué?
  • Si la ruta Madrid-Zaragoza se cierra por obras, ¿cómo afecta a los viajes a Bilbao?
  • Madrid → Lleida más rápida: Madrid→Zaragoza(90) → Bilbao(60) → Lleida(35) = 185 min u otras combinaciones. Necesitas Dijkstra para calcularlo con todos los pesos.
  • Minimizar km de vía: Kruskal o Prim (Árbol de Expansión Mínima). No hay que llegar al destino más rápido, sino conectar todos con el menor coste total de infraestructura.
  • Si cierran Madrid-Zaragoza: Para ir a Bilbao desde Madrid solo queda Madrid→Valencia→Barcelona→Lleida→Bilbao si existen esas conexiones, o Madrid→Sevilla(150)→... Puede que no haya ruta directa. BFS/Dijkstra sobre el grafo sin esa arista respondería.
cypher
// ¿Qué pasa si quitamos Madrid-Zaragoza? Ruta a Bilbao:
MATCH (o:Estacion {nombre: "Madrid"}),
      (d:Estacion {nombre: "Bilbao"})
MATCH p = shortestPath((o)-[r:RUTA*]-(d))
// Filtramos la arista cerrada por obras
WHERE NONE(rel IN relationships(p)
  WHERE (startNode(rel).nombre="Madrid"
    AND endNode(rel).nombre="Zaragoza"))
RETURN [n IN nodes(p) | n.nombre] AS ruta_alternativa

Problema 2 — Red de Influencers de Instagram

Detectar comunidades, influencers clave y propagar mensajes virales

Dígrafo PageRank Betweenness centrality Community detection

Una empresa quiere lanzar una campaña de marketing en Instagram. Tienen un grafo de 10.000 cuentas con relaciones de "sigue" entre ellas. Quieren maximizar el alcance de la campaña contratando solo 5 influencers.

Modelo del grafo:

  • Nodos: (:Cuenta {username, seguidores, categoria})
  • Aristas: -[:SIGUE {fecha}]-> (dirigido — A sigue a B no implica que B siga a A)
📱 Caso real: Twitter/X, Instagram y TikTok tienen equipos de Data Science que modelan su plataforma exactamente así. Detectar "comunidades" (grupos de cuentas muy conectadas entre sí) es clave para el algoritmo de recomendación de follows.

Preguntas y abordaje:

  • ¿Qué métrica usarías para identificar los 5 influencers más importantes? ¿PageRank o grado de entrada (in-degree)?
  • Dos influencers en la misma "comunidad" pueden tener audiencias solapadas. ¿Cómo detectaría Neo4j esas comunidades?
  • Si quieres que el mensaje llegue a los grupos más desconectados entre sí, ¿qué métrica de centralidad usas?
  • PageRank vs In-degree: PageRank es mejor: no solo cuenta cuántos te siguen, sino si quienes te siguen son a su vez influyentes. Un influencer con 50.000 seguidores muy activos puede valer más que uno con 500.000 bots.
  • Detección de comunidades: Algoritmo de Louvain (en Neo4j GDS: gds.louvain). Agrupa nodos que tienen muchas conexiones entre sí y pocas con el exterior.
  • Alcanzar grupos desconectados: Betweenness centrality. Los influencers puente conectan diferentes "burbujas" de Instagram y pueden llevar tu mensaje a comunidades que de otra forma quedarían fuera.
cypher — Detectar comunidades con Louvain
// Proyectar el grafo e identificar comunidades
CALL gds.louvain.stream('instagram-graph')
YIELD nodeId, communityId
WITH communityId, collect(gds.util.asNode(nodeId).username)
     AS miembros
RETURN communityId, size(miembros) AS tamaño,
       miembros[0..5] AS muestra
ORDER BY tamaño DESC LIMIT 10

Problema 3 — Cadena de Suministro de Amazon

Optimizar las rutas de entrega y detectar cuellos de botella en la red logística

Grafo ponderado Dirigido Dijkstra Betweenness MST

Amazon tiene almacenes, centros de distribución, y puntos de entrega final. Cada ruta tiene un coste (tiempo + distancia + coste de transporte). El objetivo es entregar en el mínimo tiempo minimizando costes.

Modelo del grafo:

  • Nodos tipos: (:Almacen), (:CentroDistribucion), (:PuntoEntrega)
  • Aristas: -[:ENVIO {tiempo_h, coste_eur, distancia_km}]->
  • Propiedades extra de nodos: capacidad máxima, stock disponible
📦 Caso real: Amazon, Mercadona, FedEx y UPS tienen departamentos enteros dedicados a grafos de logística. UPS estima que optimizar solo 1 km por conductor al día les ahorra 50 millones de dólares al año.

Preguntas y ejercicios:

  • ¿Qué algoritmo encuentra el camino más rápido de un almacén a varios puntos de entrega simultáneamente?
  • ¿Qué nodo del grafo es el "cuello de botella" más crítico? ¿Qué algoritmo lo detecta?
  • Si cae un centro de distribución (por inundación, huelga...), ¿cómo calculas rutas alternativas?
  • ¿Cómo detectarías si hay dependencias circulares en la cadena de suministro (A necesita B, B necesita C, C necesita A)?
  • Un almacén → múltiples destinos: Dijkstra de fuente única (single-source shortest path). En GDS: gds.allShortestPaths o Dijkstra con múltiples targets.
  • Cuello de botella crítico: Betweenness centrality — el nodo que aparece en más caminos mínimos. Si falla, más rutas se ven afectadas.
  • Ruta alternativa si cae un nodo: Eliminar ese nodo del grafo en memoria (nodeFilter en GDS) y recalcular shortest path. Neo4j ya tiene soporte para esto.
  • Ciclos en cadena de suministro: DFS para detectar ciclos en el dígrafo. En Cypher: MATCH path=(a)-[:NECESITA*]->(a)

Problema 4 — Detección de Fraude Bancario

Identificar redes de cuentas coordinadas para el blanqueo de dinero

Multigrafo Detección de ciclos Community detection Anomaly detection

Un banco tiene millones de transacciones entre cuentas. Quiere detectar patrones fraudulentos como dinero que circula en bucle entre cuentas, o grupos de cuentas aparentemente independientes que comparten información (mismo teléfono, misma IP, misma dirección).

Modelo multi-tipo:

  • (:Cuenta)-[:TRANSACCION {importe, fecha}]->(:Cuenta)
  • (:Cuenta)-[:USA_TELEFONO]->(:Telefono)
  • (:Cuenta)-[:USA_IP]->(:DireccionIP)
  • (:Cuenta)-[:TITULAR]->(:Persona)
🏦 Caso real: BBVA, Santander y PayPal usan Neo4j específicamente para detección de fraude. Lo que en SQL requeriría horas de JOINs anidados, Neo4j lo resuelve en segundos navegando el grafo.

Patrones fraudulentos a detectar:

cypher — Patrón 1: Dinero en círculo (Money Mule Ring)
// Detectar ciclos de transacciones entre 3 y 5 cuentas
MATCH path = (c:Cuenta)-[:TRANSACCION*3..5]->(c)
WHERE ALL(r IN relationships(path)
  WHERE r.importe > 5000)  // solo transacciones grandes
RETURN [n IN nodes(path) | n.numero] AS anillo,
       length(path) AS pasos
cypher — Patrón 2: Cuentas que comparten identidad
// Cuentas conectadas por teléfono, IP o persona — sospechosa red
MATCH (c1:Cuenta)-[:USA_TELEFONO|USA_IP|TITULAR]->
      (elemento)
      <-[:USA_TELEFONO|USA_IP|TITULAR]-(c2:Cuenta)
WHERE c1 <> c2
  AND c1.estado = "activa"
  AND c2.estado = "activa"
RETURN c1.numero, c2.numero,
       type(elemento) AS elemento_compartido,
       elemento.valor

Ejercicio: Diseña el modelo

Un banco tiene los siguientes datos en CSV: transacciones con importe/fecha/origen/destino, listado de clientes con su DNI/email/teléfono, y listado de IPs de acceso con fecha. ¿Cómo montarías el modelo de grafo? ¿Qué nodos, qué aristas, qué propiedades? ¿Qué patrones buscarías primero?

Nodos: :Cuenta, :Cliente, :Telefono, :Email, :DireccionIP

Relaciones:

  • -[:TRANSACCION {importe, fecha, concepto}]-> entre Cuentas
  • -[:TITULAR]-> entre Cuenta y Cliente
  • -[:USA_TELEFONO]-> entre Cliente y Telefono
  • -[:ACCESO_DESDE {fecha}]-> entre Cuenta y DireccionIP

Patrones prioritarios:

  • Ciclos de 3-7 cuentas con transacciones altas → blanqueo
  • Cuentas distintas con mismo teléfono/email → identidades robadas
  • Múltiples cuentas accediendo desde la misma IP → un operador controlando varias cuentas
  • Componentes aisladas que de repente se conectan → mulas de dinero nuevas

Problema 5 — Sistema de Recomendación tipo Netflix

Recomendar contenido usando patrones de visualización en un grafo bipartito

Grafo bipartito Collaborative filtering Similitud de coseno Link prediction

Netflix, Spotify y Amazon Prime tienen millones de usuarios y contenidos. El grafo representa quién vio qué. El objetivo es predecir qué contenido querrá ver un usuario que aún no ha visto.

Modelo:

  • Nodos: (:Usuario {id, pais}) y (:Pelicula {titulo, genero, año})
  • Aristas: -[:VIO {rating, fecha, completada}]->
Grafo bipartito Usuario-Película. ¿Qué debería ver Ana?

La lógica de recomendación:

  • Ana y Luis vieron Inception e Interstellar → tienen gustos similares
  • Sara y Luis vieron Oppenheimer e Inception
  • Luis vio Tenet (que Ana no ha visto) → recomendarla a Ana tiene sentido
  • En grafos: encontrar nodos similares a Ana (mismos vecinos-películas) y mirar qué más han visto
cypher — Collaborative Filtering básico
// Para Ana: recomendar lo que vieron usuarios con gustos similares
MATCH (ana:Usuario {nombre: "Ana"})-[:VIO]->(pelicula:Pelicula)
                              <-[:VIO]-(similar:Usuario)
                              -[:VIO]->(recomendacion:Pelicula)
WHERE NOT (ana)-[:VIO]->(recomendacion)
RETURN recomendacion.titulo,
       recomendacion.genero,
       count(*) AS popularidad_entre_similares
ORDER BY popularidad_entre_similares DESC
LIMIT 10

Problema 6 — Monitorización de Infraestructura IT

Análisis de impacto: si un servidor falla, ¿qué servicios se caen?

Dígrafo DFS Componentes Análisis de impacto

Una empresa tiene servidores, servicios web, bases de datos y APIs. Cada servicio depende de otros (bases de datos, servicios de autenticación, etc.). Si un servidor falla, ¿qué servicios en cascada dejan de funcionar?

Modelo:

  • (:Servidor {nombre, ip, datacenter})
  • (:Servicio {nombre, tipo, SLA})
  • -[:EJECUTA_EN]-> (Servicio → Servidor)
  • -[:DEPENDE_DE {critico: true/false}]-> (Servicio → Servicio)
cypher — Análisis de impacto en cascada
// ¿Qué servicios se caen si el servidor "prod-db-01" falla?
MATCH (servidor:Servidor {nombre: "prod-db-01"})
MATCH (servidor)<-[:EJECUTA_EN]-(servicio_caido:Servicio)
MATCH (servicio_caido)<-[:DEPENDE_DE*1..5]-(afectado:Servicio)
RETURN DISTINCT afectado.nombre AS servicio_afectado,
       afectado.SLA AS criticidad,
       afectado.propietario AS equipo_a_alertar
ORDER BY afectado.SLA DESC
🏢 Caso real: LinkedIn usa Neo4j para exactamente esto en su infraestructura interna. Cuando van a hacer un despliegue, consultan el grafo de dependencias para saber qué equipos deben estar en guardia.

Ejercicio de diseño:

Imagina que tienes una app web con: frontend (React), API REST (Node.js), base de datos PostgreSQL, servicio de autenticación (OAuth), y servicio de emails (SendGrid). Dibuja el grafo de dependencias. ¿Cuál es el servicio más crítico (el que más cosas dependen de él)? ¿Qué pasaría si OAuth falla?

Resumen: Qué algoritmo para qué problema

Problema Nodos Aristas Algoritmo/Enfoque
Ruta más rápida A→B con pesos Ciudades, nodos de red Conexiones con distancia/tiempo Dijkstra / A*
Grados de separación Personas Relaciones sociales BFS
Red mínima de infraestructura Ciudades, edificios Posibles conexiones Kruskal / Prim
Influencers más importantes Cuentas, páginas web Follows, enlaces PageRank
Puentes críticos de la red Cualquier entidad Cualquier relación Betweenness Centrality
Detectar comunidades / clusters Usuarios, cuentas Interacciones Louvain, Label Propagation
Detección de fraude / ciclos Cuentas, transacciones Transferencias DFS + detección de ciclos
Recomendaciones Usuarios + Items Interacciones (visto, comprado) Collaborative Filtering
Análisis de impacto IT Servicios, servidores Dependencias DFS + alcanzabilidad
