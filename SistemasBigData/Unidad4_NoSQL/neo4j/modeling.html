Módulo 4: Modelado de Grafos

La filosofía, las decisiones y los patrones que separan un grafo eficiente del caos. Modelar para las consultas, no para la normalización.

1. La Filosofía del Modelado en Grafos

En bases de datos relacionales modelas para normalización: sin redundancia, tercera forma normal. En grafos modelas para las consultas que vas a ejecutar. El objetivo primario es la traversal eficiente, no la pureza académica.

🎯

Primero las preguntas

¿Cuáles son las 5 consultas más importantes que tu sistema hará? El modelo debe hacer esas consultas triviales.

↔️

Relaciones son ciudadanos de primera

En tablas, las relaciones son una penalización (JOIN). En grafos, son traversal O(1) por nodo, completamente gratuitas.

🔄

Desnormalización controlada

Duplicar una propiedad para evitar un MATCH adicional es legítimo. La redundancia al servicio del rendimiento es una virtud.

🌍

El dominio es el grafo

Las entidades del mundo real se convierten en nodos. Las acciones y relaciones se convierten en edges con tipo.

2. Metodología Paso a Paso

proceso — Cómo modelar un dominio nuevo
1. Identificar entidades (candidatos a nodo)
   → Sustantivos del dominio: Usuario, Vehiculo, Sensor, Paper, Producto

2. Identificar relaciones (candidatos a edge)
   → Verbos del dominio: CONDUCE, EQUIPADO_CON, PUBLICÓ, COMPRÓ

3. Identificar propiedades de identidad
   → ¿Qué hace único a cada nodo? email, doi, sku, vin, id

4. Definir las 5-10 queries principales del sistema
   → "Dame los vehículos con LIDAR a menos de 50km del accidente X"
   → "Papers del investigador Y con más de 100 citas"

5. Evaluar cada atributo: ¿propiedad o nodo?

6. Identificar los antipatrones

7. Crear el Schema Diagram (nodos, labels, rel types, propiedades key)

3. Árbol de Decisión: ¿Propiedad, Nodo o Relación?

Cuando modelas datos en LPG, cada pieza de información puede vivir en tres lugares distintos: como propiedad dentro de un nodo, como un nodo propio, o como una relación (o propiedad de una relación). La elección no es estética — afecta directamente al rendimiento de las consultas, a la capacidad de búsqueda y a la consistencia del grafo. El árbol de decisión siguiente fuerza las preguntas correctas en el orden correcto.

Árbol de Decisión — ¿Dónde va este dato?

Cómo leer el árbol

El árbol fuerza cuatro preguntas en secuencia. Seguirlas en orden evita los errores clásicos de modelado:

  1. ¿Tiene identidad propia? Si el dato es una entidad del mundo — una persona, una ciudad, un sensor, un documento — merece ser nodo. Si es solo un atributo descriptivo de otra entidad, no.
  2. ¿Lo comparten múltiples nodos o tiene propiedades propias? Si varios vehículos tienen marca: "Toyota", Toyota es un nodo. Si cada nodo tiene su propio valor único e irrepetible (un VIN, un timestamp), probablemente es propiedad.
  3. ¿Es una conexión entre dos entidades? Si el dato dice "A hizo algo con B", es una relación. Si ese "algo" tiene contexto propio (cuándo, cuánto, cómo), ese contexto son propiedades de la relación.
  4. ¿Necesitas buscar o filtrar por él? Todo atributo sobre el que vayas a hacer WHERE o necesites indexar debe ser propiedad explícita — no puede estar enterrado en un string JSON ni en un array sin índice.

Los tres destinos con ejemplos paralelos

El mismo dominio — vehículos autónomos — modelado de las tres formas posibles:

cypher — El mismo dato en sus tres formas posibles
-- 1. PROPIEDAD: el fabricante como string dentro del vehículo
--    Cuándo aplica: solo hay unos pocos fabricantes y nunca los consultas por separado
(v:Vehiculo {vin: "1HGCM826", fabricante: "Toyota", autonomia_km: 450})

-- 2. NODO: el fabricante como entidad propia con relación
--    Cuándo aplica: múltiples vehículos comparten fabricante,
--    o necesitas datos del fabricante (sede, certificaciones, modelos)
(v:Vehiculo {vin: "1HGCM826"})-[:FABRICADO_POR]->(f:Fabricante {nombre: "Toyota", pais: "Japón"})

-- 3. RELACIÓN con propiedades: la asignación del vehículo a una flota
--    Cuándo aplica: la conexión tiene contexto propio (desde cuándo, con qué rol)
(v:Vehiculo)-[:ASIGNADO_A {desde: "2024-01-01", rol: "reparto_urbano"}]->(f:Flota {nombre: "Madrid-Centro"})

-- Caso de fallo típico: meter la relación como propiedad string
-- MAL: (v:Vehiculo {flota: "Madrid-Centro", asignado_desde: "2024-01-01"})
-- Si un vehículo cambia de flota, ¿borras el dato anterior? ¿Guardas historial?
-- Con una relación [:ASIGNADO_A] puedes tener múltiples con timestamps distintos.
WARN — La regla de los tres "nunca":
  • Nunca guardes en una propiedad un valor que actualmentees o será compartido por múltiples nodos — crea un nodo.
  • Nunca metas en una propiedad un JSON o array que necesites recorrer con lógica — crea nodos y relaciones.
  • Nunca uses una relación genérica [:RELACIONADO_CON] cuando el tipo del vínculo tiene significado — el tipo de relación es parte del modelo, no decoración.

4. Patrones Canónicos

4.1 Nodo Intermedio (Hyperedge)

Cuando una relación entre dos entidades necesita propiedades propias, o cuando la misma relación se da múltiples veces con contexto diferente.

cypher — Nodo intermedio de autoría
-- MAL: relación con demasiadas propiedades
(inv)-[:PUBLICÓ {ano: 2025, rol: "first_author", revision_score: 4.5}]->(p:Paper)

-- BIEN: nodo intermedio que captura la autoría completa
(inv:Investigador)-[:ES_AUTOR]->(a:Autoria)-[:DE_PAPER]->(p:Paper)
SET a.rol = "first_author", a.orden = 1, a.revision_score = 4.5

-- Ahora puedes: "dame las autorías de Deivit con score > 4"
MATCH (inv:Investigador {nombre: "Deivit"})-[:ES_AUTOR]->(a:Autoria)-[:DE_PAPER]->(p:Paper)
WHERE a.revision_score > 4
RETURN p.titulo, a.rol, a.revision_score

4.2 Time Tree

cypher — Time Tree
-- Jerarquía temporal: Año → Mes → Día
(:Año {valor: 2026})-[:TIENE_MES]->(:Mes {valor: 3})-[:TIENE_DIA]->(:Dia {valor: 26})

-- Accidentes de Q1 2026 por mes
MATCH (año:Año {valor: 2026})-[:TIENE_MES]->(mes:Mes)
WHERE mes.valor IN [1, 2, 3]
MATCH (mes)-[:TIENE_DIA]->(:Dia)<-[:OCURRIÓ_EL]-(acc:Accidente)
RETURN mes.valor, count(acc) AS accidentes_por_mes

4.3 Versioning / Linked List

cypher — Historial de ubicaciones como linked list
(v:Vehiculo {id: "AV-001"})-[:UBICACION_ACTUAL]->(ub:Ubicacion {ts: datetime()})
(ub)-[:ANTERIOR]->(ub_prev:Ubicacion)

-- Últimas 5 ubicaciones
MATCH (v:Vehiculo {id: "AV-001"})-[:UBICACION_ACTUAL]->(ub:Ubicacion)
MATCH cadena = (ub)-[:ANTERIOR*0..4]->(hist:Ubicacion)
RETURN hist.latitud, hist.longitud, hist.ts ORDER BY hist.ts DESC

5. Antipatrones Críticos

Un antipatrón en modelado de grafos es una decisión que parece razonable al principio pero que degrada el rendimiento, dificulta las consultas o introduce inconsistencias a medida que el grafo crece. Los tres siguientes son los más frecuentes en producción.

5.1 Fan-out — El Super Nodo

Ocurre cuando un nodo acumula un número enorme de relaciones — cientos de miles o millones. El motor Neo4j, al hacer un traversal desde ese nodo, tiene que cargar y recorrer todas sus relaciones aunque solo necesites una fracción. El coste deja de ser O(1) y se convierte en O(grado del nodo).

Fan-out — Super Nodo vs modelo corregido
cypher — detectar super nodos en tu grafo
-- Nodos con más de 100k relaciones: candidatos a super nodo
MATCH (n)
WHERE size((n)--()) > 100000
RETURN labels(n), n.nombre, size((n)--()) AS grado
ORDER BY grado DESC LIMIT 10

5.2 Relaciones sin tipo semántico

Una relación cuyo tipo es :RELACIONADO_CON, :CONECTADO_A o :TIENE no aporta información al modelo. El tipo de relación en LPG es parte del esquema — es el vocabulario con el que el grafo describe el mundo. Perderlo obliga a guardar el "tipo real" como propiedad, lo que hace las consultas más lentas e ilegibles.

Relación genérica vs relación semántica

5.3 JSON embebido en propiedades

Guardar un objeto JSON como string dentro de una propiedad rompe el modelo de grafos. Neo4j no puede hacer índices sobre el contenido interno de ese string, no puede recorrerlo con Cypher y cualquier consulta sobre él requiere parseo en la capa de aplicación. Es meter una base de datos documental dentro de una de grafos — lo peor de ambos mundos.

JSON embebido vs estructura nativa de grafos
FAIL — Señales de que tienes JSON embebido: si en tu código de aplicación llamas a JSON.parse(nodo.propiedad) justo después de un MATCH, tienes un antipatrón de modelado. Esa lógica de parseo debería ser un traversal en Cypher.

Cómo se resuelve — y por qué MongoDB no es la respuesta

La solución no es mover ese JSON a MongoDB y enlazarlo desde Neo4j. Eso añade una dependencia de infraestructura, latencia de red entre dos bases de datos, y complejidad operativa para resolver lo que es simplemente un problema de modelado. El JSON ya contiene la estructura de grafo que necesitas — solo hay que extraerla.

La migración tiene tres pasos: parsear el JSON con APOC, crear los nodos y relaciones correspondientes, y eliminar la propiedad problemática.

cypher — Migración: extraer JSON embebido a nodos y relaciones
-- SITUACIÓN DE PARTIDA:
-- (:Paper {titulo: "GNN causales", autores: '["Ana","Bea"]', keywords: '["GNN","causal"]'})

-- PASO 1: migrar el array de autores a nodos :Persona con relación :ES_AUTOR_DE
MATCH (p:Paper)
WHERE p.autores IS NOT NULL
WITH p, apoc.convert.fromJsonList(p.autores) AS lista_autores
UNWIND lista_autores AS nombre_autor
MERGE (a:Persona {nombre: nombre_autor})
MERGE (a)-[:ES_AUTOR_DE]->(p)

-- PASO 2: migrar el array de keywords a nodos :Keyword con relación :TIENE_KEYWORD
MATCH (p:Paper)
WHERE p.keywords IS NOT NULL
WITH p, apoc.convert.fromJsonList(p.keywords) AS lista_kw
UNWIND lista_kw AS kw
MERGE (k:Keyword {valor: kw})
MERGE (p)-[:TIENE_KEYWORD]->(k)

-- PASO 3: eliminar las propiedades JSON ya migradas
MATCH (p:Paper)
WHERE p.autores IS NOT NULL
REMOVE p.autores, p.keywords

-- VERIFICAR: ahora los autores son traversables y los keywords indexables
MATCH (a:Persona)-[:ES_AUTOR_DE]->(p:Paper)-[:TIENE_KEYWORD]->(k:Keyword)
WHERE k.valor = "GNN"
RETURN p.titulo, collect(a.nombre) AS autores

Si el JSON tiene estructura anidada más compleja — por ejemplo un objeto con múltiples campos en lugar de un array plano — el proceso es el mismo pero usando apoc.convert.fromJsonMap para parsear el objeto y acceder a sus campos por clave.

cypher — Migrar JSON de objeto anidado a nodo con propiedades
-- SITUACIÓN: (:Sensor {id: "S-01", config: '{"frecuencia_hz": 10, "rango_m": 150, "modelo": "VLP-16"}'})
-- El config es un objeto, no un array. Puede ser un nodo propio si varios sensores comparten config.

MATCH (s:Sensor)
WHERE s.config IS NOT NULL
WITH s, apoc.convert.fromJsonMap(s.config) AS cfg
MERGE (c:ConfigSensor {modelo: cfg.modelo, frecuencia_hz: cfg.frecuencia_hz, rango_m: cfg.rango_m})
MERGE (s)-[:TIENE_CONFIG]->(c)
REMOVE s.config

-- Si la config es idéntica en múltiples sensores, MERGE los comparte automáticamente.
-- Si es única por sensor, considera aplanarla directamente como propiedades del sensor:
MATCH (s:Sensor)
WHERE s.config IS NOT NULL
WITH s, apoc.convert.fromJsonMap(s.config) AS cfg
SET s.frecuencia_hz = cfg.frecuencia_hz,
    s.rango_m       = cfg.rango_m,
    s.modelo        = cfg.modelo
REMOVE s.config

Cuándo SÍ usar MongoDB junto a Neo4j

Hay un caso legítimo para tener ambas bases de datos en el mismo sistema: cuando parte de tus datos son genuinamente documentales — texto libre largo, blobs de configuración que nunca se consultan por campos internos, o payloads raw de eventos — y otra parte son las relaciones entre esas entidades. En ese caso, el patrón correcto es:

  • En MongoDB: el documento completo con todos sus campos, incluyendo el contenido que no necesitas recorrer. Identificado por un _id.
  • En Neo4j: el nodo con solo los campos que participan en traversals o necesitan índice, más una propiedad mongo_id que actúa de referencia externa.
cypher — Referencia a documento MongoDB desde nodo Neo4j
-- Neo4j solo guarda lo que necesita para el grafo
(:Paper {
  doi:      "10.1234/gnn-causal",
  titulo:   "GNN causales en conducción",
  year:     2025,
  mongo_id: "64a3f2b1c89de12345678901"  -- apunta al documento completo en MongoDB
})

-- El traversal de relaciones vive 100% en Neo4j
MATCH (a:Autor)-[:ES_AUTOR_DE]->(p:Paper)-[:CITA]->(citado:Paper)
WHERE p.year > 2023
RETURN p.titulo, p.mongo_id, collect(citado.titulo) AS citas

-- En la capa de aplicación, con los mongo_id recuperados,
-- haces una segunda consulta a MongoDB para obtener el abstract completo,
-- las figuras, el PDF, etc.
-- Cada base de datos hace lo que mejor sabe hacer.
INFO — La regla de separación: si necesitas un campo para un MATCH, un WHERE o una relación → va en Neo4j. Si es solo para mostrarlo al usuario final y nunca participa en lógica de grafo → puede quedarse en MongoDB. Lo que nunca debes hacer es meter en Neo4j un JSON que luego parseas en la app para extraer el campo que sí necesitabas en el grafo.
cypher — Schema e-commerce y sus queries principales
-- NODOS
(:Cliente {id, nombre, email})-[:VIVE_EN]->(:Ciudad {nombre, pais})
(:Producto {sku, nombre, precio})-[:PERTENECE_A]->(:Categoria {nombre})
(:Pedido {id, fecha, estado, total})-[:HECHO_POR]->(:Cliente)
(:Pedido)-[:CONTIENE]->(lp:LineaPedido {cantidad, precio_unitario})-[:DE_PRODUCTO]->(:Producto)
(:Reseña {puntuacion, texto})-[:ESCRITA_POR]->(:Cliente)
(:Reseña)-[:SOBRE]->(:Producto)

-- Productos más comprados por vecinos de ciudad
MATCH (c:Cliente {email: "user@lab.es"})-[:VIVE_EN]->(ciudad:Ciudad)
MATCH (ciudad)<-[:VIVE_EN]-(vecino:Cliente)<-[:HECHO_POR]-(p:Pedido)
MATCH (p)-[:CONTIENE]->(lp:LineaPedido)-[:DE_PRODUCTO]->(prod:Producto)
RETURN prod.nombre, count(*) AS popularidad ORDER BY popularidad DESC LIMIT 10

-- Productos bien valorados en una categoría
MATCH (cat:Categoria {nombre: "Electrónica"})<-[:PERTENECE_A]-(p:Producto)
MATCH (r:Reseña)-[:SOBRE]->(p)
WITH p, avg(r.puntuacion) AS media
WHERE media > 4
RETURN p.nombre, media ORDER BY media DESC
