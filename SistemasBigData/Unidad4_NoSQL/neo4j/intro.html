Módulo 1: Fundamentos de Bases de Datos de Grafos

Por qué surgieron, cómo piensan, y por qué cambian todo lo que sabes sobre almacenamiento de datos.

1. Historia y motivación: el problema que SQL no puede resolver

Las bases de datos relacionales son una maravilla ingenieril. Llevan 50 años siendo el estándar, y con razón. Pero tienen un talón de Aquiles: las relaciones profundas. Cuando el valor de tus datos reside en cómo se conectan entre sí, el modelo relacional colapsa.

El problema del JOIN explosion

Imagina un grafo de fraude bancario: una cuenta → hace transacciones → a otras cuentas → que comparten dispositivos → con otras cuentas sospechosas. En SQL, cada "salto" es un JOIN adicional. Cuatro saltos = cuatro JOINs. El planner SQL calcula el producto cartesiano en cada paso. El resultado: consultas que tardan minutos en tablas de millones de filas.

sql — El infierno de los JOINs a 4 saltos
-- Encontrar cuentas a 4 saltos de una cuenta sospechosa
SELECT DISTINCT c4.id
FROM cuentas c1
JOIN transacciones t1 ON c1.id = t1.origen_id
JOIN cuentas c2 ON t1.destino_id = c2.id
JOIN transacciones t2 ON c2.id = t2.origen_id
JOIN cuentas c3 ON t2.destino_id = c3.id
JOIN transacciones t3 ON c3.id = t3.origen_id
JOIN cuentas c4 ON t3.destino_id = c4.id
WHERE c1.id = 12345;
-- Con 1M de cuentas y 10M de transacciones: minutos de ejecución
cypher — Lo mismo en Neo4j
-- Exactamente la misma consulta. Sub-segundo.
MATCH (c:Cuenta {id: 12345})-[*1..4]->(destino:Cuenta)
RETURN DISTINCT destino.id
💡
Dato de producción: Neo4j documentó con eBay un caso donde una consulta de recomendación que tardaba 2000ms en SQL pasó a 25ms en Neo4j. La razón no es que Neo4j sea "más rápido" en general — es que el modelo de grafos elimina el JOIN overhead para consultas de conectividad.

¿Por qué surgieron los grafos?

El problema apareció con las redes sociales. En 2007, Facebook tenía 50M de usuarios. Encontrar amigos en común con SQL requería auto-JOINs sobre tablas de 500M de filas. Al mismo tiempo, Google ya llevaba años usando PageRank — un algoritmo de grafos — para clasificar páginas web. La teoría de grafos llevaba 300 años existiendo (Euler, 1736), pero fue la explosión de datos relacionados la que la convirtió en necesidad de ingeniería.

🕰️ Hitos históricos
  • 1736: Euler resuelve los puentes de Königsberg (primer problema de grafos)
  • 1998: Google PageRank (grafos en producción a escala)
  • 2007: Neo4j fundado por Emil Eifrem
  • 2010: Neo4j 1.0 en producción
  • 2012: Neo4j adopta Cypher como lenguaje estándar
  • 2022-2026: Neo4j 5.x, GraphRAG, vector indexes
🎯 Casos clave que impulsaron los grafos
  • Redes sociales (Facebook, LinkedIn)
  • Detección de fraude bancario
  • Motores de recomendación (Netflix, Amazon)
  • Knowledge Graphs (Google, Wikidata)
  • Gestión de dependencias IT (CMDB)
  • Drug discovery en farmacéutica
  • GraphRAG para LLMs (2024-2026)

3. LPG vs RDF — Cuándo usar cada uno

3.1 Qué es LPG y cómo almacena datos

LPG son las siglas de Labeled Property Graph — grafo etiquetado con propiedades. Tiene cuatro componentes concretos:

  • Nodos con una o varias etiquetas (labels) que indican su tipo: :Persona, :Producto, :Cuenta.
  • Propiedades en nodos: pares clave-valor directamente dentro del nodo. {nombre: "Ana", edad: 34}.
  • Relaciones con un tipo único y dirección. [:COMPRA], [:SIGUE], [:TRABAJA_EN].
  • Propiedades en relaciones: también pares clave-valor. [:COMPRA {precio: 99, fecha: "2024-03-01"}].

Este último punto es el que separa LPG de casi todo lo demás. Una relación en LPG no es solo un enlace binario — es un objeto de primera clase que puede transportar información. No necesitas un nodo extra para decir cuándo ocurrió esa compra o a qué precio. Eso está en la propia arista.

cypher — Modelo LPG: nodo + relación con propiedades
-- Un cliente compra un producto. La relación lleva el contexto de la compra.
CREATE (ana:Persona {nombre: "Ana", ciudad: "Madrid"})
CREATE (prod:Producto {nombre: "Teclado mecánico", sku: "KB-001"})
CREATE (ana)-[:COMPRA {precio: 89.99, fecha: "2024-03-15", canal: "web"}]->(prod)

-- Consultar qué compró Ana, con contexto de la compra
MATCH (p:Persona {nombre: "Ana"})-[c:COMPRA]->(prod:Producto)
RETURN prod.nombre, c.precio, c.fecha, c.canal

3.2 Qué es RDF y cómo almacena datos

RDF son las siglas de Resource Description Framework. Es un estándar del W3C diseñado para la Web Semántica — la idea de que cualquier entidad del mundo puede describirse con URIs universales y enlazarse con otras. Su unidad básica es la tripleta:

sujeto → predicado → objeto

Cada elemento es una URI o un literal. Por ejemplo:

turtle — RDF: la misma compra expresada como tripletas
# Turtle es la serialización más legible de RDF
@prefix ex: <http://ejemplo.org/> .
@prefix xsd: <http://www.w3.org/2001/XMLSchema#> .

ex:Ana ex:nombre "Ana"^^xsd:string .
ex:Ana ex:ciudad "Madrid"^^xsd:string .
ex:TecladoMecanico ex:nombre "Teclado mecánico"^^xsd:string .

# El PROBLEMA: no puedo poner precio ni fecha directamente en la relación
# porque las tripletas son siempre sujeto→predicado→objeto (tres elementos, nada más)
# Para añadir contexto a la compra necesito REIFICACIÓN: crear un nodo intermedio

ex:Compra001 a ex:EventoCompra .
ex:Compra001 ex:comprador ex:Ana .
ex:Compra001 ex:producto ex:TecladoMecanico .
ex:Compra001 ex:precio "89.99"^^xsd:decimal .
ex:Compra001 ex:fecha "2024-03-15"^^xsd:date .
WARN — Reificación: En RDF, "añadir propiedades a una relación" no existe. Tienes que crear un nodo intermedio (ex:Compra001) que actúa de puente. Esto multiplica el número de tripletas, complica las consultas SPARQL y degrada el rendimiento en traversals profundos. En LPG, eso es simplemente [:COMPRA {precio: 89.99}].

3.3 La diferencia conceptual real: instancia de datos vs semántica formal

La distinción más importante no es técnica — es de propósito:

  • LPG está diseñado para almacenar y recorrer datos operacionales de forma eficiente. El motor sabe cómo navegar aristas porque cada nodo guarda punteros físicos a sus vecinos (index-free adjacency). Prioridad: rendimiento en traversal.
  • RDF está diseñado para interoperabilidad global. Cualquier base de datos en el mundo que use la URI http://dbpedia.org/resource/Madrid está hablando del mismo Madrid. Se puede hacer razonamiento lógico sobre los datos gracias a OWL (Web Ontology Language). Prioridad: interoperabilidad y inferencia.

Dicho de otro modo: si tu problema es recorrer y operar sobre un grafo de datos tuyo, LPG. Si tu problema es conectar tu conocimiento con el de otros sistemas del mundo o derivar nuevo conocimiento por razonamiento lógico, RDF.

AspectoLPG (Neo4j)RDF (Blazegraph, Stardog, Apache Jena)
Unidad básicaNodo con labels + relación con tipoTripleta: sujeto → predicado → objeto
Props en relacionesNativo. [:COMPRA {precio: 99}]No existe. Requiere reificación (nodo intermedio)
Identidad de entidadesID interno del grafo (local)URI global. http://dbpedia.org/resource/Madrid
Lenguaje de consultaCypherSPARQL

3.4 Cypher vs SPARQL — el mismo dato, dos mundos

Para que la diferencia sea tangible. Pregunta: ¿qué películas ha visto una persona y cuál es el director de cada una?

cypher — LPG: legible, directo, traversal nativo
MATCH (ana:Persona {nombre: "Ana"})-[:VIO]->(peli:Pelicula)<-[:DIRIGIO]-(dir:Persona)
RETURN peli.titulo, dir.nombre
sparql — RDF: más verboso, URIs en lugar de labels
PREFIX ex: <http://ejemplo.org/>
PREFIX schema: <https://schema.org/>

SELECT ?tituloFilm ?nombreDirector
WHERE {
  ex:Ana schema:watched ?film .
  ?film schema:name ?tituloFilm .
  ?film schema:director ?director .
  ?director schema:name ?nombreDirector .
}

SPARQL es potente cuando necesitas federar consultas contra múltiples endpoints RDF remotos (Wikipedia, DBpedia, Wikidata). Pero para un grafo de datos interno — un sistema de recomendaciones, un grafo de fraude, un Knowledge Graph de empresa — Cypher es mucho más ergonómico y el motor es órdenes de magnitud más rápido en traversals.

3.5 Cuándo usar cada uno — criterio de decisión

  • Usa LPG (Neo4j) si: tus datos son propios, necesitas traversals rápidos, las relaciones llevan contexto (precio, fecha, peso), o construyes una app operacional o un GraphRAG sobre datos internos de empresa.
  • Usa RDF si: necesitas publicar datos como Linked Open Data, tu dominio ya tiene una ontología estándar (biomédica, geoespacial, jurídica), o necesitas que un motor infiera automáticamente conocimiento nuevo a partir de reglas formales (razonamiento OWL).
  • Usa ambos si: tienes un Knowledge Graph interno (LPG) que quieres conectar a recursos abiertos externos (DBpedia, Wikidata). Neo4j tiene un plugin RDF (neosemantics) que permite importar/exportar tripletas RDF desde un grafo LPG.
INFO — GraphRAG (2025/2026): Microsoft y LangChain usan Neo4j como base de conocimiento para LLMs. El LPG es más eficiente para traversals operacionales que RDF: las propiedades en nodos y relaciones mapean directamente a los facts que los LLMs necesitan sin transformación. Si construyes un sistema RAG con grafos de conocimiento sobre datos de empresa, Neo4j es la opción pragmática. RDF sería la elección si quisieras federar ese knowledge graph con fuentes externas tipo Wikidata o Gene Ontology.

3b. Ontología — qué es y por qué aparece en el ecosistema de grafos

Definición

En informática, una ontología es una especificación formal y explícita de los conceptos de un dominio, sus propiedades y las relaciones entre ellos. El término viene de la filosofía (el estudio del ser y la existencia), pero en el contexto de grafos y knowledge graphs significa algo muy concreto: un vocabulario compartido con semántica precisa que permite que máquinas —y personas de distintos sistemas— entiendan los datos de la misma manera.

Una ontología responde a tres preguntas sobre un dominio:

  • ¿Qué clases de entidades existen? — Persona, Publicación, Organización, Lugar
  • ¿Qué propiedades tienen? — nombre, fecha_nacimiento, afiliación
  • ¿Qué relaciones son válidas entre ellas? — Persona → trabaja_en → Organización, Publicación → cita → Publicación

Más allá de definir clases y relaciones, una ontología puede incluir axiomas — reglas lógicas que permiten derivar nuevo conocimiento a partir de los datos. Esto es lo que la diferencia de un simple esquema de base de datos.

Ontología vs Esquema de base de datos — la diferencia real

Un schema SQL o un conjunto de constraints en Neo4j define estructura: qué campos existen, qué tipos tienen, cuáles son obligatorios. Una ontología define semántica: qué significa cada clase, qué implica pertenecer a ella, qué se puede inferir de una relación.

AspectoSchema de BDOntología
PropósitoValidar estructura de datosDefinir significado y permitir razonamiento
HerenciaNo (salvo en OOP)Sí: Investigador es subclase de Persona
InferenciaNo — el motor no deduce nadaSí — si A cita B y B cita C, un razonador puede inferir A influencia C
InteroperabilidadLocal al sistemaGlobal — URIs universales permiten enlazar con otras ontologías

Cómo se construye una ontología: clases, propiedades y axiomas

El lenguaje estándar para ontologías en la Web Semántica es OWL (Web Ontology Language), que se serializa habitualmente en Turtle o RDF/XML. Los tres bloques fundamentales son:

turtle — Ontología mínima de investigación académica
@prefix owl:  <http://www.w3.org/2002/07/owl#> .
@prefix rdfs: <http://www.w3.org/2000/01/rdf-schema#> .
@prefix ex:   <http://milab.org/onto/investigacion#> .
@prefix xsd:  <http://www.w3.org/2001/XMLSchema#> .

# 1. CLASES — qué tipos de entidad existen
ex:Persona        a owl:Class .
ex:Investigador   a owl:Class ;
    rdfs:subClassOf ex:Persona .              # herencia: todo Investigador ES una Persona
ex:Publicacion    a owl:Class .
ex:Organizacion   a owl:Class .

# 2. PROPIEDADES DE DATOS — atributos primitivos
ex:nombre         a owl:DatatypeProperty ;
    rdfs:domain ex:Persona ;                  # quién la tiene
    rdfs:range  xsd:string .                  # qué tipo de valor
ex:anioPublicacion a owl:DatatypeProperty ;
    rdfs:domain ex:Publicacion ;
    rdfs:range  xsd:integer .

# 3. PROPIEDADES DE OBJETO — relaciones entre clases
ex:trabajaEn      a owl:ObjectProperty ;
    rdfs:domain ex:Investigador ;
    rdfs:range  ex:Organizacion .
ex:escribe        a owl:ObjectProperty ;
    rdfs:domain ex:Investigador ;
    rdfs:range  ex:Publicacion .
ex:cita           a owl:ObjectProperty ;
    rdfs:domain ex:Publicacion ;
    rdfs:range  ex:Publicacion .

# 4. AXIOMAS — reglas que permiten inferencia
ex:cita           a owl:TransitiveProperty .  # si A cita B y B cita C → A cita C (transitiva)
ex:citeInverso    owl:inverseOf ex:cita .     # si A cita B → B es_citado_por A

Lo que hace potente esta ontología es el axioma de transitividad: un razonador OWL puede recorrer toda la cadena de citas y derivar relaciones de influencia que no están explícitamente almacenadas. Con un LPG puro tendrías que escribir una query Cypher con * para hacer lo mismo, y no sería automático — sería una consulta que tú ejecutas.

Ontologías de dominio conocidas — ejemplos reales

Las ontologías más usadas en la práctica no se crean desde cero: existen estándares de dominio que miles de sistemas comparten. Usar una ontología establecida permite que tu grafo sea interoperable con cualquier sistema que la entienda.

Ontología en Neo4j (LPG) — neosemantics (n10s)

Neo4j no tiene razonamiento OWL nativo, pero puede importar, almacenar y consultar ontologías RDF gracias al plugin neosemantics (abreviado n10s). Esto permite usar Neo4j como backend de un knowledge graph que es formalmente compatible con el ecosistema RDF/OWL, pero consultable con Cypher.

cypher — Importar una ontología OWL en Neo4j con neosemantics
-- 1. Inicializar el grafo para modo RDF (solo una vez por base de datos)
CALL n10s.graphconfig.init({
  handleVocabUris: 'SHORTEN',   -- acorta URIs largas a prefijo:local
  handleMultival: 'ARRAY'        -- propiedades multi-valor como arrays
})

-- 2. Importar una ontología desde URL (ej: schema.org en formato Turtle)
CALL n10s.onto.import.fetch(
  'https://schema.org/version/latest/schemaorg-current-https.ttl',
  'Turtle'
)
YIELD terminationStatus, triplesLoaded, triplesParsed
RETURN terminationStatus, triplesLoaded

-- 3. Consultar la jerarquía de clases importada
MATCH (clase:Class)-[:SCO]->(superclase:Class)   -- SCO = subClassOf
RETURN clase.name, superclase.name
ORDER BY superclase.name, clase.name
LIMIT 20

-- 4. Ver propiedades válidas para una clase
MATCH (prop:Property)-[:DOMAIN]->(clase:Class {name: 'Person'})
RETURN prop.name, prop.comment ORDER BY prop.name

Cuándo necesitas una ontología y cuándo no

La ontología añade potencia pero también complejidad. La pregunta es si esa complejidad está justificada por el problema que resuelves.

  • Necesitas ontología si: tu knowledge graph tiene que interoperar con otros sistemas externos (bases de datos de terceros, publicaciones científicas, datos gubernamentales en Linked Open Data), o si necesitas razonamiento automático — que el sistema infiera hechos nuevos sin que los hayas escrito explícitamente.
  • No necesitas ontología si: tu grafo es interno, los datos son tuyos, los queries los controlas tú y lo que necesitas es rendimiento operacional en traversals. Un LPG puro con buenos constraints en Neo4j es suficiente y significativamente más rápido.
  • Caso intermedio: usar neosemantics para importar la ontología de un dominio (ej: schema.org) como vocabulario de referencia para tus labels y tipos de relación, sin activar el razonador OWL. Ganas interoperabilidad semántica sin el coste del razonamiento.
INFO — Ontología y GraphRAG: En sistemas GraphRAG (Neo4j + LLM), la ontología actúa como el vocabulario que controla cómo el LLM interpreta el grafo. Si tus nodos usan labels de schema.org (:Person, :Organization, :CreativeWork), el modelo de lenguaje los reconoce sin fine-tuning porque ese vocabulario está en su corpus de entrenamiento. Es una razón práctica para adoptar ontologías establecidas aunque no necesites razonamiento OWL formal.

4. SQL vs NoSQL Documental vs Grafos — comparativa real

caso de uso — Red social: amigos en común
-- SQL: O(n²) para buscar amigos de amigos
SELECT a2.user_id
FROM friendships f1
JOIN friendships f2 ON f1.friend_id = f2.user_id
WHERE f1.user_id = 42
  AND f2.friend_id != 42
  AND f2.friend_id NOT IN (
      SELECT friend_id FROM friendships WHERE user_id = 42
  );

-- Neo4j Cypher: limpio, eficiente, legible
MATCH (yo:User {id: 42})-[:AMIGO]->(amigo)-[:AMIGO]->(sugerido)
WHERE NOT (yo)-[:AMIGO]->(sugerido) AND sugerido <> yo
RETURN sugerido.nombre, count(amigo) AS amigos_en_comun
ORDER BY amigos_en_comun DESC LIMIT 10

5. Arquitectura Interna — Index-Free Adjacency

Este es el secreto del rendimiento de Neo4j. En bases de datos relacionales, para encontrar los vecinos de un nodo el motor hace un index lookup en la tabla de JOINs. En Neo4j, cada nodo almacena directamente punteros físicos a sus relaciones adyacentes en disco. Navegar al nodo vecino es seguir un puntero — O(1) por salto, independientemente del tamaño total del grafo.

Index-Free Adjacency vs Relational Index Lookup

Record stores — almacenamiento físico

Neo4j 5.x mantiene archivos de registros separados para nodos, relaciones y propiedades:

  • nodestore — 15 bytes por nodo. Contiene: in-use flag, ID primera relación, ID primera propiedad, label store ID.
  • relationshipstore — 57 bytes por relación. Contiene: nodos origen/destino, tipo, punteros a relaciones anterior/siguiente para cada nodo (doubly-linked list per node).
  • propertystore — cadena de bloques de propiedades de tamaño variable.
🏗️
¿Por qué tamaño fijo? Los registros de tamaño fijo permiten acceso directo por ID sin B-Tree. Si el nodo tiene ID 12345, su registro está en el offset 12345 × 15 bytes. El SO puede pre-fetch eficientemente con el page-cache. Esto es lo que hace que seguir un puntero sea O(1).

🧪 Ejercicios — Módulo 1

📝
E1 (Básico): Levanta Neo4j con Docker Compose. Accede al Browser. Ejecuta :play movies. ¿Cuántos nodos de tipo Movie existen? Solución: MATCH (m:Movie) RETURN count(m)
📝
E2 (Básico): Con el dataset movies, encuentra el actor con más películas. Pista: MATCH + count + ORDER BY DESC + LIMIT 1
📝
E3 (Medio): Diseña en papel un grafo de conocimiento para tu laboratorio de IA: investigadores, proyectos, publicaciones, datasets, tecnologías. Define tipos de nodos, tipos de relaciones y propiedades clave. No escribas Cypher aún — solo el modelo conceptual.
📝
E4 (Medio): Argumenta en qué caso no usarías Neo4j para un sistema de vehículos autónomos. ¿Qué tipo de datos del coche irían mejor en PostgreSQL, InfluxDB o Kafka?
📝
E5 (Avanzado): Busca la documentación oficial sobre el "block format" de Neo4j 5.x. ¿En qué difiere del record store clásico? ¿Qué ventaja tiene para bases de datos multi-terabyte?
