Rendimiento extremo, herramientas de ecosistema y algoritmos de grafos para arquitectos serios.

1. Índices en Neo4j 5.x

Neo4j 5.x tiene múltiples tipos de índice. Elegir el correcto es la diferencia entre milisegundos y segundos.

TipoUsoCypher
RangeBúsquedas por valor exacto, rangos, ordenCREATE INDEX FOR (n:Label) ON (n.prop)
Full-TextBúsqueda textual (CONTAINS, ~)CREATE FULLTEXT INDEX ix FOR (n:Paper) ON EACH [n.titulo, n.abstract]
PointDatos geoespaciales (distancia, área)CREATE POINT INDEX FOR (n:Ciudad) ON (n.coords)
VectorBúsqueda por similitud semántica (embeddings)CREATE VECTOR INDEX ix FOR (n:Doc) ON (n.embedding)
Token LookupBuscar por label o rel-type (creado auto)Automático, no se crea manualmente
cypher — Constraints y uso del Full-Text
-- CONSTRAINTS: garantizan integridad + crean índice automáticamente
CREATE CONSTRAINT paper_doi_unique FOR (p:Paper) REQUIRE p.doi IS UNIQUE
CREATE CONSTRAINT user_email_unique FOR (u:User) REQUIRE u.email IS UNIQUE
CREATE CONSTRAINT sensor_not_null  FOR (s:Sensor) REQUIRE s.tipo IS NOT NULL

-- FULL-TEXT INDEX para búsqueda semántica en papers
CREATE FULLTEXT INDEX papers_ft
FOR (p:Paper) ON EACH [p.titulo, p.abstract]

-- Buscar papers que mencionan "LIDAR calibration"
CALL db.index.fulltext.queryNodes("papers_ft", "LIDAR AND calibration")
YIELD node, score
RETURN node.titulo, score ORDER BY score DESC LIMIT 10

-- VECTOR INDEX para búsqueda por embedding (GraphRAG)
CREATE VECTOR INDEX paper_embedding
FOR (p:Paper) ON (p.embedding) OPTIONS {indexConfig: {
  `vector.dimensions`: 1536, `vector.similarity_function`: "cosine"
}}

CALL db.index.vector.queryNodes("paper_embedding", 5, $query_embedding)
YIELD node, score
RETURN node.titulo, score

2. EXPLAIN y PROFILE — Leer el Plan de Ejecución

cypher — EXPLAIN vs PROFILE
-- EXPLAIN: solo muestra el plan, no ejecuta (gratis)
EXPLAIN
MATCH (p:Paper)-[:REFERENCIA]->(r:Paper)
WHERE p.citaciones > 100
RETURN p.titulo, count(r)

-- PROFILE: ejecuta y reporta coste real (db hits, rows)
PROFILE
MATCH (inv:Investigador {nombre: "Deivit"})-[:PUBLICÓ]->(p:Paper)
RETURN p.titulo

✅ Señales de plan eficiente

  • NodeIndexSeek — usa índice, O(log n)
  • Rows ≈ resultado final — sin expansión masiva
  • DB Hits bajos — pocas lecturas de storage
  • Expand(Into) en joins de grafos densos

🚫 Señales de plan lento

  • AllNodesScan — sin label filter, escanea todo
  • Rows astronómicas en steps intermedios
  • CartesianProduct — producto cartesiano no deseado
  • Filter alto antes de un Seek — reordena tus WHERE

3. Procesamiento por Lotes — CALL IN TRANSACTIONS

cypher — Batch update sin agotar heap
-- Migración masiva: añadir embeddings a 10M papers
MATCH (p:Paper) WHERE p.embedding IS NULL
CALL {
  WITH p
  CALL apoc.ml.openai.embedding([p.abstract], $api_key) YIELD embedding
  SET p.embedding = embedding[0]
} IN TRANSACTIONS OF 500 ROWS
RETURN count(*) AS procesados

4. APOC — Awesome Procedures on Cypher

4.1 Qué es APOC y por qué existe

Cypher es un lenguaje declarativo diseñado para traversals y manipulación de grafos. Lo que no hace bien de forma nativa: importar datos de fuentes externas, ejecutar lógica condicional compleja dentro de una query, convertir formatos, procesar millones de registros en batches sin volcar la memoria de la JVM, o explorar subgrafos con criterios dinámicos. Para todo eso existe APOC — Awesome Procedures on Cypher.

APOC es la librería de extensiones oficial de Neo4j. Se instala como plugin (un JAR en el directorio plugins/ de Neo4j) y expone más de 450 procedimientos y funciones invocables desde Cypher con CALL apoc.* o apoc.*() según el tipo. Está dividida en dos artefactos:

  • apoc-core: incluido por defecto en Neo4j Desktop y AuraDB. Contiene las utilidades más usadas.
  • apoc-extended: funcionalidades adicionales (conexiones JDBC, exportación a sistemas externos). Requiere instalación explícita del JAR.
INFO — Verificar APOC instalado: RETURN apoc.version(). Si devuelve la versión, APOC core está disponible. Si da error, el plugin no está cargado. En Docker Compose: el JAR debe estar en el volumen plugins/ y la variable de entorno debe incluir NEO4J_PLUGINS=["apoc"] (Neo4j 5.x) o el fichero apoc.conf con los permisos necesarios (Neo4j 4.x).

4.2 Las cuatro familias de procedimientos

4.3 apoc.load.json — importar datos de APIs externas

Hace una petición HTTP GET contra un endpoint JSON y expone el resultado como mapa Cypher accesible con notación de punto. El JSON completo se recibe en value, y desde ahí puedes navegar su estructura con value.campo.subcampo, UNWIND para arrays, etc.

Prerequisito en Docker: variable de entorno NEO4J_apoc_import_file_enabled=true y NEO4J_apoc_import_file_use__neo4j__config=false. Para URLs externas, añadir el dominio a apoc.import.file.enabled=true en neo4j.conf.

cypher — apoc.load.json: importar papers de Semantic Scholar
CALL apoc.load.json(
  "https://api.semanticscholar.org/graph/v1/paper/search?query=autonomous+lidar&fields=title,year,authors,externalIds&limit=20",
  {}, -- segundo parámetro: headers HTTP {Authorization: "Bearer TOKEN"}
  {failOnError: false}
)
YIELD value
-- value es el JSON completo. value.data es el array de papers.
UNWIND value.data AS paper_data
MERGE (p:Paper {doi: coalesce(paper_data.externalIds.DOI, paper_data.paperId)})
SET p.titulo = paper_data.title, p.year = paper_data.year
WITH p, paper_data
UNWIND paper_data.authors AS autor_data
MERGE (a:Autor {id: autor_data.authorId})
SET a.nombre = autor_data.name
MERGE (a)-[:ES_AUTOR_DE]->(p)

-- Verificar resultado
MATCH (p:Paper)<-[:ES_AUTOR_DE]-(a:Autor)
RETURN p.titulo, collect(a.nombre) AS autores LIMIT 5
cypher — apoc.load.csv: importar fichero local
-- El fichero debe estar en el directorio import/ de Neo4j
-- Docker: montado como volumen en /var/lib/neo4j/import/accidents.csv
CALL apoc.load.csv("file:///accidents.csv", {header: true, sep: ","})
YIELD map -- cada fila del CSV es un mapa {nombre_columna: valor}
MERGE (a:Accidente {id: map.accident_id})
SET
  a.fecha     = date(map.date),
  a.severidad = toInteger(map.severity),
  a.latitud   = toFloat(map.lat)
WITH a, map
MATCH (v:Vehiculo {vin: map.vehicle_vin})
MERGE (v)-[:INVOLUCRADO_EN]->(a)

-- Fallo típico: si map.date viene como "03/15/2024" (MM/DD/YYYY),
-- date() falla. Usar: apoc.date.parse(map.date, 'ms', 'MM/dd/yyyy')

4.4 apoc.periodic.iterate — batches sin matar la JVM

El problema con MATCH (n:Paper) SET n.indexed = true sobre millones de nodos: Neo4j abre una única transacción que carga todo en memoria antes del commit. Con suficientes nodos, superas el heap de la JVM y obtienes Transaction too large o OutOfMemoryError.

apoc.periodic.iterate divide el trabajo en dos queries: la outer query produce los elementos (se ejecuta una vez); la inner query los procesa en lotes de batchSize registros, cada lote en su propia transacción. Si un lote falla, los demás siguen.

cypher — apoc.periodic.iterate: actualización masiva en batches
CALL apoc.periodic.iterate(
  -- Outer query: produce los nodos a procesar. Se ejecuta UNA vez.
  "MATCH (p:Paper) WHERE p.indexed IS NULL RETURN p",
  -- Inner query: qué hacer con cada elemento. Se ejecuta en batches.
  "SET p.indexed = true, p.indexed_at = datetime()",
  {
    batchSize: 1000,  -- nodos por transacción
    parallel:  false, -- true solo si las ops son independientes entre sí (riesgo deadlock)
    retries:   3      -- reintentos por lote en caso de deadlock
  }
)
YIELD batches, total, timeTaken, failedBatches, errorMessages
RETURN batches, total, timeTaken, failedBatches, errorMessages
-- Output esperado si todo va bien: {failedBatches: 0, ...}
-- Si failedBatches > 0 revisar errorMessages para diagnosticar

4.5 apoc.path.subgraphAll — subgrafos con filtros avanzados

Cypher permite *1..n en traversals, pero los filtros son limitados. apoc.path.subgraphAll permite combinar múltiples tipos de relación con dirección, filtrar por labels de nodos intermedios y fijar niveles mín/máx de expansión, todo en la misma llamada.

cypher — apoc.path.subgraphAll: árbol causal de un accidente
MATCH (acc:Accidente {id: "ACC-001"})
CALL apoc.path.subgraphAll(acc, {
  relationshipFilter: "CAUSA_DIRECTA>|SENSOR_FALLO>|CONDICION_AMB>",
  -- '>' = salida del nodo actual; '<' = entrada; sin símbolo = ambas
  labelFilter:        "/Accidente|Sensor|CondicionAmbiental",
  -- '/' = incluir solo nodos con estos labels; '-' = excluir
  maxLevel: 4,
  minLevel: 1
})
YIELD nodes, relationships
RETURN nodes, relationships
-- Útil para visualizar en Browser o exportar a JSON con apoc.convert.toJson()

4.6 apoc.do.when — lógica IF/ELSE dentro de Cypher

Cypher no tiene IF/ELSE como sentencia. CASE funciona en expresiones pero no puede ejecutar cláusulas (SET, CREATE). apoc.do.when evalúa una condición booleana y ejecuta una de dos queries Cypher completas. Las variables del contexto externo se pasan como mapa en el tercer parámetro.

cypher — apoc.do.when: actualización condicional de sensor
MATCH (s:Sensor {id: "S-LIDAR-01"})
CALL apoc.do.when(
  s.estado = "ERROR",  -- condición booleana
  "SET s.alertas = coalesce(s.alertas,0)+1, s.last_error = datetime() RETURN s",
  "SET s.ok_count = coalesce(s.ok_count,0)+1, s.last_ok = datetime() RETURN s",
  {s: s}  -- variables del contexto externo accesibles dentro de las queries
)
YIELD value
RETURN value.s.id, value.s.estado, value.s.alertas, value.s.ok_count

-- Para múltiples ramas: apoc.do.case([condicion1, query1, condicion2, query2, ...], queryDefault, params)

4.7 apoc.meta.schema — inspeccionar el grafo sin documentación

En un LPG sin schema rígido es fácil perder la noción de qué labels y tipos de relación existen. apoc.meta.schema hace un muestreo estadístico del grafo y devuelve su estructura inferida — útil para auditar, documentar o migrar un grafo existente.

cypher — apoc.meta: auditoría de esquema
-- Esquema completo: labels, propiedades, relaciones entre labels
CALL apoc.meta.schema() YIELD value RETURN value

-- Estadísticas rápidas: conteos globales
CALL apoc.meta.stats()
YIELD labels, relTypesCount, nodeCount, relCount
RETURN labels, relTypesCount, nodeCount, relCount

-- Propiedades por label (muestra de 100 nodos por label)
CALL apoc.meta.nodeTypeProperties()
YIELD nodeType, propertyName, propertyTypes, mandatory
RETURN nodeType, propertyName, propertyTypes, mandatory
ORDER BY nodeType, propertyName

5. GDS — Graph Data Science

cypher — GDS: proyección + algoritmos
-- 1. Crear proyección nativa del subgrafo
CALL gds.graph.project(
  "red-investigadores",
  "Investigador",
  {COLABORA_CON: {orientation: "UNDIRECTED"}}
)

-- 2. PageRank: ¿quiénes son los investigadores más influyentes?
CALL gds.pageRank.stream("red-investigadores")
YIELD nodeId, score
RETURN gds.util.asNode(nodeId).nombre AS investigador, score
ORDER BY score DESC LIMIT 10

-- 3. Louvain: detectar comunidades (grupos de investigación)
CALL gds.louvain.stream("red-investigadores")
YIELD nodeId, communityId
RETURN communityId, collect(gds.util.asNode(nodeId).nombre) AS miembros
ORDER BY size(miembros) DESC LIMIT 5

-- 4. Betweenness Centrality: ¿quién es el broker de la red?
CALL gds.betweenness.stream("red-investigadores")
YIELD nodeId, score
RETURN gds.util.asNode(nodeId).nombre, score ORDER BY score DESC

-- 5. Node Similarity: investigadores con colaboraciones similares
CALL gds.nodeSimilarity.stream("red-investigadores", {topK: 5})
YIELD node1, node2, similarity
RETURN gds.util.asNode(node1).nombre AS inv1,
       gds.util.asNode(node2).nombre AS inv2, similarity
ORDER BY similarity DESC

-- 6. Escribir resultados de vuelta al grafo (mutate mode)
CALL gds.pageRank.write("red-investigadores", {writeProperty: "pagerank"})
YIELD nodePropertiesWritten

-- 7. Limpiar proyección cuando ya no se necesita
CALL gds.graph.drop("red-investigadores")
💡
GDS modos de ejecución: stream devuelve resultados fila a fila (sin escribir al grafo). write escribe propiedades de vuelta al grafo para uso posterior. mutate añade propiedades a la proyección in-memory. stats solo devuelve métricas.
