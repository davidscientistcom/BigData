Lab 2 — Red Social (Facebook-style)

Diseña y construye desde cero una red social completa: usuarios, amistades, posts y likes. Implementa las consultas que hacen útiles los grafos — recomendaciones, caminos y centralidad.

1. ¿Por qué un grafo para una red social?

Las redes sociales son el caso de uso canónico de los grafos. Las consultas más valiosas — "¿quién te recomiendo como amigo?", "¿cómo llegas a esta persona?", "¿quién influye más en tu red?" — son atravesales de grafo que en SQL requieren múltiples JOINs recursivos o tablas de cierre transitivo. En Neo4j son una sola línea de Cypher.

ConsultaSQLCypher
Amigos directos1 JOIN a tabla de amistades(u)-[:FRIENDS_WITH]-(f)
Amigos de amigos2 JOINs + DISTINCT + WHERE filtro(u)-[:FRIENDS_WITH*2]-(f)
Camino más cortoCTE recursiva + ROW_NUMBERshortestPath((a)-[*..N]-(b))
Amigos en comúnSubquery con INTERSECTcount(DISTINCT bridge) en patrón 2-hop

2. Diseño del esquema

2.1 Nodos

LabelPropiedades claveDescripción
:UseruserId (único), name, city, joinedAtCuenta de usuario
:PostpostId (único), content, createdAt, viewsPublicación de contenido
:Tagname (único)Etiqueta temática

2.2 Relaciones

TipoOrigen → DestinoPropiedadesSemántica
:FRIENDS_WITHUser → UsersinceAmistad — se consulta sin dirección con -[]-
:POSTEDUser → PostatAutoría del post
:LIKEDUser → PostatLike de un usuario a un post
:COMMENTEDUser → Posttext, atComentario de usuario en un post
:TAGGED_WITHPost → Tag—Etiqueta asociada al post
💡
¿FRIENDS_WITH debe ser bidireccional? Modelamos la amistad como una sola relación dirigida arbitrariamente (A→B). En las consultas usamos -[:FRIENDS_WITH]- sin flecha para explorar ambos sentidos. Esto evita el anti-patrón de duplicar la relación.

3.1 Limpieza e inicialización

3.2 Constraints e índices

cypher — Paso 1: Esquema
CREATE CONSTRAINT user_id_unique  IF NOT EXISTS FOR (u:User) REQUIRE u.userId  IS UNIQUE;
CREATE CONSTRAINT post_id_unique  IF NOT EXISTS FOR (p:Post) REQUIRE p.postId  IS UNIQUE;

3.3 Insertar usuarios

cypher — Paso 2: Usuarios
UNWIND [
  {id:"u1", name:"Alice",  city:"Madrid",    joined:"2019-03-10"},
  {id:"u2", name:"Bob",    city:"Barcelona", joined:"2018-07-22"},
  {id:"u3", name:"Clara",  city:"Madrid",    joined:"2020-01-05"},
  {id:"u4", name:"David",  city:"Valencia",  joined:"2017-11-30"},
  {id:"u5", name:"Eva",    city:"Sevilla",   joined:"2021-06-14"},
  {id:"u6", name:"Frank",  city:"Barcelona", joined:"2019-09-01"},
  {id:"u7", name:"Gloria", city:"Madrid",    joined:"2022-02-28"},
  {id:"u8", name:"Hector", city:"Valencia",  joined:"2016-04-18"}
] AS row
MERGE (u:User {userId: row.id})
  ON CREATE SET u.name = row.name, u.city = row.city,
                u.joinedAt = date(row.joined), u.active = true;

3.4 Insertar amistades

cypher — Paso 3: FRIENDS_WITH
UNWIND [
  {a:"u1", b:"u2", since:"2020-01-10"},
  {a:"u1", b:"u3", since:"2020-05-20"},
  {a:"u2", b:"u3", since:"2021-03-07"},
  {a:"u2", b:"u4", since:"2019-08-15"},
  {a:"u3", b:"u5", since:"2022-02-11"},
  {a:"u4", b:"u6", since:"2020-11-30"},
  {a:"u5", b:"u7", since:"2023-01-05"},
  {a:"u6", b:"u7", since:"2021-07-22"},
  {a:"u6", b:"u8", since:"2018-12-01"},
  {a:"u7", b:"u8", since:"2022-09-09"}
] AS row
MATCH (a:User {userId: row.a}), (b:User {userId: row.b})
MERGE (a)-[r:FRIENDS_WITH]->(b)
  ON CREATE SET r.since = date(row.since);

3.5 Insertar posts

cypher — Paso 4: Posts
UNWIND [
  {pid:"p1", author:"u1", content:"Mi primer post de grafos!", views:320, at:"2023-01-10"},
  {pid:"p2", author:"u2", content:"Neo4j es increible para redes sociales.", views:540, at:"2023-01-15"},
  {pid:"p3", author:"u3", content:"Tutorial: MATCH multi-hop en Cypher.", views:890, at:"2023-02-01"},
  {pid:"p4", author:"u1", content:"Alguien mas usa GDS para PageRank?", views:210, at:"2023-02-14"},
  {pid:"p5", author:"u4", content:"Comparativa: Neo4j vs ArangoDB.", views:1200, at:"2023-03-05"},
  {pid:"p6", author:"u6", content:"Comunidades detectadas con Louvain.", views:670, at:"2023-03-20"}
] AS row
MATCH (u:User {userId: row.author})
MERGE (p:Post {postId: row.pid})
  ON CREATE SET p.content = row.content, p.views = row.views, p.createdAt = date(row.at)
MERGE (u)-[:POSTED {at: date(row.at)}]->(p);

3.6 Insertar likes

cypher — Paso 5: LIKED
UNWIND [
  {user:"u2", post:"p1"}, {user:"u3", post:"p1"}, {user:"u4", post:"p1"},
  {user:"u1", post:"p2"}, {user:"u5", post:"p2"}, {user:"u7", post:"p2"},
  {user:"u1", post:"p3"}, {user:"u2", post:"p3"}, {user:"u6", post:"p3"}, {user:"u8", post:"p3"},
  {user:"u3", post:"p5"}, {user:"u6", post:"p5"}, {user:"u8", post:"p5"},
  {user:"u1", post:"p6"}, {user:"u5", post:"p6"}
] AS row
MATCH (u:User {userId: row.user}), (p:Post {postId: row.post})
MERGE (u)-[:LIKED {at: date()}]->(p);

3.7 Insertar comentarios

cypher — Paso 6: COMMENTED
UNWIND [
  {user:"u2", post:"p1", text:"Gran post Alice!"},
  {user:"u5", post:"p3", text:"Muy claro el tutorial."},
  {user:"u8", post:"p5", text:"Falta hablar de TigerGraph."},
  {user:"u3", post:"p2", text:"Totalmente de acuerdo."}
] AS row
MATCH (u:User {userId: row.user}), (p:Post {postId: row.post})
CREATE (u)-[:COMMENTED {text: row.text, at: date()}]->(p);

3.8 Insertar tags

cypher — Paso 7: Tags
UNWIND [
  {post:"p1", tag:"grafos"}, {post:"p2", tag:"grafos"}, {post:"p3", tag:"cypher"},
  {post:"p3", tag:"tutorial"}, {post:"p4", tag:"gds"}, {post:"p5", tag:"grafos"},
  {post:"p5", tag:"comparativa"}, {post:"p6", tag:"gds"}, {post:"p6", tag:"comunidades"}
] AS row
MATCH (p:Post {postId: row.post})
MERGE (t:Tag {name: row.tag})
MERGE (p)-[:TAGGED_WITH]->(t);

4. Consultas resueltas paso a paso

Q1 — Amigos directos de Alice

cypher
MATCH (me:User {name:"Alice"})-[r:FRIENDS_WITH]-(friend:User)
RETURN friend.name AS amigo, friend.city AS ciudad, r.since AS desde
ORDER BY r.since;

Salida esperada:

amigociudaddesde
BobBarcelona2020-01-10
ClaraMadrid2020-05-20

Q2 — Recomendación de amigos (amigos de amigos)

Sugerir nuevos amigos a Alice mostrando los amigos en común, excluyendo sus amigos ya directos y a sí misma.

cypher
MATCH (me:User {name:"Alice"})-[:FRIENDS_WITH]-(friend)-[:FRIENDS_WITH]-(candidate:User)
WHERE NOT (me)-[:FRIENDS_WITH]-(candidate)
  AND me <> candidate
RETURN candidate.name         AS recomendado,
       candidate.city         AS ciudad,
       count(DISTINCT friend) AS amigos_en_comun
ORDER BY amigos_en_comun DESC, candidate.name;

Salida esperada:

recomendadociudadamigos_en_comun
DavidValencia1
EvaSevilla1
ℹ️
¿Por qué count(DISTINCT friend)? Porque un candidato puede ser amigo de múltiples amigos de Alice. El DISTINCT evita contar el mismo puente más de una vez cuando hay varios caminos.

Q3 — Amigos en común entre dos usuarios

cypher
MATCH (a:User {name:"Bob"})-[:FRIENDS_WITH]-(common)-[:FRIENDS_WITH]-(b:User {name:"Clara"})
RETURN common.name AS en_comun;

Salida esperada: Alice — amiga compartida de Bob y Clara.

Q5 — Usuarios más populares (por número de amigos)

       count(DISTINCT friend) AS num_amigos

Q6 — Posts más populares (por likes)

cypher
MATCH (p:Post)
OPTIONAL MATCH (:User)-[:LIKED]->(p)
WITH p, count(*) AS likes
MATCH (author:User)-[:POSTED]->(p)
RETURN p.postId AS post, author.name AS autor,
       left(p.content, 45) AS preview,
       p.views AS views, likes
ORDER BY likes DESC;

Q7 — Intereses compartidos por tag

Encuentra pares de usuarios que han dado like a posts con el mismo tag — señal de interés compartido para futuras recomendaciones.

cypher
MATCH (a:User)-[:LIKED]->(p1:Post)-[:TAGGED_WITH]->(t:Tag)
      <-[:TAGGED_WITH]-(p2:Post)<-[:LIKED]-(b:User)
WHERE a.userId < b.userId
RETURN a.name AS u1, b.name AS u2,
       collect(DISTINCT t.name) AS tags_comunes,
       count(DISTINCT t)        AS num_tags
ORDER BY num_tags DESC
LIMIT 8;

Q8 — Recomendación de posts (collaborative filtering)

Recomendar a Alice posts que han gustado a sus amigos pero que ella todavía no ha visto (no ha dado like y no son suyos).

cypher
MATCH (me:User {name:"Alice"})-[:FRIENDS_WITH]-(friend)-[:LIKED]->(rec:Post)
WHERE NOT (me)-[:LIKED]->(rec)
  AND NOT (me)-[:POSTED]->(rec)
RETURN rec.postId AS post,
       left(rec.content, 50) AS preview,
       count(DISTINCT friend) AS amigos_que_gustaron
ORDER BY amigos_que_gustaron DESC;

5. Análisis avanzado: Centralidad y Comunidades (GDS)

⚠️
Estas consultas requieren el plugin Graph Data Science (GDS). Para activarlo en Docker añade el parámetro: -e NEO4JPLUGINS='["graph-data-science"]' al comando docker run.

5.1 PageRank — usuarios influyentes

cypher
// 1. Crear proyección en memoria
CALL gds.graph.project(
  'social',
  'User',
  {FRIENDS_WITH: {orientation: 'UNDIRECTED'}}
);

// 2. Calcular PageRank (stream: no persiste resultados)
CALL gds.pageRank.stream('social')
YIELD nodeId, score
RETURN gds.util.asNode(nodeId).name AS usuario, round(score, 4) AS pagerank
ORDER BY pagerank DESC;

// 3. Limpiar la proyección al terminar
CALL gds.graph.drop('social');

5.2 Label Propagation — detección de comunidades

cypher
CALL gds.graph.project('social-lpa', 'User', {FRIENDS_WITH: {orientation:'UNDIRECTED'}});

CALL gds.labelPropagation.stream('social-lpa')
YIELD nodeId, communityId
RETURN communityId,
       collect(gds.util.asNode(nodeId).name) AS miembros,
       count(*) AS tamano
ORDER BY tamano DESC;

CALL gds.graph.drop('social-lpa');

6. Ejercicios propuestos

ℹ️
E1: Encuentra usuarios de Madrid que no han publicado ningún post. Pista: OPTIONAL MATCH (u)-[:POSTED]->(p) + WHERE p IS NULL.
ℹ️
E2: ¿Cuántos likes acumula cada :Tag en total? Suma los likes de todos los posts etiquetados con ese tag. Ordena de mayor a menor.
ℹ️
E3: Calcula el engagement rate de cada post: (likes + comentarios) / views * 100.0. Devuelve los 3 posts con mayor engagement. Pista: usa size([(u)-[:LIKED]->(p) | u]) para contar likes inline.
ℹ️
E4 (avanzado): Detecta triángulos de amistad — grupos de 3 usuarios donde todos son amigos entre sí. ¿Cuántos triángulos distintos hay? Pista: WHERE a.userId < b.userId AND b.userId < c.userId para evitar duplicados.
