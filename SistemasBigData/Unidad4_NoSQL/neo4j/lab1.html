Lab 1 — Crear la BD desde cero

Levanta Neo4j, define el esquema con constraints e índices, y domina CREATE, MERGE y MATCH con ejercicios reales y salidas esperadas.

1. Entorno: levantar Neo4j con Docker

El entorno más reproducible para practicar es un contenedor Docker limpio. Estas instrucciones son agnósticas — funcionan en cualquier máquina con Docker instalado.

1.1 Requisitos previos

  • Docker Engine ≥ 24 o Docker Desktop
  • Puerto 7474 (HTTP Browser) y 7687 (Bolt) libres en tu máquina
  • ~512 MB de RAM disponibles para el contenedor

1.2 Iniciar el contenedor

bash
docker run \
  --name neo4j-lab \
  -p 7474:7474 -p 7687:7687 \
  -e NEO4J_AUTH=neo4j/your_password \
  -e NEO4J_server_memory_heap_max__size=512m \
  --rm \
  neo4j:5.18
💡
Flag --rm: elimina el contenedor al pararlo. Perfecto para Labs donde quieres empezar desde cero cada vez. Si quieres persistencia entre reinicios, añade -v $PWD/neo4j-data:/data.

1.3 Verificar que Neo4j está listo

El arranque tarda ~10-20 segundos. Cuando veas en la consola la línea:

log
INFO  Remote interface available at http://localhost:7474/

abre http://localhost:7474 en el navegador e introduce las credenciales: usuario neo4j, contraseña your_password.

1.4 Acceso alternativo: cypher-shell

bash
# Desde el host (requiere cypher-shell instalado localmente)
cypher-shell -u neo4j -p your_password -a bolt://localhost:7687

# O desde dentro del mismo contenedor
docker exec -it neo4j-lab cypher-shell -u neo4j -p your_password
⚠️
Primer login obligatorio: Neo4j te forzará a cambiar la contraseña en el primer acceso si usas la interfaz web. En cypher-shell puedes pasar la nueva contraseña directamente con ALTER CURRENT USER SET PASSWORD FROM 'old' TO 'new';

1.5 Seleccionar la base de datos correcta

Neo4j Enterprise permite múltiples bases de datos. En Community Edition (la imagen gratuita) solo existe neo4j por defecto. Antes de cualquier operación, asegúrate de estar en ella:

cypher
:use neo4j    -- en el Browser

-- o en cypher-shell:
SHOW DATABASES;
-- verifica que neo4j está en estado "online"

2. Esquema: Constraints e Índices

En grafos, el "esquema" no es un DDL rígido como en SQL — es opcional pero altamente recomendado. Los constraints garantizan integridad; los índices garantizan rendimiento.

2.1 ¿Por qué definirlos antes de insertar datos?

ℹ️
Si creas un constraint después de insertar datos, Neo4j intentará validar retroactivamente todos los nodos existentes. Si hay duplicados, el constraint fallará con un error. Define siempre el esquema antes de cargar datos.

2.2 Constraint de unicidad

Impide que existan dos nodos con el mismo valor en una propiedad. Implícitamente crea también un índice sobre esa propiedad.

cypher
-- Constraint de unicidad: no puede haber dos :Persona con el mismo email
CREATE CONSTRAINT persona_email_unique
  IF NOT EXISTS
  FOR (p:Persona)
  REQUIRE p.email IS UNIQUE;

-- Constraint de unicidad sobre un :Producto por su SKU
CREATE CONSTRAINT producto_sku_unique
  IF NOT EXISTS
  FOR (pr:Producto)
  REQUIRE pr.sku IS UNIQUE;

2.3 Constraint de existencia (propiedad obligatoria)

cypher
-- Todo nodo :Persona debe tener la propiedad nombre
CREATE CONSTRAINT persona_nombre_exists
  IF NOT EXISTS
  FOR (p:Persona)
  REQUIRE p.nombre IS NOT NULL;
⚠️
Constraints de existencia solo están disponibles en Neo4j Enterprise Edition. En Community Edition (imagen gratuita) solo puedes usar constraints de unicidad.

2.4 Índices adicionales (sin unicidad)

cypher
-- Índice para acelerar búsquedas por nombre (sin unicidad)
CREATE INDEX persona_nombre_idx
  IF NOT EXISTS
  FOR (p:Persona) ON (p.nombre);

-- Índice compuesto: ciudad + fechaNacimiento
CREATE INDEX persona_ciudad_fecha_idx
  IF NOT EXISTS
  FOR (p:Persona) ON (p.ciudad, p.fechaNacimiento);

2.5 Verificar los constraints e índices creados

cypher
SHOW CONSTRAINTS;
SHOW INDEXES;

Deberías ver algo como:

3. CREATE — Crear nodos y relaciones

CREATE siempre inserta nuevas entidades, sin comprobar si ya existen. Es el comando más directo pero el más peligroso si se invoca varias veces sobre los mismos datos.

3.1 Mecánica interna

ℹ️
Cada CREATE abre una transacción, asigna un Node ID interno (autoincremental, no reutilizable), escribe en el store de nodos y en el store de propiedades. Si hay un constraint de unicidad y el valor ya existe, la transacción hace rollback completo y lanza ConstraintValidationFailed.

3.2 Crear un nodo simple

cypher
-- Nodo sin propiedades (válido pero poco útil)
CREATE (:Persona);

-- Nodo con label y propiedades
CREATE (:Persona {
  nombre: "Ana García",
  email:  "ana@example.com",
  edad:   28,
  ciudad: "Valencia"
});

-- Guardar referencia para usar en la misma query
CREATE (p:Persona {nombre: "Carlos López", email: "carlos@example.com"})
RETURN p.nombre, id(p) AS interno;

3.3 Crear múltiples nodos en una sola sentencia

cypher
CREATE
  (:Persona {nombre: "Bea Ruiz",   email: "bea@example.com",   ciudad: "Madrid"}),
  (:Persona {nombre: "Diego Mora", email: "diego@example.com", ciudad: "Barcelona"}),
  (:Persona {nombre: "Elena Sanz", email: "elena@example.com", ciudad: "Sevilla"});

3.4 Crear nodos y relaciones en el mismo CREATE

cypher
-- Crea ambos nodos Y la relación en una sola transacción
CREATE (emp:Empresa {nombre: "Acme", sede: "Madrid"})
CREATE (emp)-[:TIENE_SEDE_EN]->(:Ciudad {nombre: "Madrid"});

-- O en una sola cláusula CREATE (nodos inlineados)
CREATE (:Investigador {nombre: "Raúl"})-[:TRABAJA_EN]->(:Universidad {nombre: "UPV"});

3.5 Crear relación entre nodos ya existentes

cypher
-- Primero localiza los nodos, luego conecta
MATCH (a:Persona {email: "ana@example.com"})
MATCH (b:Persona {email: "carlos@example.com"})
CREATE (a)-[:CONOCE {desde: date("2023-05-01")}]->(b);
🚨
Anti-patrón frecuente: ejecutar CREATE (:Persona {email: "ana@example.com"}) dos veces. Si hay un constraint de unicidad en email, la segunda ejecución fallará con error. Si no hay constraint, se crearán dos nodos Ana — duplicados silenciosos.

3.6 Cuándo usar CREATE (y cuándo no)

SituaciónUsa CREATEUsa MERGE
Insertar registros únicos garantizados (carga inicial única)✅
Crear eventos o logs (siempre nuevos)✅
Insertar entidades que pueden repetirse✅
Scripts idempotentes (se pueden re-ejecutar)✅

4. MERGE — Insertar o actualizar (upsert)

MERGE es el equivalente al patrón upsert: busca si el patrón especificado ya existe en el grafo; si existe, lo vincula; si no existe, lo crea. Es el comando fundamental para scripts idempotentes.

4.1 Mecánica interna de MERGE

ℹ️
MERGE evalúa el patrón completo. Para un nodo, el patrón incluye label + todas las propiedades especificadas. Si cambias una sola propiedad del patrón (p.ej. añades edad), MERGE no reconocerá el nodo existente y creará uno nuevo. Por eso: incluye en el patrón MERGE solo las propiedades que identifican la entidad (normalmente la clave natural), y usa ON CREATE SET / ON MATCH SET para el resto.

4.2 MERGE básico de nodo

cypher
-- Primera ejecución: crea el nodo
MERGE (p:Persona {email: "ana@example.com"})
RETURN p, "creado o encontrado" AS status;

-- Segunda ejecución: find el nodo existente, no crea duplicado
MERGE (p:Persona {email: "ana@example.com"})
RETURN p, "creado o encontrado" AS status;

4.3 ON CREATE SET y ON MATCH SET

cypher
MERGE (p:Persona {email: "ana@example.com"})
  ON CREATE SET
    p.nombre    = "Ana García",
    p.ciudad    = "Valencia",
    p.creadoEn  = datetime()
  ON MATCH SET
    p.ultimoAcceso = datetime(),
    p.accesos      = coalesce(p.accesos, 0) + 1
RETURN p.nombre, p.accesos, p.creadoEn, p.ultimoAcceso;
CláusulaSe ejecuta cuando...Uso típico
ON CREATE SETEl nodo fue recién creadoTimestamps de creación, valores por defecto
ON MATCH SETEl nodo ya existíaTimestamps de actualización, contadores, versiones

4.4 MERGE de relación

cypher
-- Requiere que los nodos ya existan (o existirán si se hace MERGE de ellos antes)
MATCH (a:Persona {email: "ana@example.com"})
MATCH (b:Persona {email: "carlos@example.com"})
MERGE (a)-[r:CONOCE]->(b)
  ON CREATE SET r.desde = date(), r.peso = 1
  ON MATCH SET  r.peso  = r.peso + 1
RETURN r.desde, r.peso;
⚠️
MERGE de relación sin MATCH previo: si escribes MERGE (a:Persona {email:"..."})-[:CONOCE]->(b:Persona {email:"..."}), Neo4j evaluará el patrón completo de ambos nodos más la relación. Si algún nodo no existe, lo crea. Esto puede generar nodos nuevos inesperados. Usa MATCH para anclar primero los nodos.

4.5 MERGE en carga masiva con UNWIND

cypher
-- Insertar una lista de personas de forma idempotente
UNWIND [
  {email: "ana@example.com",   nombre: "Ana"},
  {email: "carlos@example.com", nombre: "Carlos"},
  {email: "bea@example.com",   nombre: "Bea"}
] AS row
MERGE (p:Persona {email: row.email})
  ON CREATE SET p.nombre = row.nombre, p.creadoEn = datetime()
  ON MATCH SET  p.nombre = row.nombre
RETURN count(p) AS procesados;

5. MATCH — Buscar patrones en el grafo

MATCH es la cláusula principal de lectura en Cypher. Describe un patrón gráfico y Neo4j encuentra todas las instancias de ese patrón en el grafo.

5.1 MATCH de nodo simple

cypher
-- Todos los nodos :Persona
MATCH (p:Persona)
RETURN p.nombre, p.email, p.ciudad
ORDER BY p.nombre;

-- Con filtro WHERE
MATCH (p:Persona)
WHERE p.ciudad = "Madrid"
RETURN p.nombre;

-- Filtro inline (equivalente)
MATCH (p:Persona {ciudad: "Madrid"})
RETURN p.nombre;

5.2 MATCH con relación dirigida

cypher
-- Personas que Ana conoce directamente
MATCH (ana:Persona {nombre: "Ana García"})-[:CONOCE]->(conocido:Persona)
RETURN conocido.nombre;

-- La relación inversa: quién conoce a Ana
MATCH (alguien:Persona)-[:CONOCE]->(ana:Persona {nombre: "Ana García"})
RETURN alguien.nombre;

-- Sin importar dirección (ambos sentidos a la vez)
MATCH (ana:Persona {nombre: "Ana García"})-[:CONOCE]-(cualquiera:Persona)
RETURN cualquiera.nombre;

5.3 MATCH multi-hop (ruta variable)

cypher
-- Alcanzables en exactamente 2 saltos (amigos de amigos)
MATCH (ana:Persona {nombre: "Ana García"})-[:CONOCE*2]->(lejano:Persona)
RETURN lejano.nombre;

-- Entre 1 y 3 saltos (alcanzabilidad flexible)
MATCH (ana:Persona {nombre: "Ana García"})-[:CONOCE*1..3]->(lejano:Persona)
RETURN DISTINCT lejano.nombre;

-- Camino más corto entre dos personas
MATCH p = shortestPath(
  (ana:Persona {nombre: "Ana García"})-[:CONOCE*..10]-(elena:Persona {nombre: "Elena Sanz"})
)
RETURN [n IN nodes(p) | n.nombre] AS ruta, length(p) AS saltos;

5.4 MATCH con la relación capturada

cypher
-- Acceder a propiedades de la relación
MATCH (a:Persona)-[r:CONOCE]->(b:Persona)
RETURN a.nombre, b.nombre, r.desde, r.peso
ORDER BY r.peso DESC;

5.5 OPTIONAL MATCH (LEFT JOIN del grafo)

cypher
-- Devuelve todas las personas, tengan o no relación CONOCE saliente
-- (empresa_nombre será null si no la tienen)
MATCH (p:Persona)
OPTIONAL MATCH (p)-[:TRABAJA_EN]->(e:Empresa)
RETURN p.nombre, e.nombre AS empresa;

-- Salida esperada:
-- p.nombre   | empresa
-- Ana García | null          ← sin empresa asignada
-- Raúl ...   | Acme          ← con empresa

cypher
// Q1: ¿A quién conoce Carlos directamente?

E4 — MERGE idempotente: ejecuta dos veces

cypher
// Ejecuta este bloque dos veces y comprueba que accesos se incrementa
MERGE (p:Persona {email:"ana@ex.com"})
  ON MATCH SET p.ultimoLogin = datetime(), p.accesos = coalesce(p.accesos, 0) + 1
RETURN p.nombre, p.accesos, p.ultimoLogin;

Tras la segunda ejecución, p.accesos debe valer 2.

7. Errores comunes y soluciones

ErrorCausaSolución
ConstraintValidationFailed CREATE sobre email duplicado cuando existe unique constraint Usar MERGE en lugar de CREATE
CREATE crea nodos duplicados silenciosamente No existe constraint de unicidad Crear el constraint antes de insertar datos
MERGE crea nodo nuevo cuando ya debería existir El patrón MERGE incluye propiedades no identificadoras (edad, ciudad…) Solo poner la clave natural en MERGE; el resto en ON CREATE/MATCH SET
MATCH no devuelve resultados Mayúsculas/minúsculas incorrectas en nombre de label o propiedad Labels son case-sensitive: :Persona ≠ :persona
Failed to invoke procedure shortestPath No hay ruta entre los nodos (sin filtro de longitud) Añadir *..N y verificar que existen relaciones del tipo correcto
