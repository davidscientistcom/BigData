Sección 02 · Setup

🐳 Kafka con Docker — Listo en 3 minutos

Cluster Kafka completo con KRaft (sin Zookeeper), Kafka UI visual y CLI. Todo en un único docker-compose.yml que puedes ejecutar ahora mismo.

🚀 Opción 1: KRaft — Kafka moderno (sin Zookeeper)
docker-compose.yml — El fichero que lo ejecuta todo

Desde Kafka 3.3+, Kafka no necesita Zookeeper. KRaft simplifica enormemente el setup. Este fichero levanta Kafka + Kafka UI en 3 comandos.

# docker-compose.yml — Kafka KRaft + UI
version: '3.8'

services:
  # ──── KAFKA BROKER ────────────────────────────────────────────
  kafka:
    image: confluentinc/cp-kafka:7.7.0
    container_name: kafka
    ports:
      - "9092:9092"    # para tu app Python
      - "9094:9094"    # para Kafka UI, si lo pones en Docker también
    environment:
      # KRaft: Kafka gestiona su propio consenso (sin Zookeeper)
      KAFKA_NODE_ID: 1
      KAFKA_PROCESS_ROLES: 'broker,controller'
      KAFKA_LISTENERS: 'PLAINTEXT://0.0.0.0:9092,CONTROLLER://0.0.0.0:9093,EXTERNAL://0.0.0.0:9094'
      KAFKA_ADVERTISED_LISTENERS: 'PLAINTEXT://kafka:9092,EXTERNAL://localhost:9094'
      KAFKA_LISTENER_SECURITY_PROTOCOL_MAP: 'PLAINTEXT:PLAINTEXT,CONTROLLER:PLAINTEXT,EXTERNAL:PLAINTEXT'
      KAFKA_CONTROLLER_QUORUM_VOTERS: '1@kafka:9093'
      KAFKA_CONTROLLER_LISTENER_NAMES: 'CONTROLLER'
      KAFKA_INTER_BROKER_LISTENER_NAME: 'PLAINTEXT'
      # Configuración básica
      KAFKA_OFFSETS_TOPIC_REPLICATION_FACTOR: 1
      KAFKA_GROUP_INITIAL_REBALANCE_DELAY_MS: 0
      KAFKA_AUTO_CREATE_TOPICS_ENABLE: 'true'
      KAFKA_LOG_DIRS: '/var/lib/kafka/data'
      # Generar cluster ID único (solo una vez)
      CLUSTER_ID: 'MkU3OEVBNTcwNTJENDM2Qk'
    volumes:
      - kafka_data:/var/lib/kafka/data

  # ──── KAFKA UI ─────────────────────────────────────────────────
  kafka-ui:
    image: provectuslabs/kafka-ui:latest
    container_name: kafka-ui
    depends_on:
      - kafka
    ports:
      - "8080:8080"
    environment:
      KAFKA_CLUSTERS_0_NAME: local
      KAFKA_CLUSTERS_0_BOOTSTRAPSERVERS: kafka:9092

volumes:
  kafka_data: {}
✅ Guarda este fichero como docker-compose.yml en cualquier carpeta y ejecuta los comandos de abajo.
PASO 1 Levantar el cluster
# Levantar en background
docker compose up -d

# Ver que está levantado
docker compose ps

# Ver logs de Kafka
docker compose logs -f kafka

# Esperamos hasta ver:
# INFO Kafka Server started (kafka.server.KafkaServer)
💡 La primera vez tarda 20-30 segundos en arrancar. Espera hasta ver el mensaje de inicio antes de usarlo.
PASO 2 Kafka UI — interfaz visual

Abre http://localhost:8080 en tu navegador. Verás:

  • 📊 Overview del cluster (brokers, topics, consumers)
  • 📋 Topics: crear, ver mensajes, configuración
  • 👥 Consumer Groups: lag, partición asignada
  • 📨 Produce/Consume mensajes manualmente
  • 🔍 Topic Monitoring en tiempo real
🎯 Kafka UI es esencial para desarrollo. Puedes ver los mensajes que produce tu app en tiempo real.
💻 CLI de Kafka — El verdadero panel de control

Aunque Kafka UI es genial para desarrollo visual, Kafka es intrínsecamente "headless" (sin cabeza). En servidores de producción o entornos puramente backend administrarás Kafka exclusivamente mediante su CLI (Command Line Interface). Las herramientas CLI ya vienen instaladas dentro del contenedor del broker.

Paso 0 Entrar a la consola de Kafka

Primero, debemos abrir una terminal dentro del contenedor Docker que está corriendo Kafka para poder usar sus herramientas oficiales.

# Entrar al contenedor de Kafka interactivo (bash)
docker exec -it kafka bash

# (Ahora tu terminal mostrará algo como appuser@kafka:/$)
Paso 1 Crear nuestro primer Topic

Usaremos el comando kafka-topics. Kafka necesita saber dónde está el clúster usando el flag --bootstrap-server. Crearemos un topic llamado "pedidos" definiendo explícitamente cuánto paralelismo queremos (3 particiones).

# Crear un topic con 3 particiones y replication factor 1 (estamos en local)
kafka-topics --bootstrap-server localhost:9092 \
  --create --topic pedidos \
  --partitions 3 \
  --replication-factor 1

# Ver que efectivamente se ha creado listando todos los topics:
kafka-topics --bootstrap-server localhost:9092 --list
Paso 2 Producer: Enviar mensajes manualmente

La herramienta kafka-console-producer nos permite inyectar mensajes por consola escribiendo texto y pulsando Enter. Vamos a enviar datos simulando que vienen con una clave (el ID del usuario) para que mensajes del mismo usuario vayan siempre a la misma partición.

# Escribir con clave (formato clave:valor)
kafka-console-producer --bootstrap-server localhost:9092 \
  --topic pedidos \
  --property "parse.key=true" \
  --property "key.separator=:"

# (Aparecerá un cursor '>' esperando que escribas. Escribe:)
>usuario_A:compro_manzana
>usuario_B:compro_pera
>usuario_A:compro_platano
# (Pulsa Ctrl+C para salir cuando termines)
Paso 3 Consumer: Leer en tiempo real

Para ver qué ha llegado, usaremos kafka-console-consumer. Le diremos explícitamente a Kafka que queremos leer desde el principio de los tiempos (--from-beginning) y que nos muestre las entrañas técnicas del mensaje: clave, partición y offset, que por defecto vienen ocultos.

# Leer TODO el historial mostrando detalles técnicos:
kafka-console-consumer --bootstrap-server localhost:9092 \
  --topic pedidos \
  --from-beginning \
  --property print.key=true \
  --property print.partition=true \
  --property print.offset=true \
  --group analistas-pedidos   # ← Le asignamos un Consumer Group

# (Verás que "usuario_A" y "usuario_B" acaban en particiones distintas)
💡 Truco pro: Si abres dos terminales lado a lado y pones a rodar el Consumer en una y envías mensajes en el Producer en la otra sin parar (quitando el --from-beginning), verás los mensajes llegar al milisegundo. Eso es event-streaming en directo.
Paso 4 Monitorizar el Consumer Group (El Lag)

Dado que al comando anterior le añadimos --group analistas-pedidos, Kafka acaba de dar de alta nuestro grupo internamente y ha empezado a traquear por qué offset íbamos cuando le dimos a Ctrl+C. Esto lo vemos con kafka-consumer-groups.

# Listar todos los grupos registrados en el clúster:
kafka-consumer-groups --bootstrap-server localhost:9092 --list

# Ver el estado (LAG) del grupo "analistas-pedidos":
kafka-consumer-groups --bootstrap-server localhost:9092 \
  --describe --group analistas-pedidos

# Ejemplo de output analizado:
# GROUP              TOPIC    PARTITION  CURRENT-OFFSET  LOG-END-OFFSET  LAG
# analistas-pedidos  pedidos  0          150             155             5   ← 5 mensajes pendientes
# analistas-pedidos  pedidos  1          200             200             0   ← al día
# analistas-pedidos  pedidos  2          98              98              0   ← al día
💡 El LAG es la métrica #1 de salud de tus consumers. Si aumenta de forma constante, significa que la cola de mensajes crece más rápido que la capacidad que tienen tus sistemas para vaciarla.
🔧 Configuración de topics importantes
Parámetros clave al crear un topic
Parámetro Por defecto Descripción Cuándo cambiarlo
--partitions 1 Número de particiones (paralelismo máximo de consumers) Pon tantas como consumers máximos esperas
--replication-factor 1 Cuántas réplicas del dato (1 = sin redundancia) ≥3 en producción para alta disponibilidad
retention.ms 7 días Cuánto tiempo guardar los mensajes Aumentar si necesitas replay histórico
retention.bytes -1 (ilimitado) Tamaño máximo del topic Limitar en topics de alto volumen
cleanup.policy delete delete (borrar por tiempo/tamaño) o compact (solo último valor por clave) compact para tablas de estado, delete para eventos
min.insync.replicas 1 Mínimo de réplicas sincronizadas para aceptar escritura 2 o 3 en producción 
# Crear topic de producción
kafka-topics --bootstrap-server localhost:9092 \
  --create --topic eventos-usuario \
  --partitions 12 \
  --replication-factor 3 \
  --config retention.ms=604800000 \   # 7 días
  --config min.insync.replicas=2

# Topic compactado (para estado/tabla)
kafka-topics --bootstrap-server localhost:9092 \
  --create --topic usuarios-estado \
  --partitions 6 \
  --replication-factor 3 \
  --config cleanup.policy=compact \
  --config min.cleanable.dirty.ratio=0.1
🦖 Opción 2: Legacy Kafka (con Zookeeper)
Para empresas con versiones antiguas (previas a 3.3)

Aunque KRaft es el estándar actual, es posible que en algunas empresas te encuentres con clusters Legacy que aún dependen de Zookeeper. En este caso, la arquitectura requiere dos contenedores obligatorios: uno para Zookeeper y otro para Kafka.

Aquí tienes el docker-compose.yml equivalente para levantar la versión antigua:

# docker-compose-zookeeper.yml
version: '3.8'

services:
  # ──── ZOOKEEPER (Requisito Legacy) ────────────────────────
  zookeeper:
    image: confluentinc/cp-zookeeper:7.3.0
    environment:
      ZOOKEEPER_CLIENT_PORT: 2181
      ZOOKEEPER_TICK_TIME: 2000
    ports:
      - "2181:2181"

  # ──── KAFKA BROKER ────────────────────────────────────────
  kafka:
    image: confluentinc/cp-kafka:7.3.0
    depends_on:
      - zookeeper
    ports:
      - "9092:9092"
    environment:
      # Aquí le decimos a Kafka dónde está Zookeeper
      KAFKA_ZOOKEEPER_CONNECT: 'zookeeper:2181'
      KAFKA_ADVERTISED_LISTENERS: 'PLAINTEXT://localhost:9092'
      KAFKA_OFFSETS_TOPIC_REPLICATION_FACTOR: 1
⚠️ Nota importante: A nivel de código Python (Producer/Consumer), tu aplicación nunca habla con Zookeeper. Tu código siempre se conecta a Kafka por el puerto 9092 independientemente de si el cluster por detrás usa KRaft o Zookeeper.
📦 Opción 3: Confluent Cloud (sin instalar nada)
Para quien quiera Kafka gestionado en la nube

Si no quieres gestionar la infraestructura, Confluent Cloud tiene un tier gratuito con 400.000 mensajes al mes. El código Python es idéntico, solo cambia el bootstrap server y las credenciales.

# Instalar CLI de Confluent
brew install confluentinc/tap/cli   # Mac

# Login y crear cluster
confluent login
confluent kafka cluster create mi-kafka --cloud aws --region eu-west-1 --type basic

# Crear API Key
confluent api-key create --resource <cluster-id>

# Usar en Python — la única diferencia es la configuración:
conf = {
    'bootstrap.servers': 'pkc-xxxxx.eu-west-1.aws.confluent.cloud:9092',
    'security.protocol': 'SASL_SSL',
    'sasl.mechanisms': 'PLAIN',
    'sasl.username': 'TU_API_KEY',
    'sasl.password': 'TU_API_SECRET',
}
💡 Para aprender y desarrollar, usa Docker local. Para producción empresarial, evalúa Confluent Cloud o AWS MSK.
