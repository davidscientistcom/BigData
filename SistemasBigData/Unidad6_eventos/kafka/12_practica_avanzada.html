Laboratorio 03

🛡️ Práctica: Entorno Hostil (E-Commerce)

¿Qué pasa si Kafka se reinicia a medio viaje o si tu Microservicio explota insertando en tu BD? Vamos a asegurar CERO pérdida de pedidos fusionando Avro, Acks=All, Idempotencia y Commits Manuales.

El Producer Blindado (Exactly-Once)
Paso 1: Schema Registry, Idempotencia y Acks=all

En este escenario crítico bancario o de e-commerce, enviar datos con JSON ya no es aceptable (Schema Registry al rescate). Además, tenemos que obligar al clúster entero de Kafka a guardar el recibo en todas las réplicas (acks=all) sin permitir reintentos duplicados en red (enable.idempotence=true).

🐍 Python (Producer con Avro y Garantías)

from confluent_kafka import Producer
from confluent_kafka.schema_registry import SchemaRegistryClient
from confluent_kafka.schema_registry.avro import AvroSerializer
from confluent_kafka.serialization import SerializationContext, MessageField

# 1. Configuración CRÍTICA
conf = {
    'bootstrap.servers': 'localhost:9094',
    'acks': 'all',                  # <- Espera respuesta de todas las particiones hijas
    'enable.idempotence': True,     # <- Kafka detecta TIDs descartando duplicados (Exactly-once)
    'retries': 5                    # <- Se auto-reintenta si la red parpadea 
}
producer = Producer(conf)

# 2. Esquema Validado Central (Pydantic a Nivel Red)
schema_str = '{"type": "record", "name": "Venta", "fields": [{"name": "id", "type": "int"}]}'
sr_client = SchemaRegistryClient({'url': 'http://localhost:8081'})
serializer_avro = AvroSerializer(sr_client, schema_str)

# 3. Envío blindado
venta = {"id": 105}
ctx = SerializationContext('ventas_criticas', MessageField.VALUE)

producer.produce('ventas_criticas', value=serializer_avro(venta, ctx))
producer.flush()

☕ Java (KafkaProducer Blindado)

import org.apache.kafka.clients.producer.*;
import io.confluent.kafka.serializers.KafkaAvroSerializer;
import java.util.Properties;

Properties p = new Properties();
p.put("bootstrap.servers", "localhost:9094");

// GARANTÍAS DE PRODUCCIÓN HOSTIL
p.put("acks", "all");
p.put("enable.idempotence", "true");
p.put("retries", Integer.toString(Integer.MAX_VALUE));

// CONEXIÓN A SCHEMA REGISTRY
p.put("key.serializer", "org.apache.kafka.common.serialization.StringSerializer");
p.put("value.serializer", KafkaAvroSerializer.class.getName());
p.put("schema.registry.url", "http://localhost:8081");

KafkaProducer<String, Venta> producer = new KafkaProducer<>(p);

// Venta auto-generada por Maven .avsc
Venta v = Venta.newBuilder().setId(105).build();

producer.send(new ProducerRecord<>("ventas_criticas", v));
producer.close();
El Consumer Seguro (Manual Commits)
Paso 2: Confirmar Offset 'SÓLO' tras procesar (At-Least Once)
⚠️ El mayor error en Kafka: Usar enable.auto.commit=true en entornos donde interactúas con una Base de Datos. Si Kafka guarda tu offset pero la DB tira una Excepción de Timeout... ¡Pierdes la Orden porque Kafka "ya cree" que la procesaste!

La solución en Producción real es apagar los Auto Commits, e invocar nosotros la confirmación del marcador (commit_sync()) explícitamente y de manera segura bajo nuestro bloque de éxito.

🐍 Python (Commits Manuales)

from confluent_kafka import Consumer

conf = {
    'bootstrap.servers': 'localhost:9094',
    'group.id': 'micro_bd',
    'enable.auto.commit': False  # <- APAGADO. Toma de control manual.
}
consumer = Consumer(conf)
consumer.subscribe(['ventas_criticas'])

while True:
    msg = consumer.poll(1.0)
    if not msg or msg.error():
        continue
    
    try:
        # 1. Tu lógica peligrosa (Ej. Inserción SQL, Llamada REST Externa)
        venta = descodificador_avro(msg.value())
        insertar_en_postgres(venta)
        
        # 2. SÓLO si todo salió perfecto, le avisas a Kafka que mueva la marca
        consumer.commit(msg) # Confirmación síncrona/manual
        
    except Exception as e:
        # Si la BD explotó, NO ejecutamos "commit". 
        # A los pocos segundos/reinicio Kafka re-enviará este offset 
        # impidiendo que la venta se evapore.
        print("La BD está caída, reintentaremos...")
        continue

☕ Java (Manual CommitSync)

import org.apache.kafka.clients.consumer.*;
import java.time.Duration;
import java.util.*;

Properties p = new Properties();
p.put("bootstrap.servers", "localhost:9094");
p.put("group.id", "micro_bd");
p.put("enable.auto.commit", "false"); // <- APAGADO.

KafkaConsumer<String, Venta> consumer = new KafkaConsumer<>(p);
consumer.subscribe(Arrays.asList("ventas_criticas"));

while (true) {
    ConsumerRecords<String, Venta> batch = consumer.poll(Duration.ofMillis(500));
    
    for (ConsumerRecord<String, Venta> r : batch) {
        try {
            // Lógica Pelirosa: Guardado SQL
            insertDataIntoDatabase(r.value());
            
            // Confirmar Individualmente a Kafka que este evento es "HECHO"
            consumer.commitSync();

        } catch (Exception e) {
            System.err.println("Falla crítica. Abortando commit.");
            // Al no hacer commitSync, el consumer volverá a jalar este registro
        }
    }
}
💡 Nota del Experto: Esta es la base arquitectónica ("At-least-once") que usan gigantes como Netflix. Si el proceso crashea, Kafka te reenvía lo repetido. Tan solo debes asegurar que tu Base de Datos sea Idempotente usando 'ON CONFLICT DO NOTHING' para no registrar un pago de $50 dos veces.
