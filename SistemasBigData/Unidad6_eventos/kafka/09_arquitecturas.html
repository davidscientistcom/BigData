📖 Event Sourcing — el log como fuente de verdad
Guardar eventos, no estado — el patrón más poderoso con Kafka

En lugar de guardar el estado actual de una entidad (la cuenta bancaria tiene 500€), guardas todos los eventos que llevaron a ese estado (depósito 300€, retirada 100€, depósito 300€). Kafka es el log de eventos perfecto.

graph LR subgraph CRUD U[TABLE: {id: 1, saldo: 500}] end subgraph EVENT SOURCING Kafka E0[0: CREADA] --> E1[1: DEPOSITO 300] --> E2[2: RETIRADA 100] --> E3[3: DEPOSITO 300] E3 --> R[Estado: 500] end 
import faust
from datetime import datetime

app = faust.App('event-sourcing', broker='kafka://localhost:9094')

# Evento base
class EventoCuenta(faust.Record):
    tipo: str             # DEPOSITO, RETIRADA, TRANSFERENCIA
    cuenta_id: str
    monto: float
    timestamp: str

topic_eventos = app.topic('cuenta-eventos', value_type=EventoCuenta)

# Tabla de proyección: saldo actual por cuenta
saldos = app.Table('saldos', default=float)

@app.agent(topic_eventos)
async def proyectar_saldo(eventos):
    async for evento in eventos:
        if evento.tipo == "DEPOSITO":
            saldos[evento.cuenta_id] += evento.monto
        elif evento.tipo == "RETIRADA":
            saldos[evento.cuenta_id] -= evento.monto
        print(f"Cuenta {evento.cuenta_id}: €{saldos[evento.cuenta_id]:.2f}")
💡 La tabla saldos es solo una proyección del log. Si se corrompe, se puede reconstruir completamente haciendo replay desde el offset 0.

🔄 Saga Pattern — transacciones distribuidas
Coordinar flujos de negocio entre microservicios

Una "saga" es una secuencia de transacciones locales. Si alguna falla, se ejecutan transacciones compensadoras para deshacer el efecto de las anteriores.

graph TD O[Orchestrator] --> S1[Topic: reservar-stock] S1 -->|OK| S2[Topic: pago] S2 -->|OK| S3[Topic: envio] S3 -->|OK| S4[Topic: confirmacion-pedido] S2 -.->|Fallo Pago| C1[Topic: compensacion-stock] C1 -.-> C2[Topic: cancelar-pedido] 
import faust

app = faust.App('saga-orquestador', broker='kafka://localhost:9094')

class ResultadoPago(faust.Record):
    pedido_id: str
    exito: bool
    motivo: str

topic_resultado_pago   = app.topic('resultado-pago',        value_type=ResultadoPago)
topic_compensar_stock  = app.topic('compensacion-stock',    value_type=dict)
topic_notificar        = app.topic('notificaciones',         value_type=dict)

@app.agent(topic_resultado_pago)
async def manejar_resultado_pago(resultados):
    async for resultado in resultados:
        if resultado.exito:
            # Continuar saga: siguiente paso
            await app.topic('crear-envio').send(
                key=resultado.pedido_id,
                value={"pedido_id": resultado.pedido_id}
            )
        else:
            # Compensar: liberar el stock reservado
            await topic_compensar_stock.send(
                key=resultado.pedido_id,
                value={"pedido_id": resultado.pedido_id, "accion": "liberar"}
            )
            # Notificar al usuario del fallo
            await topic_notificar.send(
                key=resultado.pedido_id,
                value={"tipo": "pedido_cancelado", "motivo": resultado.motivo}
            )

⚡ Integración con el ecosistema Big Data
🔥 Kafka + Apache Spark Streaming
from pyspark.sql import SparkSession
from pyspark.sql.functions import from_json, col
from pyspark.sql.types import *

spark = SparkSession.builder \
    .appName("KafkaStreaming") \
    .config("spark.jars.packages",
            "org.apache.spark:spark-sql-kafka-0-10_2.12:3.5.0") \
    .getOrCreate()

# Leer de Kafka como stream
df = spark.readStream \
    .format("kafka") \
    .option("kafka.bootstrap.servers", "localhost:9094") \
    .option("subscribe", "pedidos") \
    .option("startingOffsets", "earliest") \
    .load()

schema = StructType([
    StructField("pedido_id", StringType()),
    StructField("total", DoubleType()),
    StructField("usuario_id", StringType()),
])

# Deserializar JSON
pedidos = df.select(
    from_json(col("value").cast("string"), schema).alias("data")
).select("data.*")

# Aggregation: total de ventas por usuario
totales = pedidos.groupBy("usuario_id").sum("total")

query = totales.writeStream \
    .format("console") \
    .outputMode("complete") \
    .start()

query.awaitTermination()
🗄️ Kafka → Data Lakehouse (Delta Lake / Iceberg)
# Escribir de Kafka directamente a Delta Lake
from pyspark.sql import SparkSession

spark = SparkSession.builder \
    .appName("KafkaToDelta") \
    .config("spark.sql.extensions",
            "io.delta.sql.DeltaSparkSessionExtension") \
    .getOrCreate()

kafka_df = spark.readStream \
    .format("kafka") \
    .option("kafka.bootstrap.servers", "localhost:9094") \
    .option("subscribe", "pedidos") \
    .load()

# Escribir a Delta Lake en modo streaming
query = kafka_df.writeStream \
    .format("delta") \
    .outputMode("append") \
    .option("checkpointLocation", "/tmp/kafka-delta-checkpoint") \
    .start("/data/lake/pedidos")

# Ahora dbt puede leer la tabla Delta con SQL ✅
# Spark SQL: SELECT * FROM delta.`/data/lake/pedidos`
🏗️ Este es el patrón Lambda/Kappa moderno: Kafka captura los eventos en tiempo real, Spark los escribe al Data Lake, y dbt los transforma para analítica.
