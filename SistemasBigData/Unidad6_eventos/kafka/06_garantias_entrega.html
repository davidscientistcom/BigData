🔒 Garantías de Entrega

At-most-once, at-least-once y exactly-once. Las configuraciones acks, idempotence y transacciones Kafka. El triángulo de imposibilidades y cuándo importa cada garantía.

🔢 Las 3 garantías de entrega
El triángulo: Velocidad — Fiabilidad — Complejidad

🔑 acks — el parámetro más importante del Producer
Valor Significado Pérdida posible Rendimiento
0 Fire and forget — no esperar confirmación ✅ Alta ⚡ Máximo
1 Confirma el líder de la partición 🟡 Si el líder cae antes de replicar 🔄 Alto
all o -1 Confirma el líder + todas las réplicas in-sync ❌ Cero (con min.insync.replicas=2) 🐢 Moderado 
# Producción: acks=all siempre
conf = {
    'acks': 'all',
    'min.insync.replicas': 2,  # configurar en el topic
}
🔄 Idempotencia del Producer

Con retries activados, un producer puede enviar el mismo mensaje dos veces si no recibe la confirmación a tiempo. El producer idempotente usa un sequence number para que el broker detecte y descarte duplicados.

# Habilitar idempotencia — SIEMPRE en producción
conf = {
    'enable.idempotence': True,
    # Automáticamente establece:
    # acks = 'all'
    # retries = MAX_INT
    # max.in.flight.requests.per.connection = 5
}

# Resultado: exactamente 1 copia del mensaje en Kafka
# incluso si el producer reintenta 100 veces
✅ enable.idempotence=True tiene un coste de rendimiento mínimo (≈1-2%) y garantiza que ningún mensaje se duplique en el broker. Úsalo siempre.
⚡ Transacciones Kafka — Exactly-once end-to-end
Leer-Procesar-Escribir atómicamente

El patrón más potente: leer de un topic de entrada, procesar, escribir en otro topic, y commitear el offset de entrada — todo en una sola transacción atómica.

from confluent_kafka import Producer, Consumer, TopicPartition

# ─── Producer Transaccional ──────────────────────────────────────────────────
producer = Producer({
    'bootstrap.servers': 'localhost:9094',
    'transactional.id': 'mi-app-transaccional-1',  # único por instancia de app
    'enable.idempotence': True,
})

consumer = Consumer({
    'bootstrap.servers': 'localhost:9094',
    'group.id': 'procesador-transaccional',
    'isolation.level': 'read_committed',  # ← solo lee mensajes committed
    'enable.auto.commit': False,
})

# Inicializar transacciones (una vez al arrancar)
producer.init_transactions()
consumer.subscribe(['pedidos-entrada'])

while True:
    msgs = consumer.consume(num_messages=100, timeout=1.0)
    if not msgs: continue

    producer.begin_transaction()
    try:
        for msg in msgs:
            if msg.error(): continue

            pedido = json.loads(msg.value())
            # Transformar
            pedido_enriquecido = enriquecer(pedido)

            # Escribir resultado en topic de salida
            producer.produce(
                topic='pedidos-procesados',
                key=msg.key(),
                value=json.dumps(pedido_enriquecido).encode()
            )

        # Commitear los offsets como parte de la transacción
        offsets = [TopicPartition(m.topic(), m.partition(), m.offset() + 1) for m in msgs]
        producer.send_offsets_to_transaction(offsets, consumer.consumer_group_metadata())

        # Commit atómico: si falla, nada se escribe ni se commitea
        producer.commit_transaction()

    except Exception as e:
        producer.abort_transaction()
        print(f"Transacción abortada: {e}")
💡 Cuándo necesitas transacciones: pagos, reservas de inventario, transferencias bancarias — cualquier caso donde procesar un mensaje dos veces tenga consecuencias económicas o de negocio.
Dead Letter Queue (DLQ) — para mensajes que no se pueden procesar
# Patrón DLQ: si el procesamiento falla N veces, manda el mensaje a un topic especial
import json
from confluent_kafka import Consumer, Producer

DLQ_TOPIC = "pedidos-dlq"
MAX_REINTENTOS = 3

def procesar_con_dlq(consumer: Consumer, producer: Producer):
    reintentos: dict = {}

    while True:
        msg = consumer.poll(1.0)
        if msg is None or msg.error(): continue

        msg_id = f"{msg.topic()}-{msg.partition()}-{msg.offset()}"
        intentos = reintentos.get(msg_id, 0) + 1

        try:
            evento = json.loads(msg.value())
            procesar(evento)
            consumer.commit(asynchronous=False)
            reintentos.pop(msg_id, None)

        except Exception as e:
            if intentos >= MAX_REINTENTOS:
                # Mandar a DLQ con metadatos del error
                dlq_value = {
                    "original_topic": msg.topic(),
                    "original_partition": msg.partition(),
                    "original_offset": msg.offset(),
                    "error": str(e),
                    "payload": msg.value().decode()
                }
                producer.produce(DLQ_TOPIC, value=json.dumps(dlq_value).encode())
                producer.flush()
                consumer.commit(asynchronous=False)  # avanzar offset
                reintentos.pop(msg_id, None)
            else:
                reintentos[msg_id] = intentos  # reintentar
🎯 Los mensajes en el DLQ pueden ser inspeccionados, corregidos y re-publicados manualmente. Es el mecanismo estándar para gestionar mensajes "poison pill".
