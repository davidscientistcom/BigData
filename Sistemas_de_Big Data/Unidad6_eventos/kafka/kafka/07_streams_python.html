🌊 Kafka Streams con Python: Faust

Procesamiento de streams en tiempo real con Faust (el Kafka Streams de Python). Transformaciones, filtros, aggregations, tablas y ventanas temporales.

🌊 ¿Qué es el stream processing?
Batch vs Stream — la diferencia fundamental
graph TD subgraph BATCH PROCESSING DB[Datos] -->|Esperar horas| PB[Procesar todos] --> RB[Resultado] end subgraph STREAM PROCESSING DS[Evento] -->|Inmediato| PS[Procesar] --> RS[Resultado <100ms] end
☕ El problema de "Kafka Streams" en Python
Kafka Streams NO es una funcionalidad del servidor Kafka

Existe una gran confusión en la industria: mucha gente cree que Kafka Streams es un motor interno que corre dentro del clúster de Kafka (como si fueran Triggers o o Procedimientos Almacenados de SQL). Esto es falso.

⚠️ La Verdad: Kafka es solo un disco duro mudo y ultra-rápido. "Kafka Streams" es puramente una librería cliente (un archivo .jar) creada por Confluent.

Entonces, ¿Puedo usar "Kafka Streams" en Python?

Nativamente NO. Dado que Kafka Streams está escrito en Java, solo puedes importarlo en aplicaciones Java, Scala o Kotlin.

¿Cuáles son las alternativas reales para Python?

graph LR K[(Kafka Broker)] -->|Java| KS[Kafka Streams Library] K -->|Python Falso| FK[❌ Imposible Nativo] K -->|Python Real| F[Faust] K -->|Big Data Distribuido| PF[Apache Flink / PyFlink]
  • Faust: Un clon de la arquitectura de Kafka Streams escrito 100% en Python Asyncio. Ideal para aplicaciones ligeras o microservicios estándar de backend en Python.
  • PyFlink: El motor estándar de las gigantes tecnológicas. Flink corre como un clúster masivo separado (basado en Java) pero te permite escribir la lógica en Python (PyFlink). Cubre casos donde el volumen de datos destruiría a Faust.
🐍 Faust — Kafka Streams para Python

# ─── app.py ──────────────────────────────────────────────────────────────────
import faust
from dataclasses import dataclass

app = faust.App(
    'mi-app-streams',
    broker='kafka://localhost:9094',
    value_serializer='json',
)

# ─── Definir el modelo del mensaje ───────────────────────────────────────────
class Pedido(faust.Record):
    pedido_id: str
    usuario_id: str
    total: float
    estado: str

# ─── Declarar los topics ──────────────────────────────────────────────────────
topic_pedidos = app.topic('pedidos', value_type=Pedido)
topic_alertas  = app.topic('alertas', value_type=str)

# ─── Agent: consume y procesa de forma continua ───────────────────────────────
@app.agent(topic_pedidos)
async def procesar_pedidos(pedidos):
    async for pedido in pedidos:
        print(f"[{pedido.pedido_id}] €{pedido.total} — {pedido.estado}")

        # Filtrar y enviar alerta si el pedido es grande
        if pedido.total > 1000:
            await topic_alertas.send(
                key=pedido.usuario_id,
                value=f"PEDIDO GRANDE: {pedido.pedido_id} = €{pedido.total}"
            )

# Ejecutar:
# faust -A app worker -l info
💡 @app.agent es el equivalente de un consumer loop pero con async/await nativo, paralelismo automático, fault-tolerance y reinicio automático ante errores.
📊 Tablas — estado persistente en streams

Las tablas de Faust son diccionarios distribuidos y persistentes respaldados por un topic Kafka compactado. Permiten mantener estado entre eventos sin una base de datos externa.

import faust

app = faust.App('contador', broker='kafka://localhost:9094')

class Click(faust.Record):
    url: str
    usuario_id: str

topic_clicks = app.topic('clicks', value_type=Click)

# Tabla: cuenta clicks por URL (sobrevive reinicios de la app)
conteos_url = app.Table(
    'conteos_url',
    default=int,        # valor por defecto si la clave no existe@app.agent(topic_clicks)
async def contar_clicks(clicks):
    async for click in clicks:
        conteos_url[click.url] += 1
        print(f"{click.url}: {conteos_url[click.url]} clicks")

# También puedes exponer los resultados vía HTTP:
@app.page('/clicks/{url}/')
@app.table_related(conteos_url)
async def get_url_clicks(web, request, url):
    return web.json({'url': url, 'clicks': conteos_url[url]})
⏱️ Ventanas temporales — aggregations en tiempo real
import faust
from datetime import timedelta

app = faust.App('ventanas', broker='kafka://localhost:9094')

class Venta(faust.Record):
    producto_id: str
    monto: float

topic_ventas = app.topic('ventas', value_type=Venta)

# Tabla con ventana TUMBLING de 1 minuto
# Acumula ventas en ventanas no solapadas de 60s
ventas_por_minuto = app.Table(
    'ventas_minuto',
    default=float,
).tumbling(
    timedelta(minutes=1),          # tamaño ventana
    expires=timedelta(hours=1),    # cuánto guardar
)

@app.agent(topic_ventas, sink=[topic_ventas])
async def acumular_ventas(ventas):
    async for venta in ventas:
        ventas_por_minuto[venta.producto_id] += venta.monto
        total_minuto = ventas_por_minuto[venta.producto_id].current()
        print(f"[1min] {venta.producto_id}: €{total_minuto:.2f}")

# Tipos de ventanas:
# .tumbling(size)         → ventanas fijas no solapadas (0-60s, 60-120s...)
# .hopping(size, step)    → ventanas solapadas (0-60s, 30-90s, 60-120s...)
✅ Las ventanas temporales permiten responder preguntas como "¿cuántas ventas en la última hora?" en tiempo real, sin necesitar una base de datos.
🔄 Pipeline completo: enriquecimiento en tiempo real
Patrón real: enriquecer eventos con lookup a BD
import faust
import asyncio

app = faust.App('enriquecedor', broker='kafka://localhost:9094')

class EventoRaw(faust.Record):
    user_id: str
    event_type: str
    timestamp: float

class EventoEnriquecido(faust.Record):
    user_id: str
    user_email: str
    user_plan: str
    event_type: str
    timestamp: float

topic_raw       = app.topic('eventos-raw',        value_type=EventoRaw)
topic_enriched  = app.topic('eventos-enriched',   value_type=EventoEnriquecido)

# Caché en memoria para evitar consultas repetidas a BD
user_cache: dict = {}

async def get_user_info(user_id: str) -> dict:
    if user_id in user_cache:
        return user_cache[user_id]
    # En real: await db.fetch("SELECT email, plan FROM users WHERE id=$1", user_id)
    await asyncio.sleep(0.01)  # simular latencia BD
    info = {"email": f"{user_id}@ejemplo.com", "plan": "premium"}
    user_cache[user_id] = info
    return info

@app.agent(topic_raw)
async def enriquecer(eventos):
    async for evento in eventos:
        user_info = await get_user_info(evento.user_id)
        enriched = EventoEnriquecido(
            user_id=evento.user_id,
            user_email=user_info["email"],
            user_plan=user_info["plan"],
            event_type=evento.event_type,
            timestamp=evento.timestamp,
        )
        await topic_enriched.send(key=evento.user_id, value=enriched)
