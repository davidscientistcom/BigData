Laboratorio 05 — Python

Práctica Compilación: Protocol Buffers

Demostración práctica de la filosofía de Google. Usando protoc para autogenerar Clases Python desde un contrato estático y produciendo bytes directos en Kafka.

1. Preparación y Compilador

En el mundo de Protobuf original (sin meter Confluent de por medio), no necesitamos un Schema Registry escuchando en la red. Todo el trato se cierra estáticamente compilando (o "transpirando") el archivo.

# 1. Instalar el compilador oficial de Google y la librería protobuf para Python
# (En Mac): brew install protobuf
# (En Ubuntu): sudo apt install protobuf-compiler

pip install protobuf kafka-python

2. El Contrato: El Archivo `.proto`

Crea el siguiente documento en la raíz. Presta atención a que asignamos números estrictamente matemáticos (Los Tags) a cada variable.

sensor_data.protoProto3
syntax = "proto3";

package bootcampprotobuf;

message SensorTelemetria {
  string sensor_id = 1;
  double temperatura = 2;
  bool alerta_roja = 3;
}

3. Paso Crítico: La Autogeneración (protoc)

No escribas un diccionario a mano. Dile al compilador de Google que lea el `.proto` y te genere una clase mágica de Python llamada sensor_data_pb2.py.

protoc -I=. --python_out=. ./sensor_data.proto

Ahora, si miras tu explorador de archivos, tendrás un script gigante que no debes tocar (sensor_data_pb2.py). Has traído el tipado fuerte y la clase compilada directo a Python.

4. El Productor Nativo

Nuestro Productor no usará librerías de Confluent ni Avro Serializer. Importará la clase de Google, inyectará las variables como si fuera Programación Orientada a Objetos, llamará a un método interno SerializeToString() y lanzará ese puré binario directamente al Kafka de toda la vida.

producer_protobuf.pyPython
from kafka import KafkaProducer
import sensor_data_pb2  # El fichero auto-generado por el milagro de protoc

# 1. Kafka clásico absoluto. Sin schema registry.
producer = KafkaProducer(bootstrap_servers='localhost:9092')

# 2. Instanciamos la clase fuerte que Google nos generó
telemetria = sensor_data_pb2.SensorTelemetria()
telemetria.sensor_id = "MOTOR_NUCLEO_1"
telemetria.temperatura = 94.6
telemetria.alerta_roja = True

# 3. La Serialización ocurre en la RAM de tu código local
payload_binario = telemetria.SerializeToString()

# 4. Enviar los bytes crudos
producer.send('telemetria_topic', value=payload_binario)
producer.flush()

print("🔥 Protobuf disparado con éxito.")
¿Ves la Diferencia Arquitectónica? En la práctica 4 de Avro dependías de un servidor externo (Schema Registry en `localhost:8081`) intercediendo en cada arranque. Aquí, dependes de que tu Equipo Data Science haya ejecutado físicamente el comando Bash protoc antes de arrancar.

5. El Consumidor (La Transmutación Inversa)

El consumidor simplemente se conecta al topic de Kafka, recibe 15 bytes ininteligibles y le pide a la clase auto-generada local de Google que actúe de decodificador (usando los Tags numéricos como piedra roseta).

consumer_protobuf.pyPython
from kafka import KafkaConsumer
import sensor_data_pb2

consumer = KafkaConsumer(
    'telemetria_topic',
    bootstrap_servers='localhost:9092',
    group_id='monitor_g1',
    auto_offset_reset='earliest'
)

print("🎧 Descodificando bytes en vivo vía Google Protobuf...")

for message in consumer:
    # Recibimos basura binaria cruda de la red
    bytes_raw = message.value 
    
    # Creamos una clase vacía y le ordenamos devorar los bytes
    datos_recibidos = sensor_data_pb2.SensorTelemetria()
    datos_recibidos.ParseFromString(bytes_raw)
    
    # Ahora recuperamos el tipado natural (str, double, bool)
    print(f"[ALERTA: {datos_recibidos.alerta_roja}] Sensor {datos_recibidos.sensor_id} reporta {datos_recibidos.temperatura} ºC")
