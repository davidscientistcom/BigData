🏭 Kafka en Producción

Configuración del broker para alta disponibilidad, retención, compactación, monitoreo con Prometheus y Grafana, alertas y troubleshooting de los problemas más comunes.

⚙️ Configuración del broker para producción
Los parámetros que más importan en server.properties
Parámetro Valor recomendado Por qué
num.partitions 6-12 Por defecto para nuevos topics
default.replication.factor 3 Alta disponibilidad (tolera 1 broker caído)
min.insync.replicas 2 Con acks=all: 2 réplicas deben confirmar
log.retention.hours 168 (7 días) Retención por tiempo
log.retention.bytes -1 (ilimitado) Retención por tamaño
log.segment.bytes 1073741824 (1GB) Tamaño de cada segmento
num.io.threads CPUs*2 o 8 Hilos de I/O para disk
num.network.threads 3-5 Hilos de red
socket.send.buffer.bytes 102400 Buffer de envío
auto.create.topics.enable false Prevenir topics mal nombrados en prod
unclean.leader.election.enable false Evitar pérdida de datos (no elegir réplica desactualizada)
🗜️ Compactación — topics como tablas

Un topic compactado solo guarda el último mensaje por clave. Es como una tabla KV distribuida.

# Crear topic compactado
kafka-topics --create --topic usuarios \
  --config cleanup.policy=compact \
  --config min.cleanable.dirty.ratio=0.1 \
  --config segment.ms=60000 \
  --config delete.retention.ms=86400000

# Caso de uso: estado de usuarios
# key="user_123" value={"nombre":"Ana","plan":"pro"}
# key="user_123" value={"nombre":"Ana","plan":"enterprise"}
# Kafka solo guarda la última: plan=enterprise

# Para borrar un usuario: enviar null value (tombstone)
producer.produce("usuarios", key=b"user_123", value=None)
📊 Cuotas y throttling

Para evitar que un producer o consumer consuma todo el ancho de banda del broker:

# Limitar un cliente específico
kafka-configs --bootstrap-server localhost:9092 \
  --alter --add-config \
  'producer_byte_rate=1048576,consumer_byte_rate=2097152' \
  --entity-type clients \
  --entity-name mi-cliente

# 1MB/s de escritura, 2MB/s de lectura
# Configurar en el producer automáticamente:
conf = {
    'client.id': 'mi-cliente',  # ← esta string aplica la cuota
}
📈 Monitoreo: Prometheus + Grafana
Stack de monitoreo completo
# Añadir a docker-compose.yml
kafka-exporter:
  image: danielqsj/kafka-exporter:latest
  command: "--kafka.server=kafka:9092"
  ports: ["9308:9308"]

prometheus:
  image: prom/prometheus:latest
  ports: ["9090:9090"]
  volumes:
    - ./prometheus.yml:/etc/prometheus/prometheus.yml

grafana:
  image: grafana/grafana:latest
  ports: ["3000:3000"]
Métrica Qué indica Umbral de alerta
kafka_consumer_group_lag Mensajes pendientes por consumer group >10000 o tendencia creciente
kafka_brokers Número de brokers activos < replication_factor
kafka_topic_partitions Particiones por topic Cambios inesperados
under_replicated_partitions Particiones sin suficientes réplicas >0 es crítico
kafka_log_log_size Tamaño del log en disco >80% capacidad disco
network_processor_avg_idle CPU libre en red <30% = saturación

🔍 Troubleshooting — los problemas más comunes
Diagnóstico rápido de los problemas más frecuentes
Síntoma Causa probable Solución
Consumer lag aumentando continuamente Consumer más lento que el producer Aumentar consumidores, optimizar procesamiento, aumentar particiones
LEADER_NOT_AVAILABLE Broker líder caído, reelección en curso Esperar 10-30s. Si persiste, verificar estado del broker.
GroupCoordinator not available Topic __consumer_offsets sin líder Verificar que hay brokers suficientes para el replication factor.
Mensajes fuera de orden Produciendo sin clave o max.in.flight>1 Usar clave consistente para datos que necesitan orden.
Rebalanceos frecuentes Consumer tarda demasiado (max.poll.interval.ms) Aumentar max.poll.interval.ms o reducir el tiempo de procesamiento.
Disco lleno en broker Retención configurada muy alta Reducir log.retention.hours o aumentar disco.
Producer bloqueado en produce() Buffer interno lleno (BufferError) Llamar a poll() regularmente o flush() antes de continuar.
Bajo throughput en producer linger.ms=0 y sin batching Aumentar linger.ms a 5-10ms y batch.size
🩺 Script de diagnóstico rápido
#!/bin/bash
# diagnostico_kafka.sh — ejecutar desde el contenedor de Kafka

echo "=== BROKERS ACTIVOS ==="
kafka-broker-api-versions --bootstrap-server localhost:9092 | grep -c "BROKER"

echo "=== TOPICS Y PARTICIONES ==="
kafka-topics --bootstrap-server localhost:9092 --describe | grep "Topic:"

echo "=== UNDER-REPLICATED PARTITIONS (debe ser 0) ==="
kafka-topics --bootstrap-server localhost:9092 \
  --describe --under-replicated-partitions

echo "=== LAG POR CONSUMER GROUP ==="
kafka-consumer-groups --bootstrap-server localhost:9092 --list | \
  xargs -I{} kafka-consumer-groups --bootstrap-server localhost:9092 \
  --describe --group {}

echo "=== OFFSETS DEL TOPIC ==="
kafka-run-class kafka.tools.GetOffsetShell \
  --broker-list localhost:9092 \
  --topic mi-topic
