Productores y Consumidores

Cómo publicar y leer mensajes en Kafka

📤 Productores: Publicar Mensajes

Un productor es tu aplicación que envía mensajes a Kafka. Puede ser un servidor web, un sensor IoT, un job de batch, lo que sea.

📤

Estructura de un Mensaje

Cada mensaje tiene:

  • Topic: Dónde va (ej: "compras")
  • Key (opcional): Para particionamiento (ej: user_id)
  • Value: Los datos (JSON, Avro, bytes, etc)
  • Headers (opcional): Metadata (timestamps, trace_id, etc)
  • Timestamp: Automático, generado por Kafka

Código del Productor (Python)

# Básico
from kafka import KafkaProducer
import json

producer = KafkaProducer(
    bootstrap_servers=['localhost:9092'],
    value_serializer=lambda v: json.dumps(v).encode('utf-8')
)

# Enviar mensaje
future = producer.send('compras', value={
    'user_id': 'user123',
    'amount': 99.99,
    'product': 'Laptop'
})

# Esperar confirmación
try:
    record_metadata = future.get(timeout=10)
    print(f"Enviado a partition {record_metadata.partition} "
          f"con offset {record_metadata.offset}")
except Exception as e:
    print(f"Error: {e}")

🎯 Send Async

El productor envía y continúa sin esperar. Más rápido pero menos seguro.

⏱️ Send Sync

El productor espera confirmación. Más lento pero garantiza que llegó.

🔄 Send con Callback

El productor envía y llama una función cuando llega el ACK.

Configuración Importante del Productor

Parámetro Efecto
acks 0, 1, o "all" - Cuándo considerar el mensaje entregado
retries Cuántas veces reintentar si falla
batch.size Número de bytes para agrupar mensajes (optimiza throughput)
linger.ms Esperar X ms antes de enviar un batch
compression.type snappy, gzip, lz4 - Comprimir antes de enviar
💡

Throughput vs Latencia

Mayor batch.size + linger.ms: Agrupa más mensajes, viaja menos veces a Kafka, pero cada mensaje tarda más en salir (mayor latencia).

Menor batch.size + linger.ms: Cada mensaje sale rápido, pero más viajes a Kafka (menor throughput).

📥 Consumidores: Leer Mensajes

Un consumidor es tu aplicación que lee mensajes de Kafka. Puede procesar, guardar, transformar, lo que necesites.

Código del Consumidor (Python)

from kafka import KafkaConsumer
import json

# Crear consumidor
consumer = KafkaConsumer(
    'compras',  # Topic
    bootstrap_servers=['localhost:9092'],
    group_id='analytics-group',  # ID del grupo
    auto_offset_reset='earliest',  # Empezar desde el inicio
    value_deserializer=lambda m: json.loads(m.decode('utf-8'))
)

# Leer mensajes
for message in consumer:
    print(f"Leído: {message.value}")
    print(f"Topic: {message.topic}")
    print(f"Partición: {message.partition}")
    print(f"Offset: {message.offset}")
    
    # Procesar...
    process(message.value)
    
    # Commit (guardar offset)
    consumer.commit()

👥 Grupo de Consumo

Todos los consumidores con el mismo group_id forman un grupo. Kafka automáticamente distribuye particiones entre ellos.

📍 Auto-Offset

auto_offset_reset define qué pasa si no hay offset guardado: "earliest" (desde el inicio) o "latest" (solo nuevos).

🔄 Polling

El consumidor hace polling (pregunta) a Kafka periódicamente. Por defecto cada 100ms. Configurable con heartbeat.interval.ms.

Configuración Importante del Consumidor

Parámetro Efecto
group_id Identificador del grupo de consumidores
auto_offset_reset "earliest" o "latest" - Qué pasa sin offset
max_poll_records Máximo de mensajes por poll (default: 500)
session.timeout.ms Si no responde en X ms, se considera muerto (default: 10s)
enable_auto_commit Auto-commit de offset (default: true)

Garantías End-to-End

Combinar la configuración del productor y consumidor te da garantías globales:

↕️ At-Most-Once

Productor: acks=0

Consumidor: Auto-commit inmediato

⚠️ RIESGO: Se pierden mensajes si algo falla

📍 Caso: Logs, telemetría no crítica

✅ At-Least-Once

Productor: acks=1 o all + retries

Consumidor: Commit DESPUÉS de procesar

⚠️ RIESGO: Se pueden duplicar mensajes

📍 Caso: Mayoría de usar (de-duplicate en tu lógica)

💎 Exactly-Once

Productor: acks=all + idempotent=true

Consumidor: Transacciones atómicas (process + commit)

✅ SEGURO: Garantía total

📍 Caso: Finanzas, crítico

🔧

Exactly-Once: Cómo Funciona

Kafka 0.11+ soporta transacciones atómicas:

  1. Productor envía con enable.idempotence=true (evita duplicados)
  2. Consumidor procesa Y commita atomicamente (todo o nada)
  3. Si crashea entre ambos, se reinicia desde antes de procesar
  4. Resultado: 0 duplicados, 0 pérdidas

Patrones de Consumo

1️⃣ Patrón: Un Consumidor, Un Topic

Caso simple: 1 aplicación lee completamente un topic.

2️⃣ Patrón: Consumer Group

3 instancias de la misma app, cada una procesa 1 partición en paralelo.

3️⃣ Patrón: Fan-Out (Múltiples Consumidores)

Varios grupos de consumo leyendo el MISMO topic.

Cada grupo mantiene su propio offset. Perfecto para arquitecturas de eventos.

4️⃣ Patrón: Seek (Replay)

Volver atrás en el log para reprocess.

# Ir al offset 100
consumer.seek(partition, 100)

# Ir al inicio
consumer.seek(partition, 0)

# Ir a hace 1 hora
timestamp_1h_ago = int(time.time() * 1000) - (3600 * 1000)
offset = consumer.offsets_for_times({partition: timestamp_1h_ago})
consumer.seek(partition, offset[partition].offset)
