Un Cluster de Kafka

Kafka no es un único servidor. Es un cluster distribuido de múltiples brokers (servidores) que trabajan juntos. Cada broker puede fallar y el sistema sigue en pie.

¿Qué es ZooKeeper?

ZooKeeper es el coordinador del cluster. Maneja: elecciones de líderes, sincronización, metadata del cluster. Sin ZK, Kafka no podría coordinar.

⚙️ ZooKeeper vs KRaft: Dos Caminos para Coordinar

Kafka tiene dos formas de coordinarse internamente:

🔴 ZooKeeper (Legacy)

¿Cómo funciona?

  • Componente externo separado del cluster
  • Maneja elecciones de leadership y ISR
  • Almacena metadata del cluster
  • Broker Kafka + ZooKeeper separados

✅ Pros: Maduro, probado

❌ Contras: Complejo, operacionalmente pesado

📦 Requerido en Kafka < 3.3

🟢 KRaft (Moderno)

¿Cómo funciona?

  • Coordinación nativa dentro de Kafka
  • Usa algoritmo de consenso (basado en Raft)
  • Brokers especiales actúan como "Controllers"
  • No requiere componentes externos

✅ Pros: Simplificación, mejor rendimiento

❌ Contras: Más reciente (pero production-ready)

🚀 Disponible 3.3+, recomendado

📊 Comparativa

Aspecto ZooKeeper KRaft
Dependencias Componente externo Integrado en Kafka
Setup Complejo (múltiples servicios) Simple (solo Kafka)
Nodos mínimos ZK: 3-5 + Kafka: 3 = 6-8 Kafka: 3 (algunos Controllers)
✅ Recomendado Solo si es obligatorio Sí, para nuevos proyectos
🎯 TL;DR: Si usas Kafka 3.3+ → KRaft. Si necesitas compatibilidad con versiones antiguas → ZooKeeper.

Replicación y Tolerancia a Fallos

Cada partición tiene réplicas en múltiples brokers. Si uno falla, la otra sigue sirviendo.

Ejemplo: Topic con 3 Brokers, Factor Replicación = 2

Partición 0:
LEADER Broker 1
ISR Broker 2
Partición 1:
LEADER Broker 2
ISR Broker 3
Partición 2:
LEADER Broker 3
ISR Broker 1
Términos:
  • LEADER: La réplica que recibe escrituras/lecturas
  • ISR (In-Sync Replica): Réplicas sincronizadas y listas para ser leader
  • Factor de Replicación: Cuántas copias se mantienen (típico: 3)

Persistencia: El Log Distribuido

Kafka usa un log (un buffer circular de datos) persistido en disco. Esto permite replay y tolerancia a fallos.

💾

¿Cómo funciona el log?

Cada partición es un log secuencial. Los mensajes se escriben en orden y se indexan. Los consumers pueden saltar a cualquier punto del log (offset). ¡Por eso es replayable!

Configuración de Retención:
  • retention.ms - Retener mensajes por X milisegundos
  • retention.bytes - Retener hasta X bytes
  • log.cleanup.policy - delete (borrar viejo) o compact (mantener última versión)

✍️ Write Path

Productor envía → LEADER escribe → Followers replican → ACK → Retorna

📖 Read Path

Consumidor pregunta → LEADER sirve (siempre del LEADER para consistencia)

⚡ Async Replication

Followers copian en background. Es RÁPIDO pero puede perder datos si todo falla al mismo tiempo.

Min ISR: Control de Seguridad

min.insync.replicas es una configuración crucial. Determina cuántos followers deben confirmar la escritura antes de que se considere "segura".

min.insync.replicas = 1

⚠️ INSEGURO

Solo el LEADER confirma. Si cae, se pierden mensajes no replicados.

Caso: Logs, no crítico

min.insync.replicas = 2

⚡ BALANCE

LEADER + 1 Follower confirman. Buen balance entre seguridad y performance.

Caso: Mayoría de usos

min.insync.replicas = 3

✅ SEGURO

LEADER + 2 Followers confirman. Máxima seguridad, más lento.

Caso: Finanzas, crítico

Multiple Topics en Un Cluster

Un cluster de Kafka puede alojar múltiples topics. Cada uno con su propia distribución de particiones.

📊

Ventaja de múltiples topics

Separas diferentes streams de eventos. Ejemplo: eventos-compras, logs-api, eventos-usuario. Cada uno con su política de retención, particiones, etc.

Niveles de Acknowledgement (ACK)

El productor puede configurar cuándo considera que un mensaje se "envió exitosamente":

acks = 0

🚀 Fire & Forget

Productor no espera confirmación. El mensaje puede perderse.

Velocidad: ⚡⚡⚡ (Máxima)
Seguridad: ❌ (Ninguna)

Caso: Metrics, telemetría no crítica

acks = 1

⚡ Leader ACK

Productor espera que el LEADER confirme. Si el leader cae antes de replicar, se pierde.

Velocidad: ⚡⚡ (Rápido)
Seguridad: ⚠️ (Media)

Caso: Mayoría de casos

acks = all

💎 Full Replication

Productor espera que LEADER + min.insync.replicas confirmen. Máxima seguridad.

Velocidad: ⚡ (Lento)
Seguridad: ✅ (Máxima)

Caso: Finanzas, crítico
