Operadores Paso a Paso

Qué hace cada operador con tus datos, explicado con ejemplos reales muy claros. Sin código: solo datos que entran y datos que salen.

📖
Cómo leer esta página: Para cada operador verás tres columnas: Datos de entrada (lo que llega al operador), el Operador (la transformación que aplica) y Datos de salida (lo que produce). Todo con un ejemplo concreto y real.
🔄

map — Transformar elemento a elemento

1 entrada → 1 salida
¿Qué hace? Aplica una función a cada elemento del stream y devuelve exactamente un elemento transformado. Es como una cadena de montaje: cada pieza entra, la pintan, y sale pintada. Ni más ni menos piezas.
📐 Para cada elemento X → produce f(X)

Ejemplo: Tenemos un stream de temperaturas en Celsius y queremos convertirlas a Fahrenheit.

📥 Datos de entrada (°C)
0 °C
20 °C
37 °C
100 °C
→
map()
Aplica la fórmula: F = (C × 9/5) + 32
→
📤 Datos de salida (°F)
32 °F
68 °F
98.6 °F
212 °F
🏭 Analogía: El cajero de un banco que convierte euros a dólares. Cada billete entra, pasa por la conversión, y sale convertido. El número de billetes no cambia.

Otro ejemplo: Extraer solo el nombre de usuario de un stream de emails.

📥 Entrada (email completo)
ana@empresa.com
luis.garcia@empresa.com
sara_2024@empresa.com
→
map()
Extraer lo que va antes del @
→
📤 Salida (solo usuario)
ana
luis.garcia
sara_2024
python — map en PyFlink
# Convertir Celsius a Fahrenheit
stream_celsius.map(lambda celsius: (celsius * 9/5) + 32)

# Extraer usuario del email
stream_emails.map(lambda email: email.split('@')[0])

# En Java:
// stream.map(celsius -> (celsius * 9.0/5) + 32);
🔀

flatMap — Expandir elementos

1 entrada → 0, 1 o N salidas
¿Qué hace? Aplica una función a cada elemento y puede devolver cero, uno o varios elementos. Es como si cogemos una caja de chocolatinas y la abrimos: de una caja salen muchas chocolatinas individuales.
📐 Para cada elemento X → produce [f1(X), f2(X), f3(X), ...]

Ejemplo: Tenemos un stream de frases y queremos obtener un stream de palabras individuales.

📥 Entrada (frases)
"Hola Mundo Flink"
"Big Data es genial"
→
flatMap()
Divide cada frase en palabras individuales
→
📤 Salida (palabras sueltas)
"Hola"
"Mundo"
"Flink"
"Big"
"Data"
"es"
"genial"
📬 Analogía: Un cartero que reparte cartas. De un sobre con varias hojas (una frase) saca cada hoja individualmente (cada palabra) y las pone en el buzón una a una.

Otro ejemplo: Un stream de pedidos con varios productos. Queremos un stream de cada producto individual.

📥 Entrada (pedidos)
Pedido A: [🍕 🥤 🍟]
Pedido B: [🍔]
Pedido C: [🍕 🍰]
→
flatMap()
Desglosa cada pedido en items individuales
→
📤 Salida (items)
🍕 (de Pedido A)
🥤 (de Pedido A)
🍟 (de Pedido A)
🍔 (de Pedido B)
🍕 (de Pedido C)
🍰 (de Pedido C)
python — flatMap en PyFlink
# Dividir frases en palabras
stream_frases.flat_map(
    lambda frase, collector:
        [collector.collect(palabra) for palabra in frase.split()]
)

# Expandir pedido en items individuales
stream_pedidos.flat_map(
    lambda pedido, collector:
        [collector.collect(item) for item in pedido.items]
)

filter — Mantener solo lo que cumple la condición

1 entrada → 0 o 1 salida
¿Qué hace? Evalúa una condición para cada elemento. Si la condición es verdadera, el elemento pasa. Si es falsa, se descarta. El número de elementos de salida siempre es ≤ al de entrada.
📐 Si condición(X) = TRUE → pasa | Si condición(X) = FALSE → se elimina

Ejemplo: Stream de transacciones bancarias. Solo nos interesan las de más de 1.000 €.

📥 Todas las transacciones
Tx 1: 50 €
Tx 2: 1.500 €
Tx 3: 230 €
Tx 4: 3.000 €
Tx 5: 80 €
Tx 6: 2.200 €
→
filter()
Condición: importe > 1.000 €
→
📤 Solo las grandes
Tx 1: 50 € ✗
Tx 2: 1.500 € ✓
Tx 3: 230 € ✗
Tx 4: 3.000 € ✓
Tx 5: 80 € ✗
Tx 6: 2.200 € ✓
🚪 Analogía: El portero de una discoteca. Revisa a cada persona y solo deja pasar a las que cumplen el criterio (edad, vestimenta...). Las que no cumplen se quedan fuera del stream.
python — filter en PyFlink
# Solo transacciones mayores de 1000€
stream_tx.filter(lambda tx: tx['importe'] > 1000)

# Solo errores HTTP 500
stream_logs.filter(lambda log: log['status'] == 500)

# Solo sensores de zona A con temperatura crítica
stream_sensores.filter(lambda s: s['zona'] == 'A' and s['temp'] > 80)
🔑

keyBy — Separar el stream por grupos

1 stream → N grupos paralelos
¿Qué hace? Divide el stream en substreams independientes según una clave. Todos los eventos con la misma clave van siempre al mismo "carril". Esencial antes de cualquier operación con estado.
📐 Elementos con misma clave → mismo grupo → mismo estado independiente

Ejemplo: Tenemos ventas de distintos productos mezcladas en el stream. Queremos contar las ventas por producto por separado.

📥 Stream mezclado
venta: Laptop 1200€
venta: Móvil 599€
venta: Laptop 999€
venta: Tablet 349€
venta: Móvil 649€
venta: Laptop 1450€
→
keyBy()
clave = nombre del producto
→
📤 Grupos separados
🔑 Grupo: Laptop
Laptop 1200€
Laptop 999€
Laptop 1450€
🔑 Grupo: Móvil
Móvil 599€
Móvil 649€
🔑 Grupo: Tablet
Tablet 349€
🏪 Analogía: Las cajas registradoras de un supermercado por sección (carnicería, panadería, pescadería). Cada producto va a su caja correspondiente. Las personas en la cola de carnicería no se mezclan con las de panadería.
⚠️
keyBy NO ordena ni agrega por sí solo: solo separa. Después de keyBy normalmente aplicas reduce(), aggregate() o una ventana para calcular algo por cada grupo.
📉

reduce — Acumular resultados

N elementos → 1 resultado acumulado
¿Qué hace? Va acumulando los resultados de una operación. Combina el resultado anterior con el elemento nuevo y emite el acumulado actualizado. Siempre opera después de keyBy, por grupo.
📐 resultado_nuevo = combinar(resultado_anterior, elemento_actual)

Ejemplo: Suma acumulada de ventas por producto (running total).

📥 Ventas de Laptop (grupo)
1.200 €
999 €
1.450 €
750 €
→
reduce()
acumulado + nuevo = nuevo acumulado
→
📤 Total acumulado emitido
1.200 €
2.199 €
3.649 €
4.399 €
🏌️ Analogía: El marcador de puntuación en un partido de tenis. Con cada punto que gana un jugador, el marcador se actualiza y muestra el total acumulado. El marcador no espera al final del partido: se actualiza en tiempo real.
python — reduce: suma acumulada de ventas
# Suma acumulada de ventas por producto
stream_ventas
    .key_by(lambda v: v['producto'])
    .reduce(lambda acumulado, nuevo: {
        'producto': acumulado['producto'],
        'total': acumulado['total'] + nuevo['importe']
    })
🪟

window + aggregate — Calcular sobre intervalos de tiempo

N elementos en ventana → 1 resultado por ventana
¿Qué hace? Agrupa los eventos que llegan en un intervalo de tiempo (ej: 5 minutos) y los convierte en un único resultado. Perfecto para informes periódicos, medias, sumas... en tiempo real.
📐 Eventos del minuto 12:00 → 12:05 → calcular(todos) → 1 resultado a las 12:05

Ejemplo: Contar los clicks en el botón "Comprar" cada minuto para saber cuándo hay más actividad.

📥 Stream de clicks
Click 12:00:05
Click 12:00:23
Click 12:00:45
Click 12:00:58
Click 12:01:10
Click 12:01:33
→
window(1 min)
Tumbling Window de 1 minuto + COUNT
→
📤 Un resultado por ventana
🪟 Ventana 12:00 - 12:01
4 clicks
🪟 Ventana 12:01 - 12:02
2 clicks
⏱️ Analogía: El taxímetro de un taxi que te muestra la tarifa cada 5 minutos. No te cobra cada segundo individualmente: espera un intervalo de tiempo, suma todo lo acumulado, y te da el resultado del período.
python — window aggregate: clicks por minuto
from pyflink.datastream.window import TumblingEventTimeWindows
from pyflink.common.time import Time

# Contar clicks por botón, por minuto
stream_clicks
    .key_by(lambda c: c['boton'])
    .window(TumblingEventTimeWindows.of(Time.minutes(1)))
    .reduce(lambda a, b: {'boton': a['boton'], 'clicks': a['clicks'] + b['clicks']})
🤝

join — Combinar dos streams que se relacionan

2 streams → 1 stream enriquecido
¿Qué hace? Combina eventos de dos streams diferentes que comparten una clave común, dentro de una ventana de tiempo. Como un JOIN de SQL entre dos tablas, pero en tiempo real y con datos que están llegando constantemente.
📐 Evento A (clave X) + Evento B (clave X) dentro de ventana → Evento combinado (A+B)

Ejemplo: Tenemos un stream de pedidos y un stream de información de usuarios. Queremos combinarlos para saber qué tipo de cliente hace qué pedido.

📥 Stream: Pedidos
Pedido-1, user_id=42, 99€
Pedido-2, user_id=17, 250€
Pedido-3, user_id=42, 45€
+ 🤝 join
📥 Stream: Usuarios
user_id=42, "Ana", VIP
user_id=17, "Luis", Normal
→
📤 Pedidos enriquecidos
Pedido-1, Ana, VIP, 99€
Pedido-2, Luis, Normal, 250€
Pedido-3, Ana, VIP, 45€
📋 Analogía: Como cuando en un hospital combinan el historial médico del paciente con los resultados del análisis. Los dos tienen el DNI del paciente (clave común). Al unirlos, el médico ve la información completa.
💡
Diferencia con connect: join requiere la misma ventana temporal para los dos streams y genera un único tipo de salida. connect (ver abajo) no necesita ventana y permite lógica más flexible.
🔗

connect — Unir dos streams para lógica flexible

2 streams de tipos distintos → 1 stream de salida
¿Qué hace? Une dos streams para que puedan compartir estado e interactuar entre sí. Útil cuando un stream "controla" o "configura" al otro: por ejemplo, un stream de reglas de fraude que se actualiza en tiempo real y afecta cómo se procesan las transacciones.
📐 Stream A (datos) + Stream B (reglas/config) → procesarlos juntos con estado compartido

Ejemplo: Un stream de transacciones y un stream de reglas antifraude que cambian en tiempo real (el banco actualiza sus reglas sin parar el sistema).

📥 Stream 1: Transacciones
tx: usuario_99, 500€
tx: usuario_42, 8.000€
tx: usuario_07, 300€
→ 🔗 ←
📥 Stream 2: Reglas antifraude
Nueva regla: límite = 5.000€
Actualizar: límite = 7.500€
→
📤 Resultado
✅ tx 500€ — OK
🚨 tx 8.000€ — ALERTA
✅ tx 300€ — OK
🎛️ Analogía: Como el panel de control de un aeropuerto. El stream de vuelos (datos) y el stream de actualizaciones meteorológicas (config) llegan por separado. Pero los controladores usan ambos juntos para tomar decisiones: si cambia el tiempo, cambia cómo gestionan los vuelos en tiempo real.

Resumen Visual: ¿Cuándo usar cada operador?

Si quiero... Operador Ejemplo real
Transformar cada evento en otro evento map Celsius → Fahrenheit, extraer campo
De 1 evento sacar varios (o ninguno) flatMap Frase → palabras, pedido → items
Descartar eventos que no me interesan filter Solo tx > 1000€, solo errores 500
Separar eventos por grupos independientes keyBy Agrupar por usuario, producto, sensor
Acumular resultados en tiempo real reduce Suma running, máximo acumulado
Calcular algo cada N minutos/segundos window + aggregate Ventas por hora, clicks por minuto
Unir dos streams por clave común join Pedidos + datos de usuario
Un stream "configura" cómo procesar otro connect Reglas antifraude + transacciones
