SQL vs DataStream API

¿Qué API deberías elegir para tu proyecto? Desgranamos las diferencias clave y diseñamos una arquitectura completa basada en un caso real: procesar telemetría de aviones en tiempo real usando la API de OpenSky.

1. Tabla Resumen: ¿Cuándo usar qué?

La regla general en Flink es: Usa SQL siempre que puedas, y baja al DataStream API (Java/Python) cuando SQL no sea suficiente.

Característica Flink SQL / Table API DataStream API (Java/Scala/Python)
Curva de Aprendizaje Baja (Si sabes SQL, sabes Flink SQL) Alta (Requiere entender serialización, abstracciones funcionales, estado)
Casos de Uso Ideales Filtros, Agrupaciones temporales, ETLs simples, Joins estandar, Métricas BI. Lógica compleja de negocio, reglas dinámicas (Pattern Matching), llamadas a APIs externas.
Nivel de Control Bajo (El motor Catalyst optimiza todo por debajo) Total (Controlas el estado interno de Flink al byte y los Timers)
Gestión de Eventos Complejos Limitado (Pattern Recognition básico con MATCH_RECOGNIZE) Excelente (Usando ProcessFunction o la librería CEP de Flink)
Integraciones "Custom" Difícil (Si el conector no existe, tienes que programarlo en Java) Muy Fácil (Puedes escribir tu propio código de lectura/escritura)

2. Caso Práctico: Rastreador de Vuelos (OpenSky)

Supongamos que queremos construir un sistema que procese los datos de la OpenSky Network API (que da datos de vuelos en directo: altitud, velocidad, posición GPS, avión). Nuestro objetivo es detectar dos cosas:

  1. Métrica sencilla: Número total de aviones sobrevolando cada país (ventana de 5 mins).
  2. Regla compleja: Alerta si un avión cae más de 3000 metros de altitud en menos de 10 segundos.

Arquitectura General

📡 OpenSky API Polling origen
→
🐍 Python Script Productor Kafka
→
📨 Kafka Topic "vuelos-telemetria"
→
⚡ Apache Flink Procesamiento
→
🔔 Sinks Grafana (SQL) / Alertas (DataStream)
Por qué Kafka en medio: Nunca conectes Flink directamente haciendo polling a una API HTTP externa como OpenSky en la pieza de origen, porque Flink se bloqueará esperando respuestas de red, rompiendo el procesamiento paralelo y el "exactly-once". Siempre usa Kafka como buffer intermedio de streaming constante.

3. Resolviendo el Reto #1 con Flink SQL (Métricas)

Objetivo: Número total de aviones sobrevolando cada país en ventanas de 5 minutos.

Este es el escenario PERFECTO para Flink SQL. Las agrupaciones temporales y tabulaciones son el pan de cada día de las bases de datos relacionales. Implementar una "Tumbling Window" en DataStream API requiere unas 15 líneas de código Java. En Flink SQL son 5 líneas.

Escenario IDEAL para SQL 
-- 1. Declaramos la entrada de datos desde Kafka
CREATE TABLE vuelos (
  icao24 STRING,           -- ID único del avión
  callsign STRING,
  pais_origen STRING,
  logitud DOUBLE,
  latitud DOUBLE,
  altitud DOUBLE,
  velocidad DOUBLE,
  tiempo_lectura TIMESTAMP(3),
  WATERMARK FOR tiempo_lectura AS tiempo_lectura - INTERVAL '5' SECOND
) WITH (
  'connector' = 'kafka',
  'topic' = 'vuelos-telemetria',
  'format' = 'json'
);

-- 2. Declaramos la salida hacia un Elasticsearch / Kibana (Grafana)
CREATE TABLE metricas_espacio_aereo (
  ventana_inicio TIMESTAMP(3),
  pais STRING,
  total_aviones BIGINT
) WITH (
  'connector' = 'elasticsearch-7',
  'hosts' = 'http://localhost:9200',
  'index' = 'vuelos-metricas'
);

-- 3. La consulta mágica que Flink ejecutará eternamente
INSERT INTO metricas_espacio_aereo
SELECT 
  window_start,
  pais_origen,
  COUNT(DISTINCT icao24) AS total_aviones
FROM TABLE(
  TUMBLE(TABLE vuelos, DESCRIPTOR(tiempo_lectura), INTERVAL '5' MINUTE)
)
GROUP BY window_start, window_end, pais_origen;

Ventaja: Un analista de datos sin saber Java puede crear este dashboard en 10 minutos. Flink manejará por debajo la memoria de las ventanas, gestionará eventos retrasados y escribirá en Elastic Search automáticamente. Es elegante, declarativo y optimizado.

4. Resolviendo el Reto #2 con DataStream API (Reglas Complejas)

Objetivo: Alerta inmediata si un avión cae más de 3000 metros de altitud en menos de 10 segundos.

¿Podemos hacer esto en SQL? Casi podríamos usando la cláusula MATCH_RECOGNIZE de SQL Avanzado, pero se vuelve una consulta horriblemente compleja y difícil de mantener o testear unitariamente.

Este es el escenario PERFECTO para DataStream API y KeyedProcessFunction. Aquí necesitamos mantener un "Estado" manual (State) para recordar la altitud anterior de cada avión, y programar "Timers" (Temporizadores) y disparar un evento de alarma de forma programática.

Escenario IDEAL para DataStream API 
// Creamos la pipeline desde Kafka
DataStream<Vuelo> streamVuelos = env.addSource(new FlinkKafkaConsumer<>("vuelos", ...));

// Agrupamos el stream por ID de avión. Flink garantiza que el mismo avión
// siempre será procesado por el mismo nodo del cluster.
DataStream<AlertaCaida> flotasEnPeligro = streamVuelos
    .keyBy(vuelo -> vuelo.getIcao24()) 
    .process(new DetectorCaidaLibre()); // <-- Aquí ocurre la magia de bajo nivel

// --- LA FUNCIÓN DE NEGOCIO ---
public class DetectorCaidaLibre extends KeyedProcessFunction<String, Vuelo, AlertaCaida> {
    
    // Flink guardará esta variable de forma segura y tolerante a fallos
    private ValueState<Double> altitudMaximaReciente;

    @Override
    public void processElement(Vuelo vueloActual, Context ctx, Collector<AlertaCaida> out) {
        
        Double altitudAnterior = altitudMaximaReciente.value();
        
        if (altitudAnterior == null || vueloActual.getAltitud() > altitudAnterior) {
            // El avión está subiendo de altitud o es la primera vez que lo vemos
            altitudMaximaReciente.update(vueloActual.getAltitud());
            
            // Programamos un "Timer" (si pasan 10 segundos, borraremos el estado para no gastar RAM infinita)
            ctx.timerService().registerEventTimeTimer(ctx.timestamp() + 10000);
        } 
        else {
            // El avión ha bajado
            double caida = altitudAnterior - vueloActual.getAltitud();
            if (caida >= 3000.0) {
                // DISPARAR ALERTA EN TIEMPO REAL
                out.collect(new AlertaCaida(vueloActual.getIcao24(), caida));
                // Reseteamos el estado para no mandar alertas duplicadas
                altitudMaximaReciente.clear();
            }
        }
    }
    
    // Cuando pasan 10 segundos, limpiamos la memoria de ese avión
    @Override
    public void onTimer(long timestamp, OnTimerContext ctx, Collector<AlertaCaida> out) {
        altitudMaximaReciente.clear();
    }
}

Ventaja: Con DataStream API tienes el control absoluto del hardware y el flujo. Puedes leer y escribir variables en la memoria de disco gestionada por RocksDB (mediante ValueState), puedes programar temporizadores dinámicos (Timers), y reaccionar ante eventos a bajo nivel.

Conclusión del Dilema

💡
En arquitecturas modernas (como la que usan Netflix, Uber o Airbnb), no se elige una u otra exclusivamente.

A menudo, un pipeline DataStream limpia, enriquece y modeliza el ruido en crudo y lo guarda en un topic de Kafka limpio. Sobre ese topic limpio de Kafka, los Analistas y Data Engineers lanzan decenas de consultas de Flink SQL para rellenar diferentes dashboards sin molestar a los ingenieros de backend.
