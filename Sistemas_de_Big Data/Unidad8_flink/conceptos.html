Conceptos Fundamentales

Streams, eventos, estado, tiempo de evento vs procesamiento y watermarks. Todo lo que necesitas antes de programar con Flink.

1. ¿Qué es un Stream de Datos?

Un stream es una secuencia de datos que se genera de forma continua. A diferencia de una tabla estática, un stream es infinito y dinámico: siempre llegan nuevos datos.

Piensa en un río: el agua fluye constantemente. No puedes esperar a que el río "termine" para analizarlo. Tienes que analizar el agua mientras fluye.

Stream de eventos de compras — infinito en el tiempo
📊 Datos Estáticos (Batch)
  • Datos en reposo: archivos, BD
  • Tienes todos los datos disponibles
  • Alta latencia (minutos a horas)
  • Ej: informes diarios de ventas
⚡ Datos en Streaming
  • Datos en movimiento (eventos)
  • Los datos llegan continuamente
  • Baja latencia (milisegundos)
  • Ej: alertas de fraude en tiempo real

2. Eventos: los Átomos del Stream

Un evento es la unidad básica de un stream: un registro de algo que ocurrió en un momento concreto. Cada evento tiene un timestamp, unos datos (payload) y opcionalmente una clave (key) para agrupar eventos relacionados.

json — Ejemplo de evento en Flink 
// Un evento de compra típico en e-commerce
{
  "event_id":   "evt-8821-abc",
  "timestamp":  "2024-03-15T12:00:01.234Z",  // cuándo ocurrió
  "user_id":    "usr-42",             // clave para agrupar
  "event_type": "PURCHASE",
  "amount":     49.99,
  "currency":   "EUR"
}

3. Estado (State): la Memoria de Flink

Para detectar que un usuario hizo más de 5 compras en 1 minuto, Flink necesita recordar el contador de compras de ese usuario. Eso es el estado (state): la memoria que mantiene Flink entre eventos.

💡
Analogía: El estado es como la memoria de trabajo de un cajero de banco. Cada cliente (key) tiene su propio expediente (estado), y el cajero lo consulta y actualiza con cada acción.
🔑

Keyed State

Estado por clave. Cada usuario, producto o entidad tiene su propio estado independiente. El más habitual.

📋

ValueState

Guarda un único valor. Ej: total de ventas acumulado por usuario.

📝

ListState

Guarda una lista de valores. Ej: los últimos N eventos de un usuario.

🗺️

MapState

Guarda pares clave-valor por cada entidad del stream, como un diccionario.

java — ValueState para contar compras por usuario 
public class FraudDetector extends KeyedProcessFunction<String, Transaction, Alert> {

    private ValueState<Integer> purchaseCount;

    @Override
    public void open(Configuration params) {
        purchaseCount = getRuntimeContext().getState(
            new ValueStateDescriptor<>("purchase-count", Integer.class)
        );
    }

    @Override
    public void processElement(Transaction tx, Context ctx, Collector<Alert> out) throws Exception {
        Integer count = purchaseCount.value();
        if (count == null) count = 0;
        purchaseCount.update(++count);

        if (count > 5) {
            out.collect(new Alert(tx.getUserId(), "Posible fraude: demasiadas compras"));
        }
    }
}

4. Event Time vs Processing Time

🕐 Event Time
  • Cuándo ocurrió en el mundo real
  • Timestamp del propio evento
  • Resultado determinista y correcto
  • Gestiona eventos tardíos (watermarks)
  • ✅ Recomendado en producción
⏰ Processing Time
  • Cuándo el sistema recibe el evento
  • Reloj del servidor Flink
  • Resultado puede variar entre ejecuciones
  • Simple de implementar
  • ⚠️ Solo para demos y POCs
⚠️
¿Por qué importa? Un sensor IoT pierde conexión 2 minutos y luego envía todos los eventos acumulados. Con Processing Time, parecería que llegaron todos a la vez. Con Event Time, Flink sabe que ocurrieron hace 2 minutos y los asigna a la ventana correcta.

5. Watermarks: gestionar eventos tardíos

Los watermarks le dicen a Flink: "Ya no van a llegar eventos con timestamp anterior a X". Sin ellos, Flink no sabría cuándo cerrar una ventana temporal.

java — Configurar watermarks con tolerancia de 5 segundos 
DataStream<Transaction> stream = env
    .addSource(new KafkaSource(...))
    .assignTimestampsAndWatermarks(
        WatermarkStrategy
            .<Transaction>forBoundedOutOfOrderness(Duration.ofSeconds(5))
            .withTimestampAssigner((event, ts) -> event.getTimestamp())
    );
// Flink esperará 5s extra antes de cerrar ventanas por si llegan eventos tardíos

Característica Apache Flink Spark Streaming
