Ejemplos Progresivos

Aprende Flink SQL paso a paso. Desde un simple SELECT hasta ventanas de tiempo complejas (Tumbling, Hopping) y Joins en streaming continuo usando SQL estándar.

💡
Contexto para todos los ejemplos: Imagina que tenemos un e-commerce y nos entra un flujo infinito de datos desde Kafka en formato JSON que representa "Clicks" de usuarios en la web.

Nivel 0: Preparación (DDL)

Antes de hacer consultas continuas, Flink necesita saber cómo conectarse a la fuente de datos (Kafka) y qué esquema tienen los datos. Esto se hace con un comando DDL (Data Definition Language) estándar.

SQL 
CREATE TABLE clicks_stream (
  usuario_id INT,
  pais STRING,
  producto_id INT,
  precio DECIMAL(10,2),
  tiempo_evento TIMESTAMP(3),
  WATERMARK FOR tiempo_evento AS tiempo_evento - INTERVAL '5' SECOND
) WITH (
  'connector' = 'kafka',
  'topic' = 'web-clicks',
  'properties.bootstrap.servers' = 'localhost:9092',
  'format' = 'json'
);

Con esto, Flink ya puede consumir el topic de Kafka como si fuera una tabla mágica infinita.

Nivel 1: Filtrado y Transformación (Append Only)

Las operaciones más básicas no requieren mantener "estado". Flink lee un evento, lo evalúa y lo emite (o lo descarta) instantáneamente. Esto genera un flujo Append-Only (Solo inserciones).

Ejemplo: Quedarnos solo con clicks de España mayores a 50€

SQL 
SELECT 
  usuario_id, 
  producto_id, 
  precio, 
  tiempo_evento 
FROM clicks_stream
WHERE pais = 'ES' AND precio > 50.00;
Comportamiento en Streaming: Si entra un click de Francia, Flink no hace nada. Si entra uno de España de 60€, Flink emite una nueva fila inmediatamente como resultado. La tabla resultante solo crece.

Nivel 2: Agrupación Continua (Retract Stream)

Si introducimos un GROUP BY, Flink debe guardar el "estado" de esa agrupación en memoria y actualizarlo cada vez que llegue un evento nuevo.

Ejemplo: Total de ingresos acumulados por país

SQL 
SELECT 
  pais, 
  COUNT(*) AS total_clicks,
  SUM(precio) AS ingresos_totales
FROM clicks_stream
GROUP BY pais;
Retract Stream (Actualizaciones):

Entra: {pais: 'ES', precio: 10} → Flink emite: +I(ES, 1, 10)

Entra: {pais: 'ES', precio: 5} → Flink emite: -U(ES, 1, 10) (Borra anterior) y +U(ES, 2, 15) (Nueva suma)

Cuidado en producción con este tipo de queries porque guardan estado infinito (el "pais" nunca se borra de memoria).

Nivel 3: Tumbling Windows (Ventanas Fijas)

Para no acumular estado infinito y poder analizar el flujo de datos segmentado por tiempo, usamos Funciones de Ventana (Window TVF) introducidas en Flink 1.13+.

Una ventana Tumbling divide el tiempo en bloques fijos y no superpuestos (e.g. cada 1 minuto exacto).

Ejemplo: Ingresos totales por país cada 1 minuto cerrado

SQL 
SELECT 
  window_start, 
  window_end, 
  pais, 
  SUM(precio) AS total_ventas
FROM TABLE(
  TUMBLE(TABLE clicks_stream, DESCRIPTOR(tiempo_evento), INTERVAL '1' MINUTE)
)
GROUP BY window_start, window_end, pais;

¿Qué pasa aquí? Cuando el tiempo de Flink superó el minuto 10:01:00 (basado en Event Time y watermarks), Flink emite el resultado final para la ventana 10:00:00 - 10:01:00. Es un Append Only stream para el consumidor final, porque una vez pasada la ventana, el resultado "ya no cambia".

Nivel 4: Hopping Windows (Ventanas Deslizantes)

Las ventanas Hopping / Sliding se solapan. Tienen un tamaño (size) y un avance (slide).

Ejemplo: Dashboard en vivo. Cuenta de clicks de los "Últimos 10 minutos" actualizándose "Cada 1 minuto"

SQL 
SELECT 
  window_start, 
  window_end, 
  pais, 
  COUNT(*) AS clicks_activos
FROM TABLE(
  HOP(TABLE clicks_stream, DESCRIPTOR(tiempo_evento), INTERVAL '1' MINUTE, INTERVAL '10' MINUTE)
)
GROUP BY window_start, window_end, pais;

Un mismo click aparecerá sumado en 10 ventanas diferentes porque estás emitiendo un reporte cada minuto evaluando la última media hora.

Nivel 5: Time-windowed Joins

¿Qué ocurre si queremos hacer un JOIN entre DOS colas de Kafka que entran en tiempo real? En Flink SQL es de lo más avanzado y potente.

Por ejemplo, si tienes una tabla dinámica de pedidos_stream y otra de pagos_stream. El pago debe llegar máximo 10 minutos después del pedido.

Ejemplo: Interval Join

SQL 
SELECT
  p.pedido_id,
  p.usuario_id,
  p.total,
  g.metodo_pago
FROM pedidos_stream p
JOIN pagos_stream g 
  ON p.pedido_id = g.pedido_id
  AND g.tiempo_pago BETWEEN p.tiempo_pedido AND p.tiempo_pedido + INTERVAL '10' MINUTE;
Eficiencia de memoria: Como Flink sabe que el JOIN solo aplica a un margen de 10 minutos (gracias a BETWEEN), no guarda los pedidos más antiguos en memoria. ¡Esto permite a los ingenieros mantener Joins de streams infinitos sin saturar la RAM!
