Streams & Batch

DataStream API, ventanas temporales (tumbling, sliding, session), watermarks y procesamiento batch con Flink.

1. DataStream API: el corazón de Flink

La DataStream API es la API principal de Flink para procesamiento de streams. Representa un flujo de datos que puede ser transformado, filtrado y agregado mediante operadores encadenados.

java — Estructura básica de un job con DataStream API 
// 1. Crear el entorno de ejecución
StreamExecutionEnvironment env =
    StreamExecutionEnvironment.getExecutionEnvironment();

// 2. Definir la fuente (source)
DataStream<Transaction> transactions = env
    .addSource(new FlinkKafkaConsumer("transactions", schema, kafkaProps));

// 3. Transformar los datos
DataStream<Alert> alerts = transactions
    .filter(tx -> tx.getAmount() > 1000)         // solo importes altos
    .keyBy(Transaction::getUserId)              // agrupa por usuario
    .window(TumblingEventTimeWindows.of(Time.minutes(5)))
    .aggregate(new CountAndSumAggregator())       // cuenta y suma
    .filter(result -> result.getCount() > 3);         // más de 3 en 5 min

// 4. Escribir el resultado (sink)
alerts.addSink(new ElasticsearchSink(esConfig));

// 5. Ejecutar el job
env.execute("Detección de transacciones sospechosas");

2. Ventanas Temporales (Windows)

Las ventanas son fundamentales en streaming: permiten agrupar eventos en intervalos de tiempo para realizar agregaciones. Sin ventanas, no podríamos calcular "cuántas compras se hicieron en los últimos 5 minutos".

2.1 Tumbling Window (Ventana Fija)

Ventanas de tamaño fijo sin solapamiento. Cada evento pertenece exactamente a una ventana.

Tumbling Window de 5 minutos
java — Tumbling Window de 5 minutos 
stream
    .keyBy(Transaction::getUserId)
    // Ventana fija de 5 minutos en Event Time
    .window(TumblingEventTimeWindows.of(Time.minutes(5)))
    .sum("amount");  // suma de importes por usuario cada 5 min

2.2 Sliding Window (Ventana Deslizante)

Ventanas de tamaño fijo pero con solapamiento. Definidas por tamaño y paso (slide). Un evento puede pertenecer a varias ventanas.

Sliding Window: tamaño 10min, slide 5min
java — Sliding Window: media móvil de precio en 10 min cada 5 min 
stream
    .keyBy(Trade::getSymbol)
    // Ventana de 10 min, desliza cada 5 min
    .window(SlidingEventTimeWindows.of(
        Time.minutes(10),  // tamaño de la ventana
        Time.minutes(5)   // slide: cada 5 minutos se genera un resultado
    ))
    .aggregate(new AverageAggregate());  // precio medio

2.3 Session Window (Ventana de Sesión)

Agrupa eventos por períodos de actividad. Una ventana de sesión se cierra cuando pasa un tiempo X sin actividad. Ideal para analizar sesiones de usuario.

💡
Ejemplo práctico: Si un usuario navega en una web, cada clic es un evento. Si pasa más de 30 minutos sin hacer nada, consideramos que la sesión ha acabado. La siguiente acción abre una nueva sesión. Esto es exactamente lo que modela una Session Window.
java — Session Window: análisis de sesiones web 
stream
    .keyBy(WebEvent::getUserId)
    // Cierra la ventana si pasan más de 30 min sin actividad
    .window(EventTimeSessionWindows.withGap(Time.minutes(30)))
    .aggregate(new SessionSummaryAggregator());
// Resultado: duración total, páginas visitadas, tipo de sesión por usuario

3. Resumen de Ventanas

Tipo Solapamiento Caso de uso API
Tumbling No Reportes por períodos fijos (ventas cada 5 min) TumblingEventTimeWindows.of(Time.minutes(5))
Sliding Sí Medias móviles, detección de tendencias SlidingEventTimeWindows.of(size, slide)
Session Variable Análisis de sesiones, comportamiento de usuario EventTimeSessionWindows.withGap(Time.minutes(30))
Count No Procesar exactamente N eventos (micro-batches) GlobalWindows + CountTrigger.of(N)

4. Flink para Batch: Table API y SQL

Flink también puede procesar datos en batch (ficheros, datasets finitos). La Table API y Flink SQL ofrecen una interfaz de alto nivel similar a SQL que funciona igual para streaming y batch.

sql — Flink SQL: análisis de ventas en tiempo real 
-- Crear tabla que lee de Kafka
CREATE TABLE ventas (
    user_id     STRING,
    product_id  STRING,
    amount      DECIMAL(10, 2),
    ts          TIMESTAMP(3),
    WATERMARK FOR ts AS ts - INTERVAL '5' SECOND
) WITH (
    'connector' = 'kafka',
    'topic'     = 'ventas',
    'format'    = 'json'
);

-- Top usuarios por ventas en ventanas de 1 hora
SELECT
    user_id,
    TUMBLE_START(ts, INTERVAL '1' HOUR) AS hora,
    SUM(amount) AS total_ventas,
    COUNT(*) AS num_compras
FROM ventas
GROUP BY
    user_id,
    TUMBLE(ts, INTERVAL '1' HOUR)
ORDER BY total_ventas DESC;
💡
Ventaja clave: El mismo código SQL funciona sobre datos históricos (batch) y sobre un stream en tiempo real. Cambias la fuente de datos (un archivo CSV vs un topic de Kafka) pero la query es idéntica. Esto simplifica enormemente el desarrollo.
